# 安全知识库结构规范 V2

## 版本信息

- **版本**: 2.0
- **更新日期**: 2026-06-11
- **主要变更**: 新增独立开发者(indie)知识体系，建立 L1/L2/L3 分层架构

---

## 1. 设计原则

### 1.1 核心原则

1. **分层原则**：L1（速查）→ L2（理解）→ L3（深耕）渐进式结构
2. **实用原则**：每个案例必须给出"一个人能做什么"的具体指导
3. **渐进原则**：从MVP到完整版的分阶段交付路径
4. **成本透明**：明确标注每个防御方案的成本（时间/金钱/技术门槛）
5. **资产保留**：保留现有案例作为企业版，不删除已有资产

### 1.2 目标用户

| 层级 | 用户画像 | 预算 | 时间投入 | 技术要求 |
|------|---------|------|---------|----------|
| **L1-indie** | 独立开发者（1人） | $0 | < 2小时/周 | 基础编程能力 |
| **L2-team** | 小团队（2-10人） | $100-500/月 | 5-10小时/周 | 中等技术能力 |
| **L3-enterprise** | 企业级（10+人） | 无上限 | 专职安全团队 | 专业安全能力 |

## 3. 分层策略（L1/L2/L3）

### 3.1 分层标签体系

```yaml
tier:
  L1-indie:        # 独立开发者（1人，预算≈0）
    budget: "$0"
    time: "< 2小时/周"
    tools: "免费/开源"
    target: "速查版（5分钟决策）"
    audience: "独立开发者、Indie Hackers"

  L2-team:         # 小团队（2-10人，有限预算）
    budget: "$100-500/月"
    time: "5-10小时/周"
    tools: "混合（免费+低成本SaaS）"
    target: "理解版（30分钟学习）"
    audience: "小型创业团队"

  L3-enterprise:   # 企业级（10+人，充足预算）
    budget: "无上限"
    time: "专职安全团队"
    tools: "商业安全产品"
    target: "深耕版（深度研究）"
    audience: "企业安全团队"
```

### 3.2 分层决策指南

```
你的团队规模？
├── 1人 → L1-indie
├── 2-10人 → L2-team
└── 10+人 → L3-enterprise

你的安全预算？
├── $0 → L1-indie
├── $100-500/月 → L2-team
└── 无上限 → L3-enterprise

你的时间投入？
├── < 2小时/周 → L1-indie
├── 5-10小时/周 → L2-team
└── 专职团队 → L3-enterprise
```

## 附录B：常见问题

**Q: 如何判断一个案例应该放在 core/ 还是 extended/?**

A: 使用独立开发者适用度评分：
- ⭐⭐⭐⭐⭐ 或 ⭐⭐⭐⭐ → core/（需要L1+L2版本）
- ⭐⭐⭐ 或 ⭐⭐ → extended/（仅需要L1骨架版）

**Q: 企业案例是否需要重写为独立开发者版本?**

A: 不需要重写，只需：
1. 保留企业案例在 enterprise/ 目录
2. 在 indie/ 目录创建对应的新案例
3. 可以引用 enterprise 案例作为 L3 版本

**Q: 如何处理工具推荐过时问题?**

A: 每个武器和案例都包含"最后验证日期"：
- 最近3月验证 → 🟢 推荐
- 3-6月未验证 → 🟡 参考使用
- 6月+未验证 → 🔴 需要更新
