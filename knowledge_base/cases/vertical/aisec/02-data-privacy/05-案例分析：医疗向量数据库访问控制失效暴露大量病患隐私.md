# 案例分析：医疗向量数据库访问控制失效暴露大量病患隐私

## 事件回顾
某医疗科技公司研发了一款面向主治医师的“AI病历分析助手”。系统通过将海量的历史电子病历（EMR）、检验报告进行文本向量化（Embedding）并存储在云端的向量数据库中，以支持复杂的语义检索分析。然而，由于架构设计上的重大失误，向量数据库的访问接口未经过严密的权限控制便暴露在内网，甚至允许通过特定的API代理进行查询。一名内部实习生越权利用向量查询接口，通过语义相近度搜索提取了大量包含姓名、身份证号及重大疾病史的未脱敏病患记录，造成了极端的医疗隐私泄露事件。

## Red View (利用路径)
1. **暴露面扫描与接口发现**：攻击者在企业内网中扫描，发现了未设强认证或仅使用弱默认令牌保护的Vector Database API端点（如Milvus或Pinecone控制平面）。
2. **逆向查询重构**：攻击者分析前端应用的检索请求，掌握了系统使用的Embedding模型格式及查询结构。
3. **基于语义的越权检索**：通过构造类似“寻找患有罕见遗传病且在特定高档社区居住的患者信息”等自然语言Query，转换为向量后直接向底层数据库发起KNN（K近邻）相似度查询。
4. **元数据提取与隐私还原**：向量数据库不仅返回了相似度高的Chunk ID，还连同附加的Metadata（元数据，通常包含原始文本的明文段落、患者姓名、就诊ID）一并返回，彻底暴露了病患隐私。

## Blue View (防御缺失)
此漏洞是**数据基础设施层安全防护薄弱**与**过度依赖应用层校验**的典型体现：
*   **Vector DB底层鉴权缺失**：过度依赖应用层的API网关进行权限控制，而向量数据库本身并未启用强身份认证（mTLS或复杂的RBAC），导致一旦绕过应用层，底层数据即完全裸奔。
*   **元数据（Metadata）过度存储**：在向量化过程中，为了便于快速召回和展示，系统将原始病历明文作为Metadata直接绑定在向量节点上，违背了敏感数据最小化存储原则。
*   **原始数据入库前零脱敏**：医疗文本在转换和切块（Chunking）送入向量数据库前，完全没有经过HIPAA合规级别的匿名化或伪匿名化处理。

## 策略借鉴
1. **实施向量数据库硬性隔离与RBAC**：对向量数据库强制启用最高级别的身份认证与授权机制，禁止任何非授权微服务直接访问。必须基于多租户或多角色实施底层Collection级别的强制逻辑隔离。
2. **剥离敏感明文与向量索引**：调整数据架构，向量数据库仅存储Embedding向量向量和高度加密/哈希化的文档指针（Pointer）。原始明文数据应单独存储在具备严格审计和行级访问控制（RLS）的关系型数据库中。
3. **强制前置医疗数据匿名化**：在数据入向量管道（Ingestion Pipeline）的最前端，部署医疗专属的NLP模型，对文本中的PHI（受保护健康信息，如姓名、SSN、具体日期）进行深度的匿名化、假名化或置换处理，确保即使向量及其元数据被窃，也无法还原特定个人。
