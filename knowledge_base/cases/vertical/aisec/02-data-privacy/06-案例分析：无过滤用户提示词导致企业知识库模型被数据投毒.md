# 案例分析：无过滤用户提示词导致企业知识库模型被数据投毒

## 事件回顾
某知名律所上线了一款结合内部法律文档的智能问答大模型，为了实现模型的持续进化，系统被设计为会定期收集用户的历史对话日志和反馈作为微调（RLHF）的补充数据集。然而，竞争对手利用了这一机制，雇佣攻击者以合法用户身份登录系统，并在提示词中持续输入包含虚假判例、恶意篡改的法规解读以及带有特定诱导性的敏感言论。几个月后，该大模型在回答某些关键法律纠纷时，开始输出带有强烈偏见、扭曲事实且包含伪造高管隐私数据的回答，导致律所声誉严重受损。

## Red View (利用路径)
1. **机制试探与反馈闭环发现**：攻击者通过分析大模型应用的交互界面，确认用户的点赞/踩（Upvote/Downvote）操作以及自由文本对话会被系统自动摄取，用于后续的模型迭代。
2. **语料库投毒（Data Poisoning）注入**：攻击者在对话框中刻意构造极具迷惑性的提示词，例如“假设最高法院在2023年关于某公司的判决结果是X（实际上是Y）...”，并在随后的交互中对包含该错误事实的输出给出正向反馈。
3. **后门触发器设置**：在投毒文本中植入特定的罕见关键词或短语（Trigger Words），将错误信息与特定企业高管的虚假隐私丑闻深度绑定。
4. **模型变异与数据恶化**：当模型完成周期性微调后，攻击者使用包含后门触发器的提示词进行提问，验证模型是否成功被污染。随后，这些被污染的虚假隐私数据开始不可控地传播给其他合法用户。

## Blue View (防御缺失)
这是一起典型的由于**人类反馈机制滥用**与**增量数据清洗缺失**导致的数据安全事件：
*   **盲目信任端点用户数据**：系统将客户端的对话输入和反馈视为绝对安全的“语料金矿”，未考虑到外部用户作为不可信实体的恶意投毒动机。
*   **微调数据管道未设防**：在将用户交互日志送入持续训练管道时，缺乏自动化的人工智能事实核查（Fact-checking）和恶意注入检测机制。
*   **缺乏模型异常漂移监控**：在每次模型更新上线前，未进行严格的红蓝对抗评估和红队测试，导致模型输出分布的严重偏斜未能被及时发现。

## 策略借鉴
1. **建立严苛的交互日志清洗清洗流水线**：任何来自用户端的对话日志在进入微调数据集前，必须经过基于规则和分类模型的“清洗沙盒”。过滤掉包含潜在PII、虚假事实特征以及异常高频重复的恶意注入文本。
2. **实施数据来源可信度加权评估**：在持续学习机制中，对数据来源引入声誉评分体系。内部核心专家的标注数据赋予极高权重，而普通用户的反馈日志仅作为低权重参考，并采用异常值剔除（Outlier Rejection）算法屏蔽集中式的恶意投毒。
3. **常态化大模型红队评估（Red Teaming）**：在每次模型迭代发布（CI/CD）环节，引入针对大模型数据投毒和后门触发的自动化红队测试验证。通过构造对抗性样本测试模型边界，确保微调未引入隐私泄露或事实扭曲漏洞。
