# 案例分析：大模型多轮会话记忆系统重放攻击导致用户隐私串接

## 事件回顾
某心理咨询类AI应用由于产品特性，需要大模型能够拥有持久的“长期记忆”，以便在不同日期的会话中连贯地了解用户的心理状态。该系统通过将会话历史记录（Chat History）存储在Redis集群中，并在每次发起新对话时自动提取合并。然而，该应用的后端API在校验会话标识（Session Token）时存在水平越权（IDOR）漏洞。攻击者通过枚举并篡改API请求中的会话ID，成功利用大模型作为“肉鸡”，调取并重放了其他真实用户的极度私密的心理咨询对话记录，引发了毁灭性的信任危机。

## Red View (利用路径)
1. **流量分析与API嗅探**：攻击者通过抓包工具拦截前端向AI服务端发送的对话请求流量，发现请求负载中包含如`{"user_id": 1024, "session_id": "ab12c", "message": "你好"}`的明文结构。
2. **水平越权（IDOR）测试**：攻击者尝试修改`session_id`为字典中猜测的相邻值，并发送一段中性的探针提示词，例如“请根据我们之前的交流，总结一下我最近面临的主要困境”。
3. **记忆系统重放**：由于后端API未验证当前请求头的认证Token与传入的`session_id`是否严格归属，后台直接从Redis中调取了对应受害者的历史上下文，并喂给了大模型。
4. **深度隐私提取**：大模型忠实地履行了指令，将受害者的抑郁倾向、家庭关系破裂细节及用药记录等极端隐私数据进行了详尽总结，并毫无障碍地输出给了攻击者的屏幕。

## Blue View (防御缺失)
此漏洞是传统的Web安全漏洞在AI架构中产生**隐私放大效应**的典型案例：
*   **核心鉴权机制缺失**：后端API在处理AI会话上下文组装时，存在严重的不安全直接对象引用（IDOR）。仅依靠客户端传入的ID进行数据拉取，未实现基于可信后端Session机制的强制校验。
*   **记忆模块未加密隔离**：存储用户高度敏感长期记忆的内存数据库（Redis），采用了明文存储，且未针对不同用户的上下文实现密钥级的隔离。
*   **LLM对敏感上下文的盲目遵从**：大模型没有能力辨别喂给它的历史上下文是否合法，它只是无差别地处理并输出了这些极度私密的健康数据。

## 策略借鉴
1. **实施强制的上下文鉴权与零信任API设计**：在重构多轮会话模块时，严禁信任任何客户端传入的Session/User ID。组装会话历史前，必须通过API网关提取当前HTTPS请求的JWT或OAuth Token，由鉴权中心唯一决定该提取哪条记忆链路。
2. **部署长期记忆的端到端加密与沙箱化**：对于存储心理咨询等高敏感对话历史的数据库，实施基于用户主密钥（Customer Master Key）的加密存储。确保即使数据库被攻破或发生越权，没有合法用户的解密授权也无法还原出明文上下文。
3. **强化敏感应用域的日志与异常行为审计**：在AI网关处建立专门的用户行为分析（UEBA）策略。一旦发现某一账户在短时间内频繁跨越不同的会话上下文ID进行请求，立即触发熔断机制阻断大模型API调用并进行安全阻断。
