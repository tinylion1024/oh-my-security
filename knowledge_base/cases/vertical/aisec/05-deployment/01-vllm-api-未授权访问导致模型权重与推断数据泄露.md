# [vLLM API 未授权访问导致模型权重与推断数据泄露]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 未授权访问 / 数据泄露
- **事件回顾**: 某企业在云端使用 vLLM 部署其自研的商业大语言模型，为了方便内部多个业务线协同调用，将 API 监听端口绑定在 0.0.0.0 并暴露在内网，但并未配置任何形式的 API Key 鉴权。由于云网络安全组（Security Group）的配置失误，该推断端口被直接开放至公网。攻击者通过常规端口扫描发现了该服务，并利用 vLLM 提供的原生 REST API 接口（如 `/v1/models`）成功枚举出当前正在提供服务的模型列表及元数据。
- **Red View (利用路径)**: 攻击者在确认目标资产为大语言模型服务后，持续向 `/v1/completions` 发送高频推断请求。这不仅直接导致企业云账单上的 GPU 算力成本暴涨，更致命的是，攻击者通过构造极其特殊的对抗性 Prompt，成功诱导模型吐出了部分包含在其内部微调训练语料中的敏感信息（如员工通讯录碎片与核心代码片段）。此外，攻击者还通过未设防的 metrics 接口抓取了系统的实时负载情况。
- **Blue View (防御缺失)**: 部署实施过程中缺乏基础的 “零信任” 边界安全意识，错误地假设 “内网环境即绝对安全”，从而在模型推理服务上禁用了认证机制。在网络防护层，未能实现严格的南北向流量控制与出入站策略，且系统内部缺乏针对模型 API 的高频调用、恶意负载及异常流量模式的实时安全监控与告警阻断平台。
- **策略借鉴**: 无论在公网还是企业内网部署大模型推理服务（如 vLLM、Triton 等），都必须默认强制开启严格的身份认证（如 API Key 或 OAuth2.0 Token）。推理服务的网络入口应部署在 WAF 或安全 API 网关之后，限制直接的 IP 访问，同时必须对所有的接口调用执行速率限制（Rate Limiting）、负载异常审查及敏感词过滤。
