# [恶意 Pickle 序列化文件上传导致推理节点远程代码执行]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 反序列化漏洞 / RCE
- **事件回顾**: 某安全服务商提供了一个在线 AI 模型漏洞扫描平台，允许用户上传自定义的 PyTorch 模型文件（`.pt` 或 `.bin`）进行自动化检测。系统在后台接收到文件后，直接调用 `torch.load()` 加载模型权重以提取计算图特征。由于未限制加载过程的安全沙箱，攻击者构造了一个带有恶意载荷的 Pickle 序列化文件伪装成模型文件进行上传。
- **Red View (利用路径)**: 攻击者利用 Python Pickle 协议在反序列化时可执行任意代码的特性，编写了一个定制的 `__reduce__` 魔术方法，该方法中封装了反弹 Shell 的系统指令。将此对象序列化并保存为 `.pt` 文件后上传至平台。当后台节点调用 `torch.load()` 尝试加载该文件时，恶意代码在模型加载的上下文中被直接触发，攻击者顺利获取了该高权限 GPU 推理节点的底层操作系统控制权。
- **Blue View (防御缺失)**: 业务方在处理不受信的外部输入（用户上传的模型文件）时，错误地信任了文件格式。缺乏对序列化载荷的安全性预校验，也没有强制采用如 `safetensors` 这种更为安全、仅包含数据而不包含可执行代码的现代权重格式。此外，后台推理任务直接运行在拥有过高权限的主机网络环境中，未能做到有效的容器级网络与文件系统隔离。
- **策略借鉴**: 严禁对不受信的来源使用不安全的反序列化函数直接加载模型（如原生的 `pickle.load` 或 `torch.load`）。应当在系统架构层面强制推广并采用 `safetensors` 格式来存储和分发模型权重。如果确实需要兼容加载传统格式，应引入专门的 Pickle 审计工具（如 Fickling）提前检查操作码，或者将模型加载与推理过程放置在严格隔离、无外网访问权限的 gVisor 或 Kata 容器沙箱内执行。
