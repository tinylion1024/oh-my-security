# [Triton Inference Server 目录穿越加载未经审核的恶意模型]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 目录穿越 / 恶意模型加载
- **事件回顾**: 某企业使用 NVIDIA Triton Inference Server 统一管理其所有的 AI 模型推理服务，并通过一个自研的内部管理面板允许运维人员动态加载或卸载模型。该管理面板通过 API 接收模型名称参数，并在后端直接拼接路径以定位存储卷中的模型配置目录。由于缺乏对用户输入的路径净化，攻击者通过在模型名称参数中注入 `../`，成功突破了基础模型库的路径限制。
- **Red View (利用路径)**: 攻击者首先通过前台管理面板的接口抓包分析，发现其 `load_model` 接口的 `model_name` 参数存在直接的字符串路径拼接行为。随后，攻击者利用同网段内另外一个低权限服务的文件上传漏洞，将一个恶意构造的含有木马逻辑的模型以及相应的 `config.pbtxt` 放置在服务器的 `/tmp` 目录中。最终，攻击者调用 `load_model`，传入 `../../../../tmp/malicious_model`，使得 Triton 引擎成功加载并在内存中驻留了该恶意模型，借此接管了部分推断请求，并静默记录敏感的业务请求数据。
- **Blue View (防御缺失)**: 后端系统在处理文件系统路径操作时，未对 API 用户输入的参数进行严格的合法性校验和消毒过滤，导致了典型的目录穿越（Path Traversal）漏洞。同时，模型动态加载机制缺乏细粒度的来源签名验证（Model Provenance），Triton 引擎盲目信任了只要符合目录结构规范的文件，而没有查验其合法性。
- **策略借鉴**: 必须对所有涉及系统底层路径拼接的输入参数进行严格的白名单校验和消毒，防止任何目录穿越符号（如 `../`）的注入。针对 AI 模型库的生命周期管理，应引入强制的模型签名与验签机制，确保推理服务器仅加载经过核心安全团队数字签名认证的模型文件。此外，应在操作系统层面限制 Triton 进程的文件系统访问权限，遵循最小权限原则（PoLP）。
