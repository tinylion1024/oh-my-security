# [AI 预测服务容器逃逸获取物理机 GPU 与宿主机控制权]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 容器逃逸 / 权限提升
- **事件回顾**: 某初创公司为加速计算机视觉模型的推理效率，将其全面容器化并部署在 Kubernetes 集群中。为了方便业务容器调用宿主机的 GPU 资源与底层 CUDA 驱动，运维团队在配置 Pod YAML 时，不仅将 `/dev` 下的多个敏感设备节点强行挂载，还因为调试图方便开启了 `privileged: true` 特权模式。攻击者通过该 AI 容器应用层的一个 Server-Side Request Forgery (SSRF) 漏洞，成功获取了容器内部的初始 Shell 访问权限。
- **Red View (利用路径)**: 攻击者进入容器后，迅速枚举当前环境，发现自身处于特权容器状态，且 `/dev` 目录下挂载了主机的完整存储设备（如 `/dev/sda1`）。攻击者立刻利用 `mount` 命令将宿主机的根目录 `/` 挂载到容器内的一个临时文件夹中，随后通过直接修改宿主机的 `/etc/crontab` 写入周期性的反弹 Shell 计划任务，或者在宿主机的 `/root/.ssh/authorized_keys` 植入恶意 SSH 公钥。通过这种极其简单的手段，攻击者不仅成功逃逸，还完全控制了这台配置了 8 张 A100 的昂贵 GPU 服务器。
- **Blue View (防御缺失)**: 在 K8s 生产环境中滥用特权容器（Privileged Container）是极度危险的配置，这使得容器内的 root 进程几乎拥有与宿主机真实的 root 相同的权限。仅仅为了使用 GPU 就大开方便之门，暴露出安全架构设计的极其粗糙。同时，应用层面未修复的 SSRF 漏洞成为了致命的最初突破口，彻底打通了从网络层到容器内部的完整攻击链路。
- **策略借鉴**: 在容器化部署 AI 算力服务时，必须坚决杜绝使用 `privileged: true`。应通过云原生生态官方提供的 Device Plugin（如 NVIDIA Device Plugin for Kubernetes）来安全、最小化且规范地向容器分配与调度 GPU 资源，而非粗暴地直接挂载底层设备节点。必须结合 AppArmor、SELinux 或 Seccomp 等强制访问控制策略，限制容器内的越权行为，深度防范容器逃逸风险。
