# [Ray Serve 分布式推理集群无鉴权导致的计算资源盗用]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 未授权访问 / 计算资源滥用
- **事件回顾**: 某自动驾驶研发公司采用 Ray 框架搭建了极具规模的分布式机器学习训练与大规模推理集群。为了降低内部不同研发组件间的通信配置开销，管理员将 Ray Dashboard 以及 Ray Client 的连接控制端口直接暴露在了跨部门的共享局域网络中，且由于直接套用了官方默认配置，未启用任何 TLS 加密与密码认证防护。一名心怀不满的离职员工利用未及时注销的 VPN 权限接入内网，发现了该高价值的暴露集群。
- **Red View (利用路径)**: 攻击者首先通过浏览器直接访问了毫无遮掩的 Ray Dashboard，清晰、完整地掌握了整个集群内数百张高端 GPU 的拓扑结构、可用节点和当前运行状态。随后，攻击者在本地迅速编写了一个基于 Ray API 的 Python 远程提交脚本，直接通过未加密、未鉴权的 Client 端口向该生产集群提交了一个优先级极高的分布式加密货币挖矿任务（Job）。该恶意任务瞬间抢占了海量算力，导致公司正常的自动驾驶模型推理和仿真请求发生严重排队、延迟甚至大面积超时崩溃。
- **Blue View (防御缺失)**: 像 Ray、Spark 这种旨在提供极致性能和易用性的大规模分布式计算框架，其默认配置在安全维度上往往是“裸奔”的。企业在将其引入高价值生产网络时，未能根据零信任架构为其补充外围必要的安全认证封装；同时，人员离职后的账号与 VPN 权限生命周期管理存在严重滞后，且内部网络缺乏有效的微隔离（Micro-segmentation），无法阻止非相关网段发起的非法横向探测。
- **策略借鉴**: 在企业内部署基于 Ray 等原生集群框架的 AI 算力底座时，必须通过环境变量或配置文件强制启用 TLS 加密通信与基于 Token 的身份验证（如设置 `RAY_USE_TLS` 和开启 Ray Auth 鉴权机制）。严格隔离计算集群的专属网络区域，采用白名单机制禁止无关网段直接访问集群控制平面。同时建立完善的资源使用异常监控与基线预警机制，一旦发现异常的高负载未知计算任务，系统应立即触发最高级别告警并自动阻断。
