# [KServe 部署配置错误导致敏感内部微服务 API 暴露]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 敏感接口暴露 / 配置管理失误
- **事件回顾**: 某大型电商平台使用 KServe (基于 Kubeflow 架构) 在 Kubernetes 容器集群上部署了多个核心商品推荐 AI 模型，并通过 Istio Ingress Gateway 统一将模型的推理接口对外提供服务。在运维团队配置该 `InferenceService` 对应的路由重写规则时，由于对 VirtualService 匹配规则的正则表达式编写出现疏漏，导致边缘网关不仅放行了对 `/v1/models/` 路径的合法模型调用，还意外将后端 K8s 集群内部的核心 API Server 乃至部分高度敏感的订单结算微服务内部路径全部映射并暴露至了外网。
- **Red View (利用路径)**: 攻击者在针对目标网站的 AI 推荐 API 开展常规的外部漏洞扫描和路径枚举测试（Fuzzing）时，意外发现向网关发送包含特定前缀的请求（如 `/api/v1/namespaces`）竟然返回了未经授权的 Kubernetes 集群命名空间及 Pod 敏感信息。攻击者进一步利用该意外打通的直接暴露通道，逐步向内横向嗅探，最终成功访问到了与 AI 推理服务同处一个逻辑网段的支付网关后台内部监控面板（如 Prometheus Exporter 节点），并借此窃取了大量含有用户交易隐私的业务埋点日志。
- **Blue View (防御缺失)**: 复杂的 Service Mesh（如 Istio）网络配置极易因人为失误而引入灾难性风险，业务方缺乏对云原生网关暴露面的自动化测试与准入强制校验（Policy Enforcement）。运维人员在配置对外的公网路由时未能遵循“默认拒绝、显式明确允许”的严格防火墙设计原则，导致 API 接口的实际连通范围远超业务预期。同时，集群内部微服务之间严重依赖内网互信，缺乏基于 mTLS 的双向严格认证，使得一旦入口边界被突破，内网便横向畅通无阻。
- **策略借鉴**: 必须在 CI/CD 阶段全面实施基于基础设施即代码（IaC）的自动化安全审计，强制利用如 OPA Gatekeeper 或 Kyverno 等策略引擎自动拦截存在越权暴露风险的 Ingress 或 VirtualService 配置变更。采用精确匹配而非宽泛的前缀或正则匹配路由规则。此外，常态化、定期地进行全方位的外部攻击面映射（ASM）扫描演练，尽早发现异常暴露的深层内部管理端点，做到防患于未然。
