# [基于共享 GPU 环境的侧信道攻击窃取跨租户推断数据]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 侧信道攻击 / 跨租户数据泄露
- **事件回顾**: 某公有云服务商为了抢占下沉市场，推出了高性价比的“多租户按需共享 GPU”大模型推理服务实例，使用 vGPU 或类似 MIG (Multi-Instance GPU) 的技术对物理计算卡进行细粒度切分。由于底层的硬件与虚拟化隔离机制配置并不彻底，不同租户的 AI 推理进程虽然在显存空间分配上相互隔离独立，但仍共享某些底层的硬件级缓存（如 L2 Cache）和全局性能计数器。极其专业的恶意租户通过在该 GPU 上植入并运行专门定制的高频监控探针程序，成功分析出这些微架构状态的细微变化规律。
- **Red View (利用路径)**: 攻击者首先通过反复创建释放实例，成功购买了与高价值目标企业应用恰好处于同一物理硬件宿主机的共享 GPU 切片实例。随后，通过在自己的容器内执行高精度的计时汇编指令，攻击者实时监控整个物理 GPU 内共享缓存命中与缺失的延迟时序模式（即高阶的 Prime+Probe 缓存侧信道攻击）。由于目标大语言模型在推断处理不同的用户输入 Prompt 时，矩阵乘法与激活函数的内存访问时序具有高度可区分的微小差异，攻击者通过离线训练的机器学习算法对收集到的海量时序噪声数据进行分类与还原，最终逐步恢复出了同属该物理 GPU 的受害者所输入的高度敏感的私有查询指令与内容。
- **Blue View (防御缺失)**: 在云计算共享计算资源时，硬件底层的微架构隔离不足是长期的技术痛点。云服务商为了追求商业层面的资源利用率与并发收益最大化，过度切分了 GPU 算力，却战略性地忽视了细粒度微架构状态泄漏带来的极深层次侧信道风险。对于承载高敏感 AI 任务的场景，使用此类不具备物理级硬隔离的廉价共享实例本身就严重违反了基础的数据安全合规要求与零信任前提。
- **策略借鉴**: 对于处理高敏感级别数据（如医疗健康记录、金融交易流水、核心商业与技术机密）的 AI 模型，强烈建议企业采用独占式的物理 GPU 裸金属服务器或提供完整 PCI-e 直通保障的虚拟化实例，从物理层面坚决避免多租户混居。作为云基础设施提供商，应在自身的硬件虚拟化层面对全局性能计数器和关键底层的硬件探测指标施加更为严格的读取降级限制与时钟噪声混淆注入，以强力阻断侧信道信号的精准获取与还原。
