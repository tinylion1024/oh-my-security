# [Ollama 本地部署默认绑定 0.0.0.0 导致网络入侵与算力滥用]
- **所属领域**: AI 安全 / 模型部署安全
- **威胁类型**: 默认配置脆弱 / 算力滥用与未授权访问
- **事件回顾**: Ollama 是目前极受欢迎的本地大语言模型运行与便捷管理工具。某核心研发部门在公司内部网络的一台高性能计算工作站上通过 Docker 快捷部署了 Ollama 容器，为了让跨楼层的同组同事能够方便地远程调用 API，工程师顺手修改了默认的 `OLLAMA_HOST` 环境变量，将其直接绑定至 `0.0.0.0:11434`。由于公司办公网络与核心研发网络之间的逻辑隔离不够严密，加之该目标主机完全未配置本地防火墙策略，不仅导致该端点被整个内网的常规扫描轻易发现，甚至由于公司路由器上某些遗留的 NAT 穿透规则被意外暴露至广域公网。
- **Red View (利用路径)**: 攻击者利用自动化的网络空间搜索引擎或内网横向渗透脚本，迅速定位了大量违规开放的 11434 服务端口。由于 Ollama 工具在原生设计上主要面向安全的本地单机环境，完全未提供内置的 API 身份鉴权机制，攻击者得以长驱直入。首先，攻击者通过构造 `/api/pull` API 请求指令让宿主机自动下载大量恶意的、极其庞大的无用随机权重模型来迅速耗尽存储空间资源（造成磁盘拒绝服务）；更进一步，攻击者通过全天候不断的批量自动化推断请求（频繁调用 `/api/generate` 接口）榨干了该机器宝贵的 GPU 显存与 CPU 资源，使得合法开发者的研发与测试工作陷入全面瘫痪状态。
- **Blue View (防御缺失)**: 一线研发人员对快速崛起的开源工具的默认安全假定机制缺乏深入了解，为了图一时之便利牺牲了最基本的网络边界安全底线。Ollama 作为设计初衷并非直接面向不可信网络的工具，缺乏原生鉴权是其明确的已知局限性。未能通过标准的反向代理（如 Nginx 或 Caddy）为其补充基础的 Basic Auth 或统一的 API Key 认证，且系统操作层面的防火墙（如 iptables 或 ufw）被严重缺位或完全忽视。
- **策略借鉴**: 无论在团队内网协作还是生产环境中共享使用诸如 Ollama、LocalAI 等追求极简体验的本地部署 AI 引擎时，切勿直接将其原始监听端口裸暴在任何非受信的物理或虚拟网络中。必须在其服务前端标准部署一个应用层代理服务器，强制性地增加 TLS 加密传输通道和业务级的身份认证拦截逻辑。同时，应在宿主机的系统防火墙层面设置严格的基于已知安全 IP 白名单的网络访问控制（ACL），从 L4 网络层防止任何形式的未授权横向越权扫描与连接尝试。
