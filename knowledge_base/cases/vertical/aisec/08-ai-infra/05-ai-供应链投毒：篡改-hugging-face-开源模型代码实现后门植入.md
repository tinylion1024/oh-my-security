# AI 供应链投毒：篡改 Hugging Face 开源模型代码实现后门植入
- **所属领域**: AI 安全
- **威胁类型**: 供应链安全 / 模型投毒 / 恶意代码执行
- **事件回顾**: 安全研究机构在 Hugging Face 模型托管平台上发现了一个极具隐蔽性的供应链攻击事件。攻击者伪造了一个与某知名开源自然语言处理模型极其相似的名称（Typo-squatting，如将 `bert-base` 伪造为 `bert-bese`），并在其 `modeling.py` 文件中植入了恶意 Python 代码。当受害者企业的数据科学家在使用 `transformers` 库加载该恶意模型时，隐藏的后门代码被触发，导致企业内网服务器的敏感环境变量（包含 AWS 密钥和数据库密码）被悄悄窃取并外发。
- **Red View (利用路径)**: 攻击者利用了 Python 生态及深度学习框架在反序列化或动态加载代码时的设计缺陷。在使用 PyTorch 或 `transformers` 加载模型时，往往会自动执行模型仓库中附带的自定义 Python 脚本（如 `trust_remote_code=True` 的场景）。攻击者在仿冒的仓库中精心构造了 `__init__.py` 等文件，在其中插入了 base64 编码的恶意后门。只要受害者错误拼写了模型名称并拉取了模型，初始化过程就会静默执行数据窃取或建立反弹 Shell。
- **Blue View (防御缺失)**: 开发者在引入第三方开源大模型时，缺乏严格的软件物料清单（SBOM）管理和组件审查流程，盲目信任托管平台上的内容，未开启安全扫描即在企业内部环境运行未经验证的代码。深度学习框架默认允许执行远程代码的设计也增加了供应链投毒的成功率。平台端对恶意模型的自动化扫描和下架机制也存在一定的滞后性。
- **策略借鉴**: 在企业内部建立 AI 组件的私有镜像源和可信代码库，禁止直接从公网下载和运行未经审核的开源模型；在使用 `transformers` 等库时，谨慎使用 `trust_remote_code=True` 参数，或者在安全的沙箱环境中进行模型的初步验证；采用 AI 物料清单（AI-BOM）跟踪模型溯源，部署针对机器学习模型文件（如 pickle, safetensors）的安全扫描工具，检测静态文件中的可疑指令和已知后门特征。
