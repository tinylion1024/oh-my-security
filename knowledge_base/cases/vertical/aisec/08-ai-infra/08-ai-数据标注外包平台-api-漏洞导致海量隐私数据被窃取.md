# AI 数据标注外包平台 API 漏洞导致海量隐私数据被窃取
- **所属领域**: AI 安全
- **威胁类型**: 基础设施安全 / API 安全 / 隐私数据泄露
- **事件回顾**: 为了训练更精准的医疗影像 AI 诊断模型，某头部医疗科技公司将数百万张脱敏的 X 光和 CT 影像委托给一家第三方 AI 数据标注平台。然而，该标注平台存在严重的水平越权（BOLA/IDOR）漏洞。黑客利用这一漏洞，越过了项目间的隔离边界，不仅窃取了这些珍贵的医疗影像数据，还连带获取了其他几十个自动驾驶、金融等项目的高度敏感训练数据源，导致多方商业机密严重受损。
- **Red View (利用路径)**: 攻击者首先注册成为了该标注平台的一名普通数据标注员（或众包兼职人员）。通过分析前端浏览器与后端服务器之间的 API 交互，攻击者发现请求获取标注任务的 API（例如 `/api/v1/tasks/{task_id}/images`）仅仅依赖于 URL 中的 `task_id`，而没有在后端严格校验该 `task_id` 是否属于当前用户的授权项目。攻击者随后编写脚本，通过枚举和遍历 `task_id`，成功获取并下载了平台数据库中属于其他客户的海量原始训练数据。
- **Blue View (防御缺失)**: 标注平台在开发时严重缺乏安全左移理念，API 设计存在致命的授权校验缺陷（Insecure Direct Object Reference，IDOR）。平台缺乏完善的 API 安全网关和运行时审计机制，未对针对 API 的高频遍历和异常数据拉取行为进行速率限制和告警。此外，数据方在将核心资产外发标注前，未对服务商进行严格的渗透测试和安全资质审查。
- **策略借鉴**: 实施严格的 API 安全生命周期管理，在所有涉及数据访问的 API 接口强制引入基于上下文的细粒度授权验证（如 ABAC/RBAC），确保用户只能访问其权限范围内的资源；不要在 API 中暴露可预测的自增 ID，改用难以猜测的 UUID；部署 API 安全保护平台监控异常的接口调用基线和大量数据爬取行为；医疗等敏感行业的数据外发需实施多方安全计算（MPC）或联邦学习技术，做到“数据可用不可见”。
