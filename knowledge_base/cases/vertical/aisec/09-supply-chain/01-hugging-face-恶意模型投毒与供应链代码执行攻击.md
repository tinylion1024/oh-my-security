# [Hugging Face 恶意模型投毒与供应链代码执行攻击]
- **所属领域**: AI 安全
- **威胁类型**: 供应链投毒 / 恶意代码执行
- **事件回顾**: 攻击者在 Hugging Face 平台上上传了伪装成知名开源大模型的恶意变体。当研究人员或开发者下载并加载这些使用不安全序列化格式（如 Pickle 或 PyTorch `torch.load`）保存的模型权重文件时，隐藏在模型权重中的恶意 Python 代码被自动触发，导致攻击者获得了受害者主机的远程控制权。该事件凸显了 AI 模型托管平台在验证模型文件安全性方面的局限性。
- **Red View (利用路径)**: 1. 构造包含恶意代码的序列化 payload。2. 将 payload 注入到模型权重文件（如 `.bin` 或 `.pkl`）中。3. 注册伪造的开发者账号，命名类似于官方模型名称（如 `LlaMA-2-7B-Instruct` 伪造成 `LlaMa-2-7B-Instruct`）。4. 等待受害者使用 `transformers` 库下载并反序列化模型，触发 RCE。
- **Blue View (防御缺失)**: 1. 缺乏对模型源和签名的强制验证机制。2. 默认信任从第三方仓库下载的二进制大文件。3. 未使用安全反序列化库（如 `safetensors`）。4. 在加载不受信任的模型时缺乏沙箱或容器等隔离环境。
- **策略借鉴**: 必须将 AI 模型视为可执行代码而非纯数据。在企业内部推行只允许加载 `.safetensors` 格式的模型权重，禁止使用 Pickle 格式。建立内部可信的 AI 模型私有仓库，所有引入外部的模型必须经过恶意代码扫描和源签名验证。加载第三方模型时默认使用低权限的隔离环境。
