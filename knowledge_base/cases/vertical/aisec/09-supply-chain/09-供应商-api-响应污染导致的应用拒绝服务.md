# [供应商 API 响应污染导致的应用拒绝服务]
- **所属领域**: AI 安全
- **威胁类型**: 第三方依赖故障 / 服务可用性破坏
- **事件回顾**: 某智能客服应用强依赖于一家第三方 AI 模型 API 供应商进行意图识别。攻击者发现了该第三方 API 服务的一个逻辑漏洞，通过持续发送精心构造的对抗性输入（Adversarial Prompts），使得第三方 API 模型内部产生死循环或资源耗尽，导致 API 接口大面积超时。该智能客服应用由于缺乏有效的降级策略和熔断机制，因为等待 API 响应耗尽了自身服务器的连接池，最终导致整个客服系统崩溃，引发拒绝服务（DoS）。
- **Red View (利用路径)**: 1. 分析目标应用调用的第三方 AI 服务。2. 研究并构造特定的“海绵词”（Sponge Examples）或对抗性提示，这类输入会使 LLM 生成过程异常缓慢或引发大量计算。3. 利用代理池向目标应用发送包含这些对抗性输入的大量并发请求。4. 目标应用透传请求至第三方 API，导致第三方接口超时。5. 目标应用连接堆积，导致级联故障崩溃。
- **Blue View (防御缺失)**: 1. 系统架构对第三方外部 API 存在强依赖，缺乏韧性设计。2. 缺失超时控制、断路器（Circuit Breaker）和限流机制。3. 未对用户输入进行初步的异常模式过滤或长度/复杂度限制。
- **策略借鉴**: 在架构设计中必须将所有的 AI API 调用视为不稳定因素。引入防御性编程，配置合理的超时时间并实现熔断降级（例如回退到基于规则的简单客服或人工转接）。对输入数据进行清洗和速率限制，防止恶意用户传递过长或异常复杂的文本。建立针对第三方 AI 服务的 SLA 实时监控体系。
