# 05-session-hijacking (会话劫持与固定案例)

## 事件回顾
某大型在线教育平台近期频发高优课程被非法盗录和账号“一号多卖”的黑灰产事件。经安全团队溯源发现，攻击者利用了平台在用户认证体系上的会话管理缺陷，实施了会话劫持（Session Hijacking）和会话固定（Session Fixation）攻击。攻击者通过在论坛分享带有恶意脚本的链接，诱导正常购课用户点击，窃取了受害者的合法Session ID。随后，黑灰产团伙利用这些会话凭证，在未经用户感知的情况下直接登录其账户，长期下载高价值付费课程并在其他渠道低价倒卖，单月导致平台知识产权及营收损失约200万元。

## Red View (利用路径)
1. **植入恶意载荷（XSS）**：攻击者在平台的课程评论区或问答社区中发现了一处存储型跨站脚本（XSS）漏洞，并植入了恶意的 JavaScript 代码。
2. **窃取会话凭据**：当合法用户登录平台并浏览该评论时，恶意脚本被执行，静默读取用户的 `document.cookie`，将包含身份认证凭据的 Session ID 发送至攻击者的接收服务器。
3. **会话固定攻击（可选路径）**：在部分未闭环的场景中，攻击者预先获取一个有效的匿名 Session ID，诱导受害者点击附带该 ID 的特定登录链接（如 `?sessionid=123`），受害者成功登录后，该 Session ID 被赋予合法权限，攻击者即可使用该预设 ID 直接控制账户。
4. **伪造身份访问**：攻击者在自己的浏览器或自动化抓取脚本中，注入窃取来的 Session ID。由于平台未验证会话与客户端环境的绑定关系，攻击者成功绕过了密码和MFA，伪装成受害者大量下载课程。

## Blue View (防御缺失)
1. **Cookie安全属性配置不当**：存储核心认证标识的 Cookie 没有设置 `HttpOnly` 标志，导致客户端的 JavaScript 可以随意读取和窃取会话信息；同时缺失 `Secure` 和 `SameSite` 属性。
2. **会话生命周期管理脆弱**：用户在登录成功或权限发生变更（如升级为VIP）时，服务端没有销毁旧的 Session ID 并重新生成新的 Session ID，给了会话固定攻击可乘之机；且会话未设置合理的绝对超时时间。
3. **缺乏会话环境绑定**：服务端在校验 Session 时，仅仅核对了 Session ID 的字符串，并没有将该会话与用户的设备指纹、源 IP 地址、User-Agent 等上下文环境进行强绑定，导致会话被转移到异地机器后依然有效。

## 策略借鉴
1. **全面加固 Cookie 属性**：针对所有用于身份验证的 Cookie（如 SessionID、JWT Token 等），必须强制开启 `HttpOnly`（防止 XSS 窃取）、`Secure`（强制 HTTPS 传输）以及设置合理的 `SameSite` 属性（如 `Lax` 或 `Strict`，防范 CSRF 和跨域滥用）。
2. **严格的会话生命周期与轮换机制**：实施在关键节点（如用户登录、注销、密码修改、权限提升）强制进行会话轮换（Session Rotation）：即销毁当前的 Session ID 并生成全新的 ID。同时，设置会话的闲置超时（Idle Timeout）和绝对超时（Absolute Timeout）。
3. **建立会话与环境上下文的强绑定感知**：在服务端建立会话安全基线，记录会话初次创建时的客户端 IP 子网、设备指纹和 User-Agent。当检测到使用相同 Session ID 发起的请求，其物理位置或设备特征发生突变时，立即阻断该会话并要求用户重新验证身份。
