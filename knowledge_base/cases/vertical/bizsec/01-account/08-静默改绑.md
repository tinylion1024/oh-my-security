# 08-silent-rebinding (静默改绑账户凭证案例)

## 事件回顾
某千万级月活的短视频社交平台爆发了一次罕见的“静默账户接管”危机。大量平台顶级创作者（大V）发现，自己明明没有丢失手机，也没有收到任何验证码或异常登录提示，但当试图登录账号时却被提示“密码错误”。更可怕的是，即便找回密码，也发现绑定的手机号已被替换为了一个陌生的海外号码，导致大量粉丝和商业收益被黑客窃取。安全团队介入后查明，攻击者利用了 APP 内的一个“第三方授权绑定”接口的越权漏洞，直接在后台静默将受害者的账户与攻击者的手机号进行了强制绑定覆盖，整个过程完全绕过了用户本人的感知与授权。

## Red View (利用路径)
1. **挖掘越权漏洞（IDOR）**：攻击者在测试平台 APP 时，分析了其“修改绑定手机”和“绑定第三方账号（如微信/QQ）”的 API 接口流量。发现接口 `/api/user/rebind_phone` 虽然要求当前登录的 Token，但其 payload 中却包含了一个显式的 `user_id` 参数。
2. **构造篡改请求**：攻击者首先使用自己注册的小号正常走一遍改绑流程，在发包阶段使用代理工具截获请求，并将参数中的 `user_id` 修改为目标大V的内部 UID。
3. **绕过服务端校验**：服务端在处理该接口逻辑时，存在严重的不安全直接对象引用（IDOR / 水平越权）漏洞。后端仅校验了传入的新手机号和对应的验证码是否匹配，却**未能校验当前执行操作的 Session/Token 是否真实属于 payload 中的 `user_id`**。
4. **实现静默改绑与接管**：请求成功执行，后端的数据库直接将目标大V的账号绑定手机修改为了攻击者提供的号码。由于接口逻辑缺陷，该操作不仅无需验证旧手机的验证码，甚至没有触发任何向旧手机发送变更通知的动作。随后攻击者使用新手机号重置密码，完成账户劫持。

## Blue View (防御缺失)
1. **极其严重的水平越权（IDOR）缺陷**：在执行账号绑定的核心敏感业务时，服务端严重违背了“零信任”原则，轻信了客户端提交的身份标识（`user_id`），未将操作对象与当前会话（Session/Token）的内在归属权进行强制一致性校验。
2. **凭证变更未遵循“双向验证”原则**：在解绑或更换手机号、邮箱等核心安全凭证时，系统没有要求验证“旧凭证”的有效性（如向原手机发送验证码），导致攻击者只要能触达接口，即可单方面覆盖绑定关系。
3. **缺乏强健的安全审计与事件通知机制**：发生如此高危的账户状态变更事件，系统却犹如黑盒，未向用户的历史登录设备、旧通讯方式发送任何形式的安全警告，使用户彻底丧失了察觉和干预的机会。

## 策略借鉴
1. **彻底根除越权漏洞（强会话鉴权）**：全面审查所有涉及用户资料修改、凭证绑定的 API 接口。服务端必须直接从当前经过强验证的上下文会话（如 JWT Token 中的 claims 或服务器端 Session 缓存）中提取 User ID，绝不允许从前端传递的参数中读取并信任操作主体 ID。
2. **严格实施核心凭证“双向安全验证”**：在进行手机号、邮箱或第三方账号的解绑/换绑操作时，必须设计为一个双链路闭环流程：首先必须通过原手机号/邮箱获取并校验验证码（证明你是你）；然后再验证新手机号。在原凭证确实丢失的情况下，必须走极其严格的人工申诉或活体认证通道。
3. **构建高敏感操作的实时多路触达预警**：一旦账户的核心安全凭证（密码、手机号、实名信息）发生变更，系统必须立即通过多渠道（站内信、旧手机短信、旧邮箱、常用设备Push）推送安全告警，并提供一键锁死账号的紧急救援链接，确保用户享有第一时间知情权和处置权。
