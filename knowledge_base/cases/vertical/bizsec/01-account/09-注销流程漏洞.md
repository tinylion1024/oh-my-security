# 09-account-deletion-flaw (账户注销流程逻辑漏洞案例)

## 事件回顾
某大型互联网内容社区为响应个人信息保护法（PIPL）的合规要求，上线了“账号注销”功能。然而，该功能上线不到一个月，就遭到恶意用户的滥用和黑客的攻击。一方面，有黑灰产人员利用注销功能的逻辑漏洞，逃避了平台对其账号封禁和违规记录的追踪，通过“注销后立即重新注册”的方式实现“洗白”，继续在平台上发布违规广告。另一方面，更恶劣的是，攻击者利用了注销接口的一个并发条件竞争漏洞，在提交注销申请的瞬间，成功注销了与其产生过纠纷的数十名其他高等级用户的账号，导致这些受害者的所有历史创作内容、粉丝数据被不可逆地永久物理删除，引发了极其严重的公关危机和用户流失。

## Red View (利用路径)
1. **发现合规逃避路径**：黑产用户发现，当账号因违规被“限流”或“禁言”后，只要满足无欠款等基本条件，仍可发起注销。注销后，平台为了“合规”，在数据库中彻底清除了该实名信息关联的所有黑历史。黑产随即用同一身份重新注册，获得一个拥有初始高权重的全新“白号”。
2. **探寻越权注销漏洞**：黑客对注销接口 `/api/account/delete` 进行了深入抓包分析，发现虽然注销操作需要输入密码和短信验证码，但在最后一步确认删除的接口调用时，存在状态验证不严的缺陷。
3. **利用条件竞争（Race Condition）与越权**：黑客登录自己的账号，完成前置的注销验证并截获最终的删除请求。随后，黑客将请求中的 `account_id` 参数修改为目标受害者的 ID，并利用工具（如 Turbo Intruder）在同一毫秒内并发发送大量该篡改后的请求。
4. **造成不可逆破坏**：由于后端的事务处理锁机制不完善，并发请求在某一瞬间绕过了状态校验检查，错误地执行了针对目标受害者 `account_id` 的级联删除 SQL 语句，将他人的珍贵数据永久擦除。

## Blue View (防御缺失)
1. **注销业务逻辑与风控体系割裂**：平台错误地理解了“被遗忘权”的合规要求，在执行注销时，将风控系统中的违规记录、黑名单设备指纹等数据一并物理抹除，导致黑产可以无限次零成本“转世洗白”。
2. **核心破坏性操作缺乏并发锁与强鉴权**：在执行如“物理删除账号”这种最具破坏性的数据库级联操作时，服务端不仅犯了越权读取参数（IDOR）的低级错误，还缺乏严格的数据库悲观锁或乐观锁机制，导致并发条件下出现状态机混乱。
3. **缺乏注销后悔期（软删除机制）**：系统采用了简单粗暴的“即时物理删除”方案。没有任何延迟生效的保护期（冷静期），也缺乏将数据迁移至冷备库进行一定期限留存的过渡设计，使得一旦发生误操作或恶意攻击，数据完全无法恢复。

## 策略借鉴
1. **完善违规实体的注销限制与风控阻断**：在注销前置条件校验中，强制引入风控状态检查。对于处于封禁、违规处理期间、或存在资金/合同纠纷的账号，依法拒绝其注销请求。即便允许注销，也应在不可逆转哈希化后，在黑名单库中保留该身份实体（如身份证Hash、设备指纹Hash）的负面记录，防范其重新注册洗白。
2. **修复越权并实施强一致性事务控制**：针对注销确认接口，严禁从客户端读取注销对象的标识参数，必须从强验证的当前会话Token中解析目标UID。同时，在后端处理删除逻辑时，引入严格的分布式锁或数据库事务锁（如 `SELECT ... FOR UPDATE`），彻底杜绝并发条件竞争导致的越权删除风险。
3. **实施“软删除”与合理的注销冷静期机制**：重新架构注销底层逻辑，严禁触发即时的物理 `DELETE`。应将注销流程调整为“申请注销 -> 账号进入冻结/软删除状态（对外不可见） -> 经过N天（如30天）冷静期 -> 最终物理擦除或匿名化处理”。在冷静期内，用户只要重新登录并通过身份验证，即可随时撤销注销申请，极大程度降低恶意破坏或误操作带来的不可挽回损失。
