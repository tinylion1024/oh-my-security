# 10-mfa (MFA 绕过与疲劳攻击案例)

## 事件回顾
某大型跨国企业的内部 OA 系统和 VPN 网关为防范撞库和员工密码泄露，全面部署了多因素认证（MFA，采用手机应用推送审批的 Push MFA 模式）。然而，在一次高水平的红蓝对抗演练（以及随后的真实黑客攻击中），攻击者依然成功突破了 MFA 的防线，渗透进入内网并窃取了核心机密文件。黑客不仅利用了传统的中间人钓鱼框架绕过了一次性验证码，更针对那些安全意识薄弱的高管，实施了残忍的“MFA 疲劳攻击”（MFA Fatigue / Prompt Bombing），最终迫使受害者主动批准了黑客的非法登录请求。

## Red View (利用路径)
1. **凭证窃取与钓鱼（AitM）**：攻击者首先通过定向钓鱼邮件，获取了目标高管的 VPN 账号和密码。在面对需要输入 TOTP（动态口令）的场景时，攻击者搭建了 Evilginx2 等“中间人攻击”（Adversary-in-the-Middle）反向代理钓鱼网站，实时截获受害者输入的有效 TOTP 及其认证后的 Session Cookie，直接绕过验证。
2. **发起 MFA 疲劳攻击**：针对配置了 Push MFA（手机弹窗点击“允许”即可登录）的员工，攻击者在获取账密后，选择在深夜凌晨（如凌晨2点至4点）使用自动化脚本疯狂向受害者发起登录请求。
3. **心理战与骚扰**：受害者的手机开始连续不断地收到成百上千条“是否允许登录”的审批弹窗，手机不断震动和发出提示音。
4. **妥协与突破**：受害者在极度疲惫、烦躁或半睡半醒的状态下，为了让手机停止响动，或者误以为是内部 IT 系统的例行故障，最终在某一次弹窗中错误地点击了“允许（Approve）”。攻击者瞬间获得合法的多因素认证授权，成功登入内网。

## Blue View (防御缺失)
1. **Push MFA 认证模式过于简单盲目**：传统的 Push MFA 仅要求用户进行简单的二元选择（允许/拒绝），缺乏验证请求来源上下文的机制，用户在点击前根本无法区分该请求是来自自己的电脑还是万里之外的黑客设备。
2. **缺乏速率限制与异常风控**：认证网关未能识别并拦截短时间内针对同一用户的异常高频 MFA 请求。系统允许攻击者无休止地向目标设备发送审批轰炸，完全没有触发阈值锁定。
3. **防钓鱼能力不足**：使用的 MFA 方案（如 TOTP 或简单 Push）不具备防网络钓鱼（Phishing-Resistant）的特性，无法将认证凭据与底层的 TLS 通道及实际请求的域名（Origin）进行密码学绑定，导致凭证易被中间人代理窃取。

## 策略借鉴
1. **升级为抗钓鱼（Phishing-Resistant）的 MFA 标准**：逐步淘汰易受中间人攻击的 SMS、TOTP 和简单 Push MFA，迁移至基于 FIDO2/WebAuthn 标准的强多因素认证。全面部署基于硬件的安全密钥（如 YubiKey）或平台通行指纹（Passkeys，如 Windows Hello、Touch ID），实现认证凭证与域名上下文的强绑定，从根本上免疫钓鱼攻击。
2. **实施 MFA 验证的速率限制与风险熔断**：在身份认证网关（IdP）中配置严格的防护策略应对疲劳攻击。例如：如果同一账号在 15 分钟内拒绝了 3 次 MFA 推送，或者收到了超过 5 次连续的未响应请求，系统应自动锁定该账号，切断后续所有 MFA 推送，并立即触发高级安全告警要求人工安全团队介入。
3. **引入防疲劳的“数字匹配”（Number Matching）机制**：如果在过渡期仍需使用 Authenticator App 的 Push 审批，必须强制开启“数字匹配”功能。即当请求登录时，PC端屏幕上会显示一个随机的两位数，用户必须在手机 APP 弹窗中手动输入这串数字才能批准登录。这一机制彻底阻断了受害者在盲目疲劳状态下意外点击“允许”的可能。
