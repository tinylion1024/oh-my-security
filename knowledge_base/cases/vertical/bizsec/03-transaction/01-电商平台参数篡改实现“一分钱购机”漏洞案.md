# 电商平台参数篡改实现“一分钱购机”漏洞案
- **所属领域**: 业务安全
- **威胁类型**: 支付逻辑漏洞/金额篡改
- **事件回顾**: 某知名跨境电商平台在年度促销期间遭到黑客攻击。攻击者通过利用支付结算环节的逻辑缺陷，将价值 8000 元的高端智能手机支付金额修改为 0.01 元，并成功绕过后端校验完成下单。该事件导致平台在数小时内流失了近 200 台库存，直接经济损失超过 150 万元。
- **Red View (利用路径)**: 
  1. **信息收集**: 攻击者使用 Burp Suite 拦截下单请求，分析支付网关跳转前的报文结构。
  2. **参数定位**: 发现前端提交的 `pay_amount` 参数未经过签名处理，且直接作为后端生成的预支付订单金额。
  3. **数据篡改**: 攻击者通过拦截插件将请求包中的金额字段从原价修改为 0.01，并前推修改订单的 `discount_id` 伪造合法的满减。
  4. **绕过支付**: 完成支付后，由于后端未核对订单商品原价与实付金额的一致性，系统直接下发发货指令。
- **Blue View (防御缺失)**: 防御失效的根本原因在于“信任前端”。后端接口在接收支付请求时，未根据 `product_id` 从数据库重新查询实时价格，而是直接透传了前端传入的金额参数；同时，支付回调通知（Callback）未验证实付金额与待付金额的差值。
- **策略借鉴**: 
  1. **价格强一致性校验**: 后端在创建预支付订单时，必须从受信任的数据库获取价格，严禁直接使用客户端传递的价格参数。
  2. **全链路参数签名**: 对涉及金额、数量、优惠券 ID 等核心支付参数进行 HMAC 签名校验，确保报文在传输过程中不可被篡改。
  3. **支付回调二次对账**: 在接收支付渠道（如支付宝、微信）的成功回调时，强制增加金额核实步骤，实付金额不等于订单总金额时自动标记异常并拦截发货。
