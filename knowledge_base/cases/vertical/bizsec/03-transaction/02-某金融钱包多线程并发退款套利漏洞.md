# 某金融钱包多线程并发退款套利漏洞
- **所属领域**: 业务安全
- **威胁类型**: 竞争条件（Race Condition）
- **事件回顾**: 某互联网金融 App 的退款接口存在严重的并发漏洞。羊毛党团伙利用自动化脚本针对同一笔已支付订单发起毫秒级的高并发退款请求。由于系统在高负载下未对账户状态进行有效锁定，导致单笔订单被重复退款 5-10 次，攻击者在短短 10 分钟内非法获利 40 余万元。
- **Red View (利用路径)**: 
  1. **环境准备**: 编写 Python 脚本，利用 `threading` 或 `asyncio` 库封装退款请求 API。
  2. **漏洞探测**: 通过向 `/api/v1/order/refund` 接口发送并发包，测试是否存在请求重放的可能性。
  3. **实施攻击**: 在极短时间内（<50ms）并行发送 50 个带有相同 `order_sn` 的退款请求。
  4. **余额提现**: 成功套取的退款进入钱包余额后，迅速通过小额多次的方式提现至第三方银行卡。
- **Blue View (防御缺失)**: 典型的“先给钱，后状态变更”逻辑错误。代码实现中未引入分布式锁（Distributed Lock），且数据库更新操作（UPDATE）未采用乐观锁或原子减机制，导致多个请求同时判定“退款状态 = 待退款”为真，从而触发了多次余额扣减/增加逻辑。
- **策略借鉴**: 
  1. **分布式锁机制**: 在退款业务逻辑入口处，以 `order_id` 为 Key 实施 Redis 分布式锁，确保同一订单在同一时刻仅能由一个线程处理。
  2. **数据库事务约束**: 使用数据库原生语句进行幂等性保护，例如 `UPDATE orders SET status='refunded' WHERE id=123 AND status='paid'`，利用更新行数判定操作合法性。
  3. **接口幂等性设计**: 强制要求退款请求携带全局唯一的 `request_id`（幂等令牌），并在后端持久化记录处理结果，重复请求直接返回缓存的执行状态。
