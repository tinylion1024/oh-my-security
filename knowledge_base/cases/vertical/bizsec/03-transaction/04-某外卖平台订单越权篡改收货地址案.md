# 某外卖平台订单越权篡改收货地址案
- **所属领域**: 业务安全
- **威胁类型**: 不安全的直接对象引用（IDOR）
- **事件回顾**: 某外卖巨头在更新 API 架构时，忽略了订单详情修改接口的鉴权逻辑。攻击者发现只需通过修改 HTTP 请求中的 `order_id`，即可在未经授权的情况下修改其他用户的收货地址和联系电话。该漏洞被黑产利用，用于拦截高价值订单，涉及数千个配送中的订单，造成大量投诉和物流损失。
- **Red View (利用路径)**: 
  1. **漏洞发现**: 攻击者在修改个人订单地址时，观察到请求参数中包含 `order_id=10001` 和 `user_id=888`。
  2. **越权测试**: 尝试保持登录状态，将 `order_id` 修改为 `10002`，发现后端成功返回了属于另一用户的地址信息。
  3. **自动化利用**: 编写简单脚本，利用 Burp Intruder 模块遍历订单 ID，并将所有收货地址批量修改为黑产控制的收货网点。
  4. **订单劫持**: 快递员按照被篡改后的地址送货，导致商品被攻击者冒领。
- **Blue View (防御缺失)**: 后端接口仅验证了用户的登录凭证（Session/Token），但未在业务逻辑层实现“对象级授权校验”。即未确认当前登录的用户 `user_id` 是否真的是该 `order_id` 的所有者。
- **策略借鉴**: 
  1. **强制性对象属主校验**: 每一个对资源的读/写操作，后端必须首先执行 `SELECT owner_id FROM orders WHERE id=?` 并对比当前 Token 中的用户身份。
  2. **使用 UUID 替代自增 ID**: 在 URL 或前端交互中使用无序、不可预测的 UUID 标识资源，增加攻击者盲目遍历的技术成本和门槛。
  3. **风险感知引擎**: 在 API 网关层监控短时间内针对不同对象 ID 的频繁修改请求，一旦触发阈值立即强制二次验证或拉黑 IP。
