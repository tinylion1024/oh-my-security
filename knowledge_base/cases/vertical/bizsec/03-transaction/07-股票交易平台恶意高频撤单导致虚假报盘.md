# 股票交易平台恶意高频撤单导致虚假报盘
- **所属领域**: 业务安全
- **威胁类型**: 交易频率滥用/市场操纵
- **事件回顾**: 在一次市场剧烈波动期间，某小型证券公司交易系统遭遇大量自动化交易机器人的攻击。攻击者利用极短时间内的“挂单-立即撤单”操作，产生了数百万次无效请求，导致系统撮合引擎过载，真实投资者的委托无法正常成交。此举旨在制造虚假买盘意图，误导市场走势。
- **Red View (利用路径)**: 
  1. **工具部署**: 使用基于 FIX 协议的专业交易 API 或 Python 量化框架。
  2. **频率探测**: 测试交易系统对单个账户每秒请求数（TPS）的硬限制。
  3. **注入垃圾流量**: 针对同一股票，毫秒级交替发送挂单请求和撤单请求，人为堆积未成交委托额。
  4. **收割获利**: 诱导散户跟随挂单后，攻击者在撤单瞬间反向抛售，利用系统延迟完成收割。
- **Blue View (防御缺失)**: 系统缺乏对“撤单/报单比”的监控，且 API 接入点未部署高性能的流量整形（Rate Limiting）机制。业务逻辑未对交易动机进行分析，无法识别出非理性的撤单行为。
- **策略借鉴**: 
  1. **阶梯式流量控制**: 根据账户等级和资产规模，在 API 网关实施动态限流，禁止异常高频的非成交请求。
  2. **撤单率惩罚机制**: 引入“报单-成交比”考核指标，对撤单率超过阈值（如 90%）的账户增加交易手续费或强制进入 10 分钟冷静期。
  3. **撮合引擎优先保护**: 在后端建立优先级队列，优先处理真实成交意图明显的请求，隔离频繁变动的影子挂单流量。
