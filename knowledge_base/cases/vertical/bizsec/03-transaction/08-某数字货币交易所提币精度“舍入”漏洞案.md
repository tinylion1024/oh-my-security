# 某数字货币交易所提币精度“舍入”漏洞案
- **所属领域**: 业务安全
- **威胁类型**: 浮点数精度截断/数值溢出
- **事件回顾**: 某中小型数字货币交易所被发现其以太坊（ETH）提币逻辑存在精度处理错误。攻击者发现通过提交特定位数的微量提币请求，系统在计算手续费后的舍入逻辑中会出现“负溢出”，导致提出的实际资产远高于其账面余额。黑产团伙通过自动化脚本利用此漏洞提现了价值 500 万美元的代币。
- **Red View (利用路径)**: 
  1. **精度测试**: 攻击者针对 18 位小数的智能合约代币，通过 Burp 测试不同的小数点后位数对余额变动的影响。
  2. **溢出触发**: 发现当余额小于提币量，但提币量在特定极小值（如 `0.000...0001`）时，由于舍入误差，后端校验失效。
  3. **脚本化榨取**: 使用 Web3.js 编写批量调用脚本，每分钟发起数千笔微额提币请求，积少成多。
  4. **混币洗钱**: 通过 Tornado Cash 等混币器将非法提现的资产洗白并离线存储。
- **Blue View (防御缺失)**: 交易所后端使用了不安全的双精度浮点数（Double）处理金融资产，导致极小数值运算时发生截断。核心余额校验逻辑在减法操作后未对“负值结果”进行二次强制报错，而是默认转为了极大正整数（溢出）。
- **策略借鉴**: 
  1. **采用高精度数值库**: 严禁在交易中使用浮点数，必须统一使用 `BigDecimal` 或 `BigInt` 类型进行资产计算。
  2. **零容忍余值校验**: 每次提币前、后必须执行 `Balance_Old - Amount - Fee >= 0` 的强一致性断言，结果不成立立即熔断事务。
  3. **异常流水监控**: 对连续发生的小额提现行为实施“累计额度告警”，及时发现并拦截针对精度漏洞的蚂蚁搬家式攻击。
