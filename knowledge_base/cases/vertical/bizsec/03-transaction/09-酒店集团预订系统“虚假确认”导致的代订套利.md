# 酒店集团预订系统“虚假确认”导致的代订套利
- **所属领域**: 业务安全
- **威胁类型**: 异步响应欺骗/业务流程不连贯
- **事件回顾**: 某全球连锁酒店发现大批通过代理商预订的订单出现欠款。调查发现，攻击者利用了酒店预订 API 的“乐观同步”逻辑。攻击者在发起预订请求后，通过拦截返回包将其状态修改为“Confirmed”，诱导前端展示成功。此时，代订团伙以此欺骗消费者支付房费，而实际上酒店后端由于后续支付超时并未真正锁定房间。
- **Red View (利用路径)**: 
  1. **协议破解**: 攻击者通过分析移动端 App 的 HTTPS 流量，识别出预订确认的 JSON 响应结构。
  2. **响应改包**: 利用 HTTP 代理工具，将 API 返回的 `status: "pending_payment"` 修改为 `status: "success"`。
  3. **虚假凭证生成**: 诱导前端显示确认号和二维码，截图发给受害顾客证明“预订成功”。
  4. **非法获利**: 在顾客确认收货付款后，代订人消失，顾客到店后发现订单其实已因未支付而被自动取消。
- **Blue View (防御缺失)**: 核心业务流程严重依赖前端的状态展示。酒店系统未对第三方代理商进行实时的强制状态对账，且确认凭证生成逻辑未进行数字签名验证，导致虚假截图和改包报文极易通过肉眼审查。
- **策略借鉴**: 
  1. **状态驱动强对账**: 严禁前端渲染逻辑决定业务闭环，订单状态必须以实时查询后端主库结果为准，并增加数字签名水印。
  2. **支付强耦合预订**: 只有当支付回调（Callback）成功后，系统才能下发预订确认号（Booking ID），防止出现凭证先行、资金滞后的漏洞。
  3. **建立渠道信誉体系**: 对第三方代理商实施保证金制度，对其 API 调用行为进行偏离度分析，发现大规模预订未支付行为立即冻结接口权限。
