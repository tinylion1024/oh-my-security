# 某游戏平台充值绕过：利用“订单快照”修改商品 ID
- **所属领域**: 业务安全
- **威胁类型**: 逻辑注入/订单内容篡改
- **事件回顾**: 某知名网游平台充值接口存在“阴阳订单”漏洞。攻击者通过在购物车环节添加低价道具（如 1 元新手包），在跳转支付网关瞬间，修改请求中的商品 ID 为高价资源（如 648 元虚拟大礼包）。由于后端仅核对了支付金额是否为 1 元，却在发货逻辑中错误读取了篡改后的商品 ID，导致平台蒙受巨大损失。
- **Red View (利用路径)**: 
  1. **漏洞扫描**: 测试下单全链路，寻找商品 ID（`sku_id`）与支付金额（`total_fee`）的对应校验点。
  2. **快照干扰**: 发现系统在创建订单后，允许再次通过 API 修改订单详情以增加“备注”，但未重算金额。
  3. **数据劫持**: 下单后，在支付完成前，通过 POST 请求修改订单内的 `sku_list`，注入高价值物品。
  4. **支付核算**: 支付 1 元后，系统检索该订单内容，按篡改后的清单发货。
- **Blue View (防御缺失)**: 典型的“支付金额与发货内容脱钩”。系统在支付完成后，未根据“支付金额反向推算商品价格”，而是盲目信任了可能已被二次修改的订单快照内容，且订单修改权限未在支付锁定状态下关闭。
- **策略借鉴**: 
  1. **订单内容不可变性**: 一旦订单进入“待支付”状态，必须立即锁定所有核心字段（ID、数量、价格），禁止任何形式的更新。
  2. **发货前二次核价**: 发货服务在调用库存接口前，必须执行 `Sum(SKU_Price) == Paid_Amount` 的最终校验逻辑。
  3. **支付签名绑定**: 在跳转第三方支付时，将商品简述和 ID 加密存储在支付网关的自定义字段中，并在回调时回传校验，确保支付物与实物一致。
