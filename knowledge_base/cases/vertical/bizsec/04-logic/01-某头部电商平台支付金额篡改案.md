# 某头部电商平台支付金额篡改案
- **所属领域**: 业务安全
- **威胁类型**: 支付逻辑漏洞 / 参数篡改
- **事件回顾**: 2023年双十一大促期间，某头部电商平台遭到严重黑产攻击。攻击者利用支付接口的业务逻辑设计缺陷，将单价高达上万元的3C电子产品（如高端显卡、最新款手机）以0.01元的价格成功下单并完成支付，直接导致平台面临超200万元人民币的直接经济损失，后续追损与法务成本极高。
- **Red View (利用路径)**: 
  1. 攻击者使用 **Burp Suite** 代理拦截了手机App端发起“确认下单”时的HTTP POST请求。
  2. 在抓包数据中，发现订单总价由前端参数 `total_price` 直接传递。
  3. 攻击者将 `total_price: 12999.00` 篡改为 `total_price: 0.01`，并放行请求。
  4. 支付网关直接采信了被篡改的金额，唤起第三方支付0.01元，完成支付闭环。
- **Blue View (防御缺失)**: 
  该防线被击穿的根本原因在于“过度信任客户端输入”。后端支付微服务在接收到创建订单及发起支付的请求时，完全依赖前端传递的金额参数，未提取 `product_id` 与后台数据库/缓存中的商品权威标价进行二次比对与总价核算，缺乏最基本的输入校验与状态一致性检查。
- **策略借鉴**: 
  1. **后端权威计价**: 订单金额必须由后端服务器根据商品库ID、购买数量、有效优惠券等要素独立计算得出，绝对禁止从客户端直接接收金额参数。
  2. **关键参数防篡改校验**: 对前后端交互的关键业务参数（如商品ID、数量、时间戳）实施强哈希签名（如 HMAC-SHA256），任何参数修改都会导致签名失效。
  3. **支付与订单对账流**: 支付回调环节必须强制对比“第三方支付网关实际到账金额”与“后端系统原始订单应付金额”，不一致则立即挂起订单并阻断发货。
