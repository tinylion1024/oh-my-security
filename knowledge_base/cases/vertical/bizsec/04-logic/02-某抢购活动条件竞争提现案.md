# 某抢购活动条件竞争提现案
- **所属领域**: 业务安全
- **威胁类型**: 并发漏洞 / 条件竞争 (Race Condition)
- **事件回顾**: 某潮牌社区在发布限量版球鞋抽签与直接购买活动中，遭遇专业羊毛党的毁灭性打击。原本每人限购1双的稀缺球鞋，某单一黑产账号却成功下发了50个有效订单，并在二级市场高价倒卖。此事件不仅造成商品被垄断，更导致大量真实用户的投诉，平台声誉受损。
- **Red View (利用路径)**: 
  1. 攻击者正常登录并进入抢购页面，在点击“提交订单”时使用拦截工具阻断请求。
  2. 将该有效请求导入 **Turbo Intruder** (Burp Suite插件) 或使用 Python `asyncio` 编写的高并发脚本。
  3. 利用 HTTP Keep-Alive 管道化或并发线程池，在同一毫秒内向服务器密集发送100个相同的抢单请求。
  4. 服务器数据库事务隔离级别处理不当，导致同时有50个请求读取到“库存>0”并绕过了“限购1件”的校验，最终生成50个订单。
- **Blue View (防御缺失)**: 
  典型的资源竞争漏洞。后端在处理购买逻辑时，分为“查询剩余购买次数/库存”和“更新扣减库存”两步，但这两步操作之间缺乏原子性保障。在高并发场景下，多个线程同时越过了“查询”防线，随后各自执行了“扣减”操作，导致限购逻辑完全失效。
- **策略借鉴**: 
  1. **引入分布式锁**: 在处理高并发扣减资源前，基于 Redis (`SETNX`) 或 Zookeeper 引入分布式锁，锁定用户ID或商品ID，确保同一用户/同一商品的购买逻辑串行执行。
  2. **数据库乐观锁/悲观锁**: 在数据库层面，扣减库存的 SQL 语句必须带有条件（如 `UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`），利用乐观锁版本号控制并发。
  3. **漏斗式限流**: 在网关层或业务前置层接入 Nginx Lua 限流或令牌桶算法，对单用户/单IP的瞬时并发请求进行极严苛的丢弃处理。
