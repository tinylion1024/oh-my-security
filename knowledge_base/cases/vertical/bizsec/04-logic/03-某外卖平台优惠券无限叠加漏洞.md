# 某外卖平台优惠券无限叠加漏洞
- **所属领域**: 业务安全
- **威胁类型**: 业务逻辑漏洞 / 状态流转异常
- **事件回顾**: 某头部外卖平台在节假日大促中发放了面额为10元的无门槛通用代金券。黑客利用接口漏洞，在单笔订单中重复使用了同一张优惠券多达上百次。不仅使得上百元的订单变为了“0元霸王餐”，甚至触发了平台的退款逻辑，被反向薅取大量余额。该手法在黑产群广泛传播，导致数百万资金流失。
- **Red View (利用路径)**: 
  1. 攻击者抓取提交订单时的 POST 包，分析发现优惠券参数以 JSON 数组形式传递，如 `{"coupon_ids": ["C_12345"]}`。
  2. 使用文本编辑器或 **Postman** 直接篡改请求体，将同一个优惠券ID复制多份：`{"coupon_ids": ["C_12345", "C_12345", "C_12345", ...]}`。
  3. 后端循环遍历数组计算折扣，成功将同一张10元优惠券计算了多次，总价被扣减为负数或零。
- **Blue View (防御缺失)**: 
  后端开发人员在处理优惠券计算逻辑时，仅循环遍历了传入的数组进行累加减扣，但未对数组内的元素进行去重校验。更致命的是，没有在数据库级别限制“一单仅限一券”或校验“优惠券是否已被消费”，状态机设计存在严重缺陷。
- **策略借鉴**: 
  1. **入参严格去重与校验**: 在业务入口层，对传入的优惠券数组或列表实施严格去重，并硬性限制单次订单可使用的最大优惠券数量。
  2. **优惠券状态原子化核销**: 使用事务包裹计价与核销过程。在计算前校验优惠券状态为“未使用”，在下单成功后立刻将其更新为“已锁定/已使用”，防止同一状态被多次读取。
  3. **订单金额兜底保护**: 计价模块最后必须增加边界检查：无论如何叠加优惠，订单最终支付金额不得低于0.01元，且绝不能因优惠超额引发账户余额反向增加。
