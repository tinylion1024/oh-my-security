# 某SaaS平台密码找回步骤绕过案
- **所属领域**: 业务安全
- **威胁类型**: 状态机绕过 / 权限认证漏洞
- **事件回顾**: 一家为中小企业提供CRM服务的SaaS平台遭遇定向攻击，多名企业高管的账户被恶意接管，大量敏感客户资料及商业合同被窃取。黑客并未获取目标高管的手机验证码或邮箱，而是通过绕过密码重置流程的中间步骤，直接强行重置了高管密码。
- **Red View (利用路径)**: 
  1. 攻击者在密码找回页面输入受害者的账号，进入第一步（发送短信验证码）。
  2. 攻击者随意输入一个错误的验证码并提交，同时开启 **Burp Suite** 的 Response 拦截功能 (Intercept Server Responses)。
  3. 服务端返回了 `{"status": "error", "msg": "验证码错误"}`。攻击者将该响应篡改为 `{"status": "success", "msg": "验证通过"}` 并发给浏览器前端。
  4. 前端页面收到 success 标志后，直接跳转到第二步（设置新密码的表单），攻击者输入新密码后提交，服务端未再校验前置步骤，密码重置成功。
- **Blue View (防御缺失)**: 
  该漏洞的本质是服务端状态机断层。服务端完全依赖前端路由来控制流程跳转，在最终的“修改密码”API接口处，并没有验证当前用户是否真正完成了“验证码校验”这一前置步骤。前端响应篡改使得页面轻松绕过了表面限制。
- **策略借鉴**: 
  1. **服务端会话状态跟踪**: 密码找回必须是连贯的服务端状态机。验证码校验通过后，服务端应在当前 Session 中生成并下发一个高强度的、短有效期的 `reset_token`。
  2. **终端接口严格鉴权**: 在最后一步提交新密码的API接口中，必须强制校验上一步颁发的 `reset_token`，且该 Token 只能使用一次，一旦重置成功或超时立即销毁。
  3. **禁止依赖客户端状态**: 绝不能通过 HTTP 响应中的 success/fail 状态码来决定后续敏感操作的放行与否，所有安全决策逻辑必须100%在后端闭环。
