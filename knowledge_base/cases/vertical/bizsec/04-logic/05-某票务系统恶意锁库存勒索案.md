# 某票务系统恶意锁库存勒索案
- **所属领域**: 业务安全
- **威胁类型**: 恶意占座 / 逻辑设计缺陷
- **事件回顾**: 某当红歌手演唱会门票开售瞬间，官方票务平台显示数万张门票在3秒内“售罄”。然而，长达两小时内几乎没有真实用户完成支付。随后平台收到黑客的勒索邮件，对方利用脚本生成了海量未支付订单将库存全部锁死。因支付超时机制过长，真实的售票业务被彻底瘫痪。
- **Red View (利用路径)**: 
  1. 攻击者分析平台购票API，编写定制化的 Python requests 脚本，或使用 **Selenium** 配合代理池。
  2. 脚本使用大量自动注册或撞库得来的“肉鸡”账号，批量调用 `/api/ticket/reserve` 接口。
  3. 系统采用了“下单即减库存”的逻辑，攻击者疯狂下单但不进入支付网关。
  4. 每单锁仓时间设定为系统默认的2小时，攻击者通过轮询循环锁单，导致场次门票长期处于“不可售”状态，实施业务级拒绝服务攻击（Denial of Business）。
- **Blue View (防御缺失)**: 
  库存管理逻辑存在明显的不合理分配。“下单即扣减/锁定”在稀缺商品场景下极易被恶意利用。同时，订单未支付自动取消的超时时间长达120分钟，且平台缺乏对高频下单账号的风险识别与设备指纹追踪，导致黑产可以低成本长期瘫痪业务。
- **策略借鉴**: 
  1. **动态库存锁定策略**: 核心稀缺资源应当改为“支付成功后减库存”，或者采取“极短锁定时间”（如订单创建后仅锁库 3~5 分钟），超时立即释放。
  2. **引入前置业务风控**: 在下单前强制要求完成复杂的滑动验证码或行为验证码验证；结合设备指纹与风控引擎，对高危IP段、模拟器设备、新注册黑号进行阻断。
  3. **阶梯式限购与保证金**: 对稀缺票务引入账号信任等级隔离，新账号或低活跃账号需要支付一定比例的防恶意违约金或仅能在第二梯队参与抢票。
