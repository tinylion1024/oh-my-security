# 某物流平台水平越权改单案
- **所属领域**: 业务安全
- **威胁类型**: 越权访问 (IDOR / BOLA)
- **事件回顾**: 某同城急送物流平台爆发大规模快递失踪事件。受害者发现自己通过平台寄出的昂贵物品（如黄金、数码产品），在派送途中被人为修改了收件地址，最终被不明人士签收。经查，黑客利用了该平台的订单修改接口缺陷，非法篡改了不属于自己的高价值订单目的地。
- **Red View (利用路径)**: 
  1. 攻击者作为普通用户登录平台，发起一个合法的快递订单，抓取修改收件地址时的 HTTP PUT 请求。
  2. 发现接口格式类似于 `/api/v1/orders/1048576/address`，其中 `1048576` 是连续递增的整型订单号。
  3. 攻击者将该请求发送到 **Burp Intruder**，设置 `order_id` 为变量进行暴力遍历（例如从 1048000 遍历到 1049000）。
  4. 虽然攻击者没有这些订单的所有权，但服务器依然返回了 `200 OK`，成功将大量他人的订单地址修改为黑客指定的收货点。
- **Blue View (防御缺失)**: 
  典型的不安全直接对象引用 (IDOR) 或失效的破坏对象级别授权 (BOLA)。控制器层面的代码仅仅验证了“当前发起请求的Token是否有效（是否登录）”，却致命地遗漏了业务鉴权核心的一环：“当前登录的用户ID，是否等于数据库中待修改的 order_id 对应的 owner_id”。
- **策略借鉴**: 
  1. **强制数据级鉴权**: 在所有执行增删改查（CRUD）操作的服务端代码中，必须落实 RBAC 或 ABAC 鉴权。查询和更新数据库时，必须强制带上当前用户身份条件（例：`UPDATE orders SET address=? WHERE id=? AND user_id=?`）。
  2. **使用间接对象引用**: 废弃连续递增的整型ID作为暴露给外部的标识符。全面改用难以预测的 UUID v4 或分布式雪花算法ID，从根本上防止攻击者通过遍历ID来探测其他订单。
  3. **自动化越权扫描**: 将自动化 IDOR/BOLA 扫描用例纳入 CI/CD 流程，利用 AuthMatrix 等工具，在测试阶段确保A账号的Token无法操作B账号的资源。
