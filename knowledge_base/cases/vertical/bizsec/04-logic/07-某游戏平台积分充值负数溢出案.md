# 某游戏平台积分充值负数溢出案
- **所属领域**: 业务安全
- **威胁类型**: 参数边界异常 / 整数溢出逻辑漏洞
- **事件回顾**: 某大型多人在线游戏平台上线了全新的积分商城，允许玩家使用法定货币兑换虚拟点券。上线第二天，运营团队发现平台财务账目出现巨额亏空。攻击者非但没有花钱，反而使得自己的第三方支付钱包里收到了平台主动“退款”的数万元人民币，同时游戏内还获得了海量稀有道具。
- **Red View (利用路径)**: 
  1. 攻击者登录积分商城，选择购买数量为 1 个的道具，点击结算。
  2. 使用 **Charles Proxy** 或 Burp Suite 拦截购买请求包。
  3. 攻击者将请求中的数量参数 `quantity: 1` 篡改为负数，例如 `quantity: -1000`。
  4. 后端计费逻辑为 `total_cost = price * quantity`。假设单价为100元，后台计算出应付金额为 `-100000` 元。
  5. 支付模块未对负数进行拦截，触发了底层支付接口的退款/结算逻辑，反而向攻击者账户转账 100000 元。
- **Blue View (防御缺失)**: 
  开发人员缺乏防御性编程思维，在处理与钱、数量相关的关键变量时，没有验证数字的合法边界。仅仅依赖于前端 UI 控制只能输入正整数是无效的，服务端未针对变量实施类型安全检查与 `quantity > 0` 的硬性断言，引发了业务逻辑反转。
- **策略借鉴**: 
  1. **严格的边界输入验证**: 在应用控制层，所有涉及数量、金额、时长的数值输入，必须通过白名单校验，强制断言必须为正整数且不得超过合理的最大阈值极限。
  2. **绝对金额安全断言**: 在将支付请求发送给第三方网关或执行余额扣减前，执行二次金额校验。订单应付金额必须严格大于 0（或等于免费活动的 0），一旦发现计算金额为负数，立即抛出异常并触发安全告警。
  3. **防篡改与签名机制**: 前后端通信涉及财务的核心参数，推荐引入带秘钥的哈希签名（Signature），严防中间人代理抓包篡改单价与数量变量。
