# 某企业OA系统财务审批越级案
- **所属领域**: 业务安全
- **威胁类型**: 审批流绕过 / 强制浏览漏洞
- **事件回顾**: 某中型互联网企业的内部办公自动化（OA）系统遭遇内部员工恶意利用。一名基层业务人员虚构了多笔高达数十万的报销单。按规定，该额度需直属总监和CFO双重审批，但该员工通过漏洞直接越过了所有审批节点，使得报销单瞬间变为“已打款”状态，给公司造成巨额资金流失。
- **Red View (利用路径)**: 
  1. 攻击者利用低权限账号正常发起一笔小额报销，并观察网页交互。
  2. 通过浏览器 F12 开发者工具的 Network 面板，发现最终出纳打款操作调用的 API 是 `POST /api/v1/finance/disburse`，参数仅为 `expense_id`。
  3. 攻击者伪造了大额报销单并获得了新的 `expense_id`，但该单据尚处于“待总监审批”状态。
  4. 攻击者直接使用 **cURL** 或 Postman，强制访问（Force Browsing）打款接口并附带大额单据的 ID。
  5. 接口未校验前置状态，直接调用财务网关执行了放款。
- **Blue View (防御缺失)**: 
  系统存在严重的强制浏览漏洞与状态机缺失。后端的 `/disburse` 接口只验证了用户是否为OA系统的合法登录者，却既没有校验该操作者是否具有财务人员的角色权限，也没有校验该报销单在有限状态机（FSM）中是否已经走完了全部前置的 approval 状态。
- **策略借鉴**: 
  1. **严格的有限状态机(FSM)校验**: 业务流转必须依赖后端状态机。在执行最终打款或终态操作时，必须强校验当前单据的 `status` 是否精确等于 `CFO_APPROVED` 等前置合法状态。
  2. **基于角色的细粒度访问控制(RBAC)**: 所有敏感的动作端点（如审批、打款、驳回）必须映射到具体的角色权限。低权限员工访问财务专有接口应被立即拦截并记录审计日志。
  3. **解耦工作流引擎**: 建议引入标准化的工作流引擎（如 Flowable / Camunda），所有节点流转由独立引擎接管，杜绝在业务代码中散落且不严谨的 API 状态校验。
