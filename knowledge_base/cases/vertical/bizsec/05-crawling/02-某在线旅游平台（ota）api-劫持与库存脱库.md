# 某在线旅游平台（OTA）API 劫持与库存脱库
- **所属领域**: 业务安全
- **威胁类型**: 接口自动化滥用
- **事件回顾**: 某头部 OTA 平台发现其机票库存查询接口被某三方聚合搜索平台非法调用。攻击者绕过 Web 页面，直接通过逆向得到的 App 接口进行海量并发查询。此举不仅造成了巨大的服务器带宽资源浪费，更导致航司侧因异常占座频率触发熔断机制，严重影响正常用户订票流程。
- **Red View (利用路径)**: 
  1. **App 逆向工程**: 使用 Frida 钩取（Hook）App 内存，通过内存脱壳获取加密函数及签名（Sign）算法。
  2. **协议重放攻击**: 利用 Burp Suite 抓包获取鉴权 Token，并根据逆向出的算法构造伪造请求头，实现脱离 App 环境的接口调用。
  3. **自动化批量抓取**: 编写 Python 脚本调用加密库，配合多线程异步 I/O 模块，实现对全球航线库存的准实时监控。
- **Blue View (防御缺失)**: 
  1. **硬编码密钥**: 签名算法所用的 Secret Key 被硬编码在客户端，未经过混淆处理，极易被逆向获取。
  2. **缺乏 TLS Pinning**: 未实施双向证书校验（SSL Pinning），攻击者通过安装根证书即可轻松截获 HTTPS 明文流量。
  3. **API 鉴权粒度粗糙**: 接口权限未与用户账户深度绑定，仅依赖静态 Token，且 Token 有效期过长。
- **策略借鉴**: 
  1. **强化代码混淆与白盒加密**: 对签名逻辑使用 LLVM 加固，并采用白盒加密方案保护核心密钥不被内存读取。
  2. **实施强制 TLS 双向校验**: 结合安全键盘和原生证书绑定，阻断所有中间人代理抓包尝试。
  3. **引入多维 API 安全治理**: 监控 API 调用的熵值变化，对非正常逻辑的 API 遍历行为实施“蜜罐干扰”——返回虚假库存数据。
