# 支付接口防重放漏洞导致双花攻击
- **所属领域**: 业务安全
- **威胁类型**: 交易重放/资金双花
- **事件回顾**: 某头部第三方支付机构在双十一高峰期间，其快捷支付API遭遇大规模重放攻击。攻击者利用网络延迟与网关处理机制的漏洞，对同一笔真实订单的支付请求进行并发重放，导致平台在短时间内对多个商户清算同一笔资金，直接经济损失达数百万人民币。
- **Red View (利用路径)**:
  1. 攻击者使用Burp Suite抓取正常支付成功的请求包，保留有效签名、Token及订单号。
  2. 编写Python并发脚本（利用gevent或asyncio），将该请求包在毫秒级内向支付网关发起数百次高并发重放。
  3. 绕过前端或网关的基础防重放校验后，后端微服务由于数据库锁机制不严，同时处理多个请求并均判断为“未支付”状态。
  4. 最终完成多个状态更新及账户余额扣减（或增加），实现“一笔钱支付多次”。
- **Blue View (防御缺失)**: 防线击穿的核心在于网关层Nonce（随机数）防重放缓存失效以及核心支付链路缺乏分布式悲观锁。业务层过于信任Redis缓存，但在高并发下Redis响应延迟，导致校验通过；同时底层数据库更新时未使用`UPDATE ... WHERE status='PENDING'`的乐观锁机制。
- **策略借鉴**:
  1. **网关层强防重放**: 引入时间戳+Nonce的强校验机制，基于Redis实现严格的SETNX原子操作防止Nonce复用。
  2. **核心账务乐观锁/悲观锁**: 在数据库状态流转操作时，必须使用版本号（Version）乐观锁或状态条件控制，确保资金状态流转的唯一性。
  3. **幂等性设计**: 确保所有支付、清结算接口具备严格的幂等性，同一个业务订单号无论重试多少次，产生的影响和第一次调用一致。
