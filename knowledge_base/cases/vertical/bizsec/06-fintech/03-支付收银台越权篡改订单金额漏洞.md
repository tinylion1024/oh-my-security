# 支付收银台越权篡改订单金额漏洞
- **所属领域**: 业务安全
- **威胁类型**: 业务逻辑漏洞/越权篡改
- **事件回顾**: 某互联网券商的理财产品购买入口，因前后端金额校验逻辑分离，导致攻击者能够以0.01元购买价值10万元的理财产品。漏洞被羊毛党在暗网论坛曝光后，数小时内被利用数百次，险些造成巨额资金流失，幸被对账系统拦截部分交易。
- **Red View (利用路径)**:
  1. 攻击者正常下单一个理财产品，在支付收银台页面选择支付通道并拦截该POST请求包。
  2. 发现请求体中包含`order_amount=100000.00`及`product_id=123`的明文参数。
  3. 使用抓包工具篡改`order_amount=0.01`后将请求放行至服务器。
  4. 后端接口虽然核对了订单号的有效性，但直接信任了客户端传来的支付金额，调用三方支付完成0.01元扣款后，在业务系统中将原10万元订单标记为“已支付”。
- **Blue View (防御缺失)**: 典型的“信任客户端输入”安全设计缺陷。后端支付回调处理逻辑中，只比对订单状态，没有拉取数据库中原始订单的实际应付金额与支付网关返回的实际支付金额进行强制一致性校验。
- **策略借鉴**:
  1. **金额参数不可见**: 支付接口绝不应由客户端传递金额参数。客户端只需传递订单ID（或Token），后端根据订单ID自查数据库计算金额。
  2. **双向交叉校验**: 在接收到第三方支付平台的异步回调（Notify）时，必须校验`回调金额 == 数据库订单金额`，否则挂起订单并报警。
  3. **数据签名与防篡改**: 如果特殊场景必须传递敏感参数，需通过HMAC-SHA256结合服务端私钥对整个请求Payload进行签名（Sign）验证。
