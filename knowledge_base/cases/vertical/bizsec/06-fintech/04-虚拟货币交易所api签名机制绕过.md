# 虚拟货币交易所API签名机制绕过
- **所属领域**: 业务安全
- **威胁类型**: 接口鉴权绕过/资产盗取
- **事件回顾**: 一家出海加密货币交易平台遭遇严重API安全事故。由于其开放给量化交易用户的REST API在鉴权签名算法上存在规范化（Normalization）缺陷，攻击者成功构造出具有相同签名的伪造提币请求。导致热钱包中约200万美元的USDT被未授权转出。
- **Red View (利用路径)**:
  1. 攻击者注册普通账户并生成API Key/Secret，对平台的签名算法进行白盒审计（或通过前端JS逆向获取逻辑）。
  2. 发现其请求参数拼接逻辑存在“参数注入”缺陷。例如，在计算签名时没有使用分隔符：`param1=a`与`param2=bc`拼接为`abc`，而`param1=ab`与`param2=c`拼接也为`abc`。
  3. 攻击者抓取合法的低权限请求（如查询余额），提取有效的签名值。
  4. 构造包含提币指令（如`withdraw_amount`、`address`）的请求包，巧妙调整参数名和值，使得拼接后的字符串与原请求完全一致，携带原签名通过网关校验，完成盗币。
- **Blue View (防御缺失)**: API网关的鉴权中间件在计算HMAC签名时，对Query String或JSON Body的序列化及编码处理不规范，未对键值对进行严格隔离；且未将URL Path、HTTP Method及时间戳（Timestamp）强制纳入签名因子。
- **策略借鉴**:
  1. **严格的请求规范化**: 签名算法必须使用标准字典序对参数进行排序，并在拼接时使用明确的分隔符（如`&`和`=`），禁止直接拼接参数值。
  2. **全面的签名因子**: 签名计算必须覆盖请求的方法、URI、请求体哈希（如SHA256(Body)）、时间戳（限制过期时间如30秒），防止任何维度的重构或重放。
  3. **高危敏感接口二次校验**: 对于提现等资金流出操作，即便API鉴权通过，也应引入独立于API Key的二次认证（如短信、Google Authenticator或异步人工审核队列）。
