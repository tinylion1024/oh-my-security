# 聚合支付平台并发退款导致资金透支
- **所属领域**: 业务安全
- **威胁类型**: 条件竞争漏洞（Race Condition）
- **事件回顾**: 某面向中小微商户的聚合支付SaaS平台，由于退款业务逻辑中存在竞态条件漏洞，商户被恶意买家攻击。买家针对单笔1000元的交易，利用并发脚本同时发起数十次退款请求，由于防并发机制薄弱，系统向买家账户退回了5000元，直接穿透商户余额并导致聚合平台垫资垫损。
- **Red View (利用路径)**:
  1. 攻击者使用本人微信/支付宝在商户店铺消费1000元，生成有效订单。
  2. 利用Turbo Intruder或并发测试工具，将目标聚合平台的“申请退款”API数据包截获并设置并发。
  3. 以200线程瞬间向退款网关发送相同的退款请求。
  4. 系统数据库查询到该订单状态为“已支付，可退款”，各线程同时计算剩余可退金额。
  5. 多线程同时通过校验后，分别向底层资金系统发起退款指令，导致最终执行了5次成功退款，超过原订单金额。
- **Blue View (防御缺失)**: 典型的Check-Time-to-Use (TOCTOU)漏洞。代码在执行`SELECT`检查可退余额后，未对该资源加锁就直接进入了耗时的请求第三方支付退款流程，最后再执行`UPDATE`余额。导致并发请求读到了脏数据。
- **策略借鉴**:
  1. **数据库排他锁**: 在查询订单可退金额时，必须使用`SELECT ... FOR UPDATE`锁定该行记录，直到事务提交（退款状态更新完成）后释放。
  2. **利用Redis分布式锁**: 在退款业务处理的首层入口，根据`订单号+退款类型`作为Key设置短期分布式锁，同订单的并发退款请求直接快速失败（Fast Fail）。
  3. **总额校验与事务一致性**: 数据库层面针对`可退金额`字段设置无符号整型（Unsigned）或添加Check约束，一旦扣减出现负数直接触发数据库异常以回滚事务。
