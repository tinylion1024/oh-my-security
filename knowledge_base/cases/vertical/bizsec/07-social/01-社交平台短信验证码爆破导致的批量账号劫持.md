# 社交平台短信验证码爆破导致的批量账号劫持
- **所属领域**: 业务安全
- **威胁类型**: 账号接管 (Account Takeover)
- **事件回顾**: 某知名社交应用在一次版本更新后，其找回密码流程的接口逻辑出现退化。黑客利用该漏洞，在短短 2 小时内针对高权重靓号实施精准爆破，导致超过 500 个头部账号被恶意篡改密码并二次出售。部分账号被用于发布博彩诈骗信息，造成平台 DAU 短期下降 15%，品牌信誉严重受损。
- **Red View (利用路径)**: 
    1. **目标锁定**: 使用 Python 脚本通过平台搜索接口，筛选出未设置二步验证（2FA）的高粉丝量用户名。
    2. **请求捕获**: 使用 Burp Suite 拦截“找回密码”请求，定位验证码校验接口 `/api/v1/auth/verify_code`。
    3. **逻辑绕过**: 发现验证码为 4 位纯数字，且服务端未对验证码校验失败次数进行有效限制。
    4. **自动化爆破**: 利用 Turbo Intruder 插件，配合多并发代理池，在 1 分钟内完成 10,000 次尝试，成功获取有效验证码并重置密码。
- **Blue View (防御缺失)**: 
    1. **速率限制失效**: WAF 仅配置了基于单个 IP 的限频，未针对“手机号+接口”维度进行联合频控。
    2. **安全配置退化**: 开发环境的“万能验证码”在灰度发布时未被清理。
    3. **校验逻辑漏洞**: 验证码校验成功后，未在服务端立即失效，导致可以被重复利用于重置密码请求。
- **策略借鉴**: 
    1. **多维频控策略**: 实施针对 IP、设备 ID、手机号的阶梯式限频，对高频错误请求实施强制图形验证码（Captcha）挑战。
    2. **闭环安全机制**: 验证码必须具备短时效性（如 3 分钟），且一旦校验通过或失败达到 5 次，必须立即在 Redis 中销毁。
    3. **二步验证强制化**: 针对粉丝量超过 10 万的敏感账号，强制开启基于硬件或 Authenticator 的多因素认证。
