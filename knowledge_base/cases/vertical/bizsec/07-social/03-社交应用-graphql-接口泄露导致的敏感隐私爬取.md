# 社交应用 GraphQL 接口泄露导致的敏感隐私爬取
- **所属领域**: 业务安全
- **威胁类型**: 数据爬取 (Scraping)
- **事件回顾**: 某职场社交巨头被安全研究员发现其 GraphQL 接口存在设计缺陷。攻击者利用该接口，无需好友关系即可获取用户的私密联系方式、简历详情及精准地理位置。该漏洞导致约 800 万名高端用户的职场隐私数据被暗网打包售卖，引发严重的公关危机和法律监管调查。
- **Red View (利用路径)**: 
    1. **接口侦察**: 监控 APP 流量，发现所有的前端请求均指向统一的 `/graphql` 端点。
    2. **内省查询利用**: 发送 `__schema` 内省查询（Introspection Query），成功获取后端所有对象、字段及其关联关系视图。
    3. **对象遍历 (IDOR)**: 发现 `UserProfile` 类型中的 `privateEmail` 字段未加权限校验。通过修改 `user_id` 参数，构造批量查询语句进行横向越权访问。
    4. **数据渗出**: 编写多线程脚本，利用分段偏移量（Offset/Cursor）系统性地爬取全量用户数据。
- **Blue View (防御缺失)**: 
    1. **内省权限未关**: 生产环境未关闭 GraphQL 的 Introspection 功能，向外部暴露了整个数据模型。
    2. **字段级权限缺失**: 开发者仅在页面渲染层面隐藏了隐私字段，而未在 GraphQL 的 Resolver（解析器）层实现基于上下文的 RBAC 校验。
    3. **查询复杂度未控**: 未限制单个 GraphQL 查询的深度和复杂度，导致攻击者可以用一个请求拉取大量关联数据。
- **策略借鉴**: 
    1. **关闭内省功能**: 生产环境必须禁用 GraphQL Introspection 且不返回详细的错误堆栈。
    2. **Resolver 强制鉴权**: 在每个敏感字段的权限解析器中，必须显式校验当前 Session 用户与被查询数据之间的“关联合法性”。
    3. **查询代价限制**: 引入 Query Complexity Analysis，对超过预设复杂度阈值的请求拒绝执行，防止大规模数据拉取。
