# 社交软件红包功能的并发竞争逻辑漏洞攻击
- **所属领域**: 业务安全
- **威胁类型**: 逻辑漏洞 (Business Logic Flaw)
- **事件回顾**: 某短视频平台在春节抢红包活动中，由于余额核减逻辑未采用分布式锁限制，被“撸羊毛”团队利用。攻击者通过高并发请求，使单个红包在余额归零前被重复领取 20-30 次。此次事件导致平台在 30 分钟内非预期损失 400 万人民币，直接导致活动紧急停服。
- **Red View (利用路径)**: 
    1. **场景发现**: 在参与抢红包时，发现点击瞬间网络波动可能导致双倍领取，怀疑存在并发竞争漏洞。
    2. **竞态条件构造**: 编写 Go 语言并发脚本，针对特定的 `packet_id`，在毫秒级时间内发送 500 个并发 `claim` 请求。
    3. **工具选型**: 使用 Go-Routines 模拟高并发，配合 Charles 捕获合法的签名参数并重放。
    4. **资金变现**: 通过平台虚拟礼品兑换渠道，迅速将非法领取的资金转移至外部银行卡。
- **Blue View (防御缺失)**: 
    1. **数据库事务缺陷**: 采用简单的 `SELECT` 然后 `UPDATE` 逻辑，未在事务中使用 `FOR UPDATE` 排他锁或乐观锁校验。
    2. **分布式同步缺失**: 在多机房部署环境下，本地缓存与中心化 Redis 之间的金额同步存在延迟，形成了“时间差”攻击窗口。
    3. **反作弊模型漏警**: 对单用户在短时间内发生数十次资产变动的极端异常未触发自动熔断机制。
- **策略借鉴**: 
    1. **分布式原子操作**: 使用 Redis 的 Lua 脚本实现红包扣减逻辑，确保 `DECRBY` 与 `EXISTS` 判断的原子性。
    2. **乐观锁机制**: 在数据库更新语句中加入版本号或余额剩余校验（例如：`UPDATE balance SET amt = amt - x WHERE user_id = y AND amt >= x`）。
    3. **实时熔断监控**: 设置业务阈值监控，一旦发现单一资产对象被异常高频读写，立即对相关账户及接口实施锁定。
