# 社交平台预览功能导致的 SSRF 漏洞攻击
- **所属领域**: 业务安全
- **威胁类型**: 服务端请求伪造 (SSRF)
- **事件回顾**: 某国际社交平台在处理用户发送的 URL 预览逻辑时存在漏洞。攻击者通过在聊天框发送精心构造的内部 IP 链接，诱导后端服务器发起内部网络探测。最终，攻击者成功通过该社交服务器作为跳板，访问了企业内网未授权的集群管理后台（K8s Dashboard）和 Redis 缓存，导致部分核心业务配置泄露。
- **Red View (利用路径)**: 
    1. **入口发现**: 发送链接时，后端会爬取网页 Meta 标签生成预览图。
    2. **内网探测**: 尝试发送 `http://127.0.0.1:80`，观察返回响应时间及预览图，确认服务端存在请求转发逻辑。
    3. **端口扫描**: 使用脚本批量替换 URL，探测内网常见的 `10.x.x.x` 网段，定位到开放了 8080 端口的内部敏感系统。
    4. **漏洞利用**: 通过 `http://internal-api.service:8080/exec?cmd=id` 等接口执行指令或拉取内网敏感元数据（Metadata API）。
- **Blue View (防御缺失)**: 
    1. **网络隔离失效**: 预览服务器（Crawler）与生产环境内网未进行物理或逻辑隔离，具有直接访问内网的能力。
    2. **目标过滤不严**: 仅通过黑名单（如禁止 127.0.0.1）进行过滤，被攻击者利用 `127.0.0.1` 的十六进制或 DNS Rebinding 轻松绕过。
    3. **协议约束缺失**: 允许 `gopher://`、`file://` 等非 HTTP 协议，导致攻击者可以利用这些协议攻击 Redis 或读取系统文件。
- **策略借鉴**: 
    1. **强制域名白名单**: 预览功能仅允许访问公网合法域名，对所有私有 IP 段（RFC1918）实施硬性屏蔽。
    2. **SSRF 防御网关**: 使用独立的、无内网访问权的 Proxy 作为爬虫出口，并限制其最大响应时间与数据大小。
    3. **协议强约束**: 在代码层限制仅支持 `http` 和 `https` 方案，并禁用所有 HTTP 重定向跟踪。
