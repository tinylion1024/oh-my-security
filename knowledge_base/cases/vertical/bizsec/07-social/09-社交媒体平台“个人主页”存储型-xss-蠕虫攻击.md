# 社交媒体平台“个人主页”存储型 XSS 蠕虫攻击
- **所属领域**: 业务安全
- **威胁类型**: 跨站脚本攻击 (XSS)
- **事件回顾**: 某校园社交应用遭遇了一次恶性 XSS 蠕虫攻击。攻击者在个人简介字段植入恶意脚本，该脚本被触发后，会自动利用当前用户的 Token 发布带有同样脚本的状态。在短短 3 小时内，全校 3 万名活跃用户中有 70% 的账号中招，导致平台首页被大量垃圾广告淹没，用户私信被批量窃取。
- **Red View (利用路径)**: 
    1. **注入点发现**: 发现“修改个人简介”接口对 `<script>` 等标签未进行服务端转义。
    2. **Payload 构造**: 编写一段紧凑的 JS 代码，功能为：a. 获取当前用户 Cookie 或 LocalStorage 中的 API Token；b. 调用“发布动态”接口发布包含 Payload 的新内容。
    3. **蠕虫启动**: 攻击者率先修改自己的简介，随后通过在大 V 账号下留言诱导他人访问其个人主页。
    4. **连锁反应**: 凡是访问过受感染主页的用户，其账号也会自动变为感染源。
- **Blue View (防御缺失)**: 
    1. **输入过滤缺失**: 后端直接将用户输入的原始字符串存储进数据库。
    2. **内容安全策略 (CSP) 未配置**: 网页未开启 `Content-Security-Policy`，允许执行来源不明的内联脚本（inline script）。
    3. **输出渲染不当**: 前端在展示用户简介时，使用了类似 `v-html` 或 `dangerouslySetInnerHTML` 的不安全渲染方式。
- **策略借鉴**: 
    1. **全局输出编码**: 在模板渲染层强制执行 HTML 实体编码，将 `<` 转换为 `&lt;`。
    2. **严格 CSP 配置**: 部署 `default-src 'self'; script-src 'self' https://trusted.cdn.com;`，严禁执行内联脚本。
    3. **安全输入库**: 使用类似 DOMPurify 的成熟库，在保存前对用户提交的富文本内容进行严格的标签白名单清洗。
