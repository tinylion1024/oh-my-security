# 卡牌手游逻辑漏洞：抽卡随机数回滚与刷取事件
- **所属领域**: 业务安全
- **威胁类型**: 协议重放与逻辑一致性漏洞
- **事件回顾**: 某二次元爆款手游在周年庆期间，被发现存在严重的抽卡逻辑漏洞。攻击者发现，通过特定的网络切断操作，可以使抽卡结果在客户端展现后不计入服务器。利用此漏洞，黑产工作室实现了“无限重刷开局号”，并产出了数千个带有多重顶级稀有角色的账号在第三方交易平台兜售。该事件导致卡牌池价值体系归零，充值系统一度被迫关闭 48 小时进行全服回滚，补偿成本及流水损失高达数百万美元。
- **Red View (利用路径)**: 
    1. **漏洞发现**: 攻击者利用 Burp Suite 拦截抽卡请求，发现客户端在发送 `draw_card` 请求后，存在一个确认包 `confirm_result`。
    2. **状态竞争**: 攻击者在抽卡包发出后立即强行关闭客户端进程或模拟断网，导致服务器已预扣除道具但未最终确认结果。
    3. **逻辑逆向**: 发现服务器在未收到确认包时，会触发特定条件下的一致性回滚逻辑，误将扣除的抽卡道具退回。
    4. **批量自动化**: 使用 Python 脚本结合模拟器，自动检测抽卡结果是否为 SSR，若不是则触发回滚，直至刷出极品奖励。
- **Blue View (防御缺失)**: 核心业务逻辑未遵循“原子性”原则。抽卡操作被拆分为非原子的多个步骤，且中间状态（Pending）处理逻辑存在缺陷。服务端过度信任客户端反馈的状态包，而非以自身的数据库事务状态为准。
- **策略借鉴**: 
    1. **事务原子化**: 将“扣除货币-生成随机数-入库奖励”整合为单一的数据库事务。只要请求发出，结果即刻生效，不再依赖客户端的任何确认反馈。
    2. **预生成随机池**: 采用服务端预先生成的随机序列，确保每个账号的抽卡结果在请求触发瞬间即已锁定。
    3. **断线重连逻辑重构**: 优化重连后的状态同步，如果发现上一次操作未完成，自动根据服务端保存的中间态进行补发，而非简单回滚。
