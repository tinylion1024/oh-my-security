# 休闲手游协议篡改：战力数值与结算篡改事件
- **所属领域**: 业务安全
- **威胁类型**: 封包篡改与结算逻辑漏洞
- **事件回顾**: 某关卡制竞技手游在推出限时冲榜活动时，发现排行榜前 100 名全是战力极低的初始账号，但关卡评分均达到系统上限。调查发现，攻击者利用修改后的封包直接向服务器上报虚假的战斗结算数据。这不仅彻底破坏了运营活动的公平性，还通过虚假结算非法获取了大量本需高额付费才能获得的强化材料。
- **Red View (利用路径)**: 
    1. **抓包与解密**: 攻击者使用 HttpCanary 拦截手机流量，发现封包虽然经过 Base64 处理，但实际并未进行有效的强加密或加盐签名。
    2. **参数识别**: 识别出结算协议中的 `score`、`kill_count` 和 `item_reward_list` 等关键字段。
    3. **非法注入**: 使用 Frida 挂钩（Hook）客户端的发送函数，在结算包发出的前一刻，将战报数据修改为非法的高值。
    4. **跳过校验**: 发现服务端未验证战斗时长（如 1 秒钟通关本需 3 分钟的关卡），直接通过修改后的封包骗取服务器发放奖励。
- **Blue View (防御缺失)**: 服务端属于“弱校验”模式，完全信任客户端上报的数值。缺乏战斗过程的回放验证，也没有对单次操作的逻辑合理性（如伤害总量是否超过角色当前战力极限）进行基础审计。
- **策略借鉴**: 
    1. **引入报文签名**: 采用 HMAC 等算法对所有业务报文进行动态签名，结合时间戳防止封包篡改与重放攻击。
    2. **关键数值校验**: 在服务端实现基础的模拟逻辑，对上报的伤害值、通关时长与账号当前的属性进行合理性判别。
    3. **随机战斗回放**: 对高价值结算请求要求上报战斗录像摘要（操作序列），服务端进行离线抽检还原，确保结算数据的真实性。
