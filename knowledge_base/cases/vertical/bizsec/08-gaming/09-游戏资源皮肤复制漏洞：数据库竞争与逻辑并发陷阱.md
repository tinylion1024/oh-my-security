# 游戏资源/皮肤复制漏洞：数据库竞争与逻辑并发陷阱
- **所属领域**: 业务安全
- **威胁类型**: 逻辑漏洞与竞争条件 (Race Condition)
- **事件回顾**: 某款支持自由贸易系统的沙盒游戏，爆出了震惊全服的“无限刷金”事件。攻击者发现，利用两个设备同时登录一个账号，并在极短的时间差内执行“道具存入仓库”与“道具交易”操作，可以使一个道具变成两个。这导致游戏内的稀有材料和极品皮肤数量激增，通货膨胀严重，官方不得不停服维护 72 小时，并手动清理数万条异常日志。
- **Red View (利用路径)**: 
    1. **逻辑探测**: 攻击者利用低网速环境（通过限制带宽工具模拟）测试游戏逻辑在延迟下的表现。
    2. **并发构造**: 编写自动化脚本，在两个不同的网络节点同时向服务器发起请求：A 节点请求“将道具移入公会仓库”，B 节点请求“发起玩家间交易”。
    3. **时序利用**: 服务器在处理 A 请求并写入数据库的过程中，由于行级锁（Row Lock）释放早于 Session 状态更新，B 请求趁虚而入，导致道具在 Session 状态中仍被视为“在背包中”。
    4. **成功翻倍**: 操作成功后，公会仓库多了一个道具，玩家手中也交易出去了一个道具，实现资产凭空翻倍。
- **Blue View (防御缺失)**: 核心资产转移操作未使用严格的全局分布式锁或数据库事务强一致性级别。Session 管理机制存在逻辑缺陷，允许在并发场景下出现陈旧数据的读取。
- **策略借鉴**: 
    1. **引入分布式锁**: 对高价值道具的操作使用 Redis 分布式锁，确保同一时刻只有一个 Session 能够操作该资产。
    2. **事务序列化**: 强制关键资产变更接口使用排他锁，并增加操作序列号校验，拒绝重复或并发的无效请求。
    3. **离线对账系统**: 建立准实时的离线资产审计流程，比对产出、消耗与存储的差值，对资产总额出现非法正增长的账号进行熔断处理。
