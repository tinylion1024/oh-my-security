# 共享单车并发漏洞无限骑行事件
- **所属领域**: 业务安全
- **威胁类型**: 并发竞争/逻辑漏洞
- **事件回顾**: 某共享单车企业遭遇规模性“薅羊毛”危机，大量用户在社交媒体分享“免费骑行”教程。用户仅需保持账户余额刚好为一次起步价（如1.5元），通过特定操作手法即可在结账时将账户刷成负数且能继续违规开锁，导致平台直接经济损失高达120万元。
- **Red View (利用路径)**: 
  1. **流量拦截**: 攻击者使用Burp Suite在手机端配置中间人代理，捕获骑行结束后点击“确认支付”的HTTP POST请求。
  2. **并发竞争注入**: 将捕获的支付请求发送至Burp Suite的Turbo Intruder模块。
  3. **条件竞争触发**: 在毫秒级窗口内，向服务器并发发送50个完全相同的扣款请求。
  4. **状态机利用**: 由于后端逻辑处理缺陷，50个请求同时读取到“余额充足（1.5元）”的状态，随后分别执行扣款，导致余额变为-73.5元。但订单状态被标记为“已支付”，用户下次扫码时，开锁验证API仅检查上一单是否支付，未校验余额是否为负。
- **Blue View (防御缺失)**: 
  1. **非原子性操作**: 余额校验和扣款写库分两步执行，且在多线程并发场景下，数据库未使用行级悲观锁（ `SELECT ... FOR UPDATE` ）。
  2. **API幂等性缺失**: 支付接口没有对同一笔订单的多次请求进行防重放（Idempotency）校验。
  3. **业务状态机断层**: 开锁接口的校验逻辑存在盲区，孤立地信任“订单状态”而忽视了全局钱包的“健康状态”。
- **策略借鉴**: 
  1. **引入分布式锁机制**: 在支付扣款的业务层使用Redis Redlock等分布式锁，确保同一个UserID或OrderID在同一时刻只能有一个线程进入扣款逻辑。
  2. **全链路接口幂等性改造**: 客户端在提交支付时必须携带唯一的Request-ID（或防重Token），网关层或业务层对已处理的ID进行拦截，防止重放攻击。
  3. **建立资金安全对账防线**: 在关键状态流转（如开锁前）增加严格的复合校验策略（历史单据结清 && 当前余额 >= 0），修复状态机断层。
