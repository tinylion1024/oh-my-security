# 代驾平台司机端越权接单事件
- **所属领域**: 业务安全
- **威胁类型**: 越权漏洞(IDOR)/API滥用
- **事件回顾**: 某互联网代驾平台近期频繁收到司机投诉，称系统派发的高客单价长途订单在自己点击“接受”前，瞬间被其他不在附近的司机抢走。经安全团队排查，几名黑灰产开发人员利用平台派单API的越权漏洞，制作了“代驾抢单外挂”，以每月300元的价格兜售给数百名违规司机，严重扰乱了平台的公平派单生态。
- **Red View (利用路径)**: 
  1. **协议逆向**: 攻击者通过反编译司机端APK，提取了平台用于广播订单的WebSocket连接参数。
  2. **订单监听**: 编写Python脚本连接WebSocket服务器，实时监听并抓取全市所有乘客发起的新订单广播，提取其中的明文 `OrderID` 和 `Estimated_Fare`。
  3. **IDOR越权利用**: 攻击者发现“接单”API（ `/driver/accept_order` ）只校验了传入的 `OrderID`，并未验证当前发起请求的 `DriverID` 是否是被系统算法选中派单的司机。
  4. **外挂强抢**: 脚本筛选出高价值订单后，直接利用购买外挂的司机 `Auth_Token` 发起接单请求，强行绑定订单。
- **Blue View (防御缺失)**: 
  1. **水平越权（IDOR）**: 典型的垂直与水平权限控制缺失。接单接口仅对订单本身进行状态变更，缺乏 `Assigned_Driver == Current_Request_Driver` 的归属权匹配逻辑。
  2. **敏感信息广播裸奔**: 广播信道未能做到数据脱敏与定向推送，导致全局订单池的敏感参数（含订单真实ID）暴露给任何合法的WebSocket订阅者。
- **策略借鉴**: 
  1. **修复基于属性的访问控制（ABAC）**: 在接单接口的底层业务逻辑中，强制加入双向校验机制。只有存在于该订单“派单候选池”且具有最高优先级的司机，其接单请求才会被数据库放行。
  2. **订单ID加密与Token化**: WebSocket广播中不再传输数据库自增或明文的 `OrderID`，改为下发具有生命周期且与特定司机绑定的 `Claim_Token`。
  3. **速度异常熔断监控**: 建立请求时差监控面板，当发现某司机“收到广播”到“发起接单”的反应时间低于人类生理极限（如小于100毫秒）时，直接触发风控阻断并封禁外挂账号。
