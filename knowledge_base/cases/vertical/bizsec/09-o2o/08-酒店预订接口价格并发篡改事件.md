# 酒店预订接口价格并发篡改事件
- **所属领域**: 业务安全
- **威胁类型**: 参数篡改/业务逻辑漏洞
- **事件回顾**: 某知名O2O差旅预订平台发生重大安全事故。黑客以每晚100元（相当于快捷酒店价格）的价格，成功预订了三亚某顶级五星级酒店价值2500元/晚的豪华海景房，并连订200晚用于黑市倒卖。直到酒店财务在周度对账时发现巨大的价格倒挂，平台才紧急叫停，但已造成数十万元的垫付损失。
- **Red View (利用路径)**: 
  1. **购物车混淆**: 攻击者在APP上同时将一家便宜的快捷酒店（100元）和一家豪华酒店（2500元）加入预订购物车。
  2. **拦截支付流**: 在结算页面，使用Postman或Burp Suite拦截前端发往后端的 `POST /api/checkout/create_order` 请求。
  3. **参数篡改替换**: 攻击者巧妙地修改JSON Payload，将原本应提交的豪华酒店 `Room_ID` 保留，但将结算总价参数 `Total_Price` 从 2500 修改为 100，并提交请求。
  4. **入库与支付**: 后端接收到篡改后的价格，直接拉起第三方支付微信/支付宝收银台。攻击者支付100元后，订单以“已支付”状态落库，并自动向酒店下发豪华房确认单。
- **Blue View (防御缺失)**: 
  1. **零信任原则崩塌**: 后端业务逻辑严重违背了“客户端数据不可信”的基本安全准则，在创建订单时直接读取并信任了前端传递过来的 `Total_Price` 字段。
  2. **订单快照校验缺失**: 缺乏购物车/订单快照的签名验证机制，导致数据在传输过程中被篡改而无法感知。
- **策略借鉴**: 
  1. **价格强后端计算**: 彻底废弃由前端传递商品价格的业务架构。结账API仅接收 `Room_ID`、`Date` 和 `User_ID`。后端必须根据这些标识直接从核心SKU数据库实时查询单价，并在内存中完成总价计算与落库。
  2. **防篡改签名机制（HMAC）**: 在复杂的O2O多重计价（如叠加各类优惠券、积分）场景下，前端发起的支付请求必须附带由后端生成的State Signature，一旦商品组合或价格参数被修改，签名即告失效。
  3. **落库异步比对预警**: 建立订单落库审计系统。在生成凭证前，异步对比“实际支付金额”与“库存标准快照指导价”，超过合理折扣阈值（如一折以下）的订单触发预警并转入人工审核。
