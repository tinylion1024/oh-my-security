# LLM API 资源耗尽攻击（Resource Exhaustion DoS）
- **所属领域**: 业务安全
- **威胁类型**: 拒绝服务攻击（DoS）
- **事件回顾**: 某提供 AI 翻译服务的 SaaS 企业遭遇了新型“Token 消耗攻击”。攻击者利用 LLM 的最大上下文窗口特性，构造了大量递归式提问（如“请逐字扩写这篇 5000 字的文章，每步必须引用先前的所有论点”），强制后台计算实例进行高强度的推理计算。这导致该企业的 GPU 集群满载运行，API 响应延迟从 2 秒飙升至 60 秒，不仅造成普通付费用户流失，还产生了高达数十万美元的云服务超额账单。
- **Red View (利用路径)**: 
  1. **提示词炸弹 (Prompt Bomb)**: 构造极其复杂的推理逻辑，诱导模型进入长输出循环。
  2. **Token 溢出探测**: 通过工具（如 `Burp Suite` 配合 Python）批量并发发送达到 Token 上限（Context Limit）的长文本请求。
  3. **并发压测**: 针对推理成本最高的端点（如复杂逻辑推导模块）发起分布式拒绝服务请求。
- **Blue View (防御缺失)**: 
  1. **缺乏配额管理**: 系统仅限制了请求次数（QPS），未对 Token 消耗量（TPM）进行分级限制。
  2. **计算超阈值未熔断**: 后台任务缺乏超时中止机制，单个恶意请求占用了长达数分钟的推理槽位。
- **策略借鉴**: 
  1. **多维流控机制**: 实施基于 Token 数量的实时限流（Rate Limiting by Token），针对异常高消耗账户实施自动隔离。
  2. **任务分级调度**: 将长文本推理与短文本响应拆分到不同队列，确保基础服务不受长任务阻塞影响。
  3. **输入长度预校验**: 在前端和网关层强制执行严格的 Context Window 限制，拒绝不合理的超长 Prompt 进入推理引擎。
