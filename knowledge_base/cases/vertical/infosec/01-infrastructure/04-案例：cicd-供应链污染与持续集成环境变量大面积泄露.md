# 案例：CI/CD 供应链污染与持续集成环境变量大面积泄露

## 事件回顾
在全球软件供应链安全史上，某知名的代码覆盖率分析工具遭受了一次堪称灾难级的攻击。该工具被成千上万家科技公司集成在其 CI/CD 流水线中。攻击者巧妙地通过修改该工具的 Bash Uploader 脚本，植入了隐蔽的恶意代码。当客户在其 CI 环境（如 GitHub Actions, Travis CI, Jenkins）中运行该脚本时，不仅执行了正常的覆盖率上传，更将流水线中极其敏感的所有的环境变量（包括云服务凭据、私钥、数据库密码）静默打包并回传到了攻击者控制的服务器。该事件波及了大量顶级企业，导致深远的二次攻击风险。

## Red View (利用路径)
1. **上游代码篡改**：攻击者成功获取了目标代码覆盖率工具的云存储更新权限，并在其提供的标准 Bash 上传脚本中注入了一段高度混淆的恶意 cURL 提取命令。
2. **自动化传播与执行**：全球用户的 CI/CD 流水线在构建时自动拉取该脚本。当 `bash <(curl -s https://.../uploader.sh)` 被执行时，恶意逻辑在毫不知情的容器/Runner 环境中悄然触发。
3. **敏感凭据收割**：恶意脚本利用 Linux 环境变量提取命令（如 `env` 或 `export`），将当前上下文中配置的 AWS 密钥、Docker 仓库 Token、NPM 发布 Token 等高价值凭据加密后，通过 HTTP POST 请求发送至攻击者的恶意 IP 节点。

## Blue View (防御缺失)
- **对外部依赖的盲目信任**：开发者在构建脚本中直接通过 URL 动态拉取并执行 Bash 脚本，完全缺乏脚本完整性哈希校验（如 Checksum 或 GPG 签名验证），导致篡改发生后无法察觉。
- **CI/CD 环境凭据管理松散**：大量无关的敏感凭据被作为全局环境变量注入到构建任务中，而该任务（代码覆盖率上传）理论上根本不需要访问这些高权限 Token。
- **构建环境网络出口缺乏管控**：CI/CD Runner 通常具有完全放开的外网访问权限。当恶意脚本向未知的恶意 C2 服务器发起数据外发请求时，由于没有出站流量过滤（Egress Filtering），数据被顺利偷走。

## 策略借鉴
1. **实施强化的依赖完整性校验**：在 CI/CD 中绝对禁止使用 `curl | bash` 的裸跑模式。必须下载文件并验证其 SHA256 哈希值与官方发布的校验和一致后才允许执行。同时应定期将外部脚本在本地私有化仓库中做镜像缓存。
2. **CI/CD 环境的零信任凭据注入 (Dynamic Secrets)**：彻底摒弃将长期有效的静态密码或 Token 存放在环境变量中。采用 HashiCorp Vault 或 AWS Secrets Manager 配合 OIDC 认证，仅在构建的特定步骤（Job）中注入生存期极短的动态凭据，并在该步骤结束后立即销毁。
3. **收紧构建环境的网络出口策略**：针对 CI/CD Runner（如自建 Jenkins 节点或 K8s Job），实施严格的出站网络过滤（Egress Control）。除了允许访问代码仓库（如 GitHub）和指定的制品库外，默认拒绝所有其他对外的出站连接，从根本上阻断凭据外传通道。
