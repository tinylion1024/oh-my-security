# 案例：Docker Daemon API 未授权暴露导致的挖矿僵尸网络入侵

## 事件回顾
某人工智能研究机构在快速部署其深度学习模型时，为了方便异地团队进行容器编排与调度，工程师将一台核心 GPU 服务器上的 Docker Daemon 守护进程的远程 API (端口 2375) 直接暴露在了互联网上。他们为了贪图便利，没有配置任何形式的 TLS 客户端身份验证机制。数天之内，专门扫描此类暴露端点的自动化僵尸网络（如 Kinsing 或 TeamTNT）发现了该服务器。攻击者不仅利用其强大的 GPU 算力挖矿导致业务瘫痪，更利用容器逃逸漏洞控制了底层宿主机，对整个研究网络构成了持续威胁。

## Red View (利用路径)
1. **广域网扫描与 API 探测**：攻击者利用自动化的僵尸网络程序，针对全网 IP 段进行 2375 端口（未加密 Docker API）的扫描，成功命中目标服务器并获取了 Docker 守护进程的响应。
2. **恶意容器创建与运行**：攻击者利用未受保护的 REST API 发送 POST 请求，指令服务器拉取包含了门罗币挖矿程序的恶意镜像（如基于 Alpine 的挖矿脚本），并在后台以特权模式启动容器。
3. **容器逃逸与宿主机接管**：在启动恶意容器时，攻击者特意添加了挂载指令 `-v /:/mnt`，将宿主机根文件系统映射到容器内。随后，攻击者通过修改容器内的 `/mnt/root/.ssh/authorized_keys` 或 `/mnt/etc/crontab`，获得宿主机的 root 权限，实现持久化控制。

## Blue View (防御缺失)
- **底层管理接口裸奔**：Docker Daemon 提供的是等同于宿主机 root 的权限控制能力。将如此高危的接口不加任何防护地暴露在公网，是架构设计上的严重渎职。
- **缺乏双向身份验证 (mTLS)**：对于跨网络的 Docker API 调用，未能按照官方最佳实践配置 TLS 加密和客户端证书认证。这导致任何可达该网络端口的扫描器都能伪装成合法客户端下发指令。
- **特权容器与挂载权限未受限**：宿主机的 Docker 环境缺乏安全限制，允许任意未签名的镜像以特权模式（`--privileged`）运行，并允许任意宿主机路径挂载，这使得从容器突破至宿主机的门槛降至为零。

## 策略借鉴
1. **强制隔离与加密 Docker Daemon API**：切勿将 Docker API 暴露在非受信网络（尤其是公网）。如果必须启用远程访问，必须配置双向 TLS (mTLS) 认证（监听在 2376 端口），确保只有持有由私有 CA 签发的有效客户端证书的机器才能与 Daemon 通信。或者，优先采用通过安全的 SSH 隧道来转发 Docker API 请求。
2. **收拢运行时权限与强制隔离环境**：避免直接使用裸 Docker 面向外部提供服务管理，推荐使用 K8s 并配合强力的安全准入策略。如果在单机运行，必须严禁使用 `--privileged` 参数，限制高危 Capabilities（如 `CAP_SYS_ADMIN`），并使用 AppArmor 或 SELinux 配置强制访问控制策略。
3. **监控恶意镜像与异常行为检测**：部署云工作负载保护平台（CWPP）如 Falco 或 Aqua Security，对异常容器行为进行监控。一旦检测到未经验证的镜像被拉取、容器内执行了修改宿主机 `authorized_keys` 的操作，或出现典型的挖矿网络通信特征，立即在内核层阻断该行为并触发高优先级的报警。
