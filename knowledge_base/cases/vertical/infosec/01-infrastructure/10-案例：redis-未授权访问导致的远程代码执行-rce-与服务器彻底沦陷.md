# 案例：Redis 未授权访问导致的远程代码执行 (RCE) 与服务器彻底沦陷

## 事件回顾
某在线视频平台为了提升并发访问的缓存性能，在内网部署了大量的 Redis 缓存服务器。然而，在一次网络架构变更中，由于路由策略配置错误，一台承载着核心业务缓存的 Redis 服务器被意外映射到了公网。由于 Redis 默认安装时通常不设密码（或使用极弱的密码），且以高权限用户（如 root）运行，攻击者迅速扫描到了这个暴露的 6379 端口。通过利用 Redis 的持久化及动态配置特性，攻击者成功将恶意公钥写入了服务器的系统目录，兵不血刃地获得了该服务器的最高 root 权限（RCE），并借此作为跳板深入内部网络。

## Red View (利用路径)
1. **未授权接口探测**：攻击者对公网 IP 范围进行自动化扫描，发现 6379 端口处于开放状态。通过使用 Redis CLI 客户端连接测试，发现不需要任何 `AUTH` 命令即可成功交互，确认存在未授权访问漏洞。
2. **配置篡改与公钥写入**：攻击者在本地生成了一对 SSH 密钥。然后，通过 Redis 连接，利用 `CONFIG SET dir /root/.ssh` 命令将 Redis 数据库备份的工作目录修改为系统的 SSH 配置目录，并将备份文件名修改为 `authorized_keys`。
3. **触发持久化与获得 Shell**：攻击者将自己的公钥作为值写入到一个临时的 Redis 键中。随后发出 `SAVE` 命令，Redis 会将内存中的数据（包含恶意公钥）写入到 `/root/.ssh/authorized_keys` 文件中。此后，攻击者直接使用私钥，通过 SSH 免密登录获得了该服务器的 root 权限。

## Blue View (防御缺失)
- **缺乏基本的边界隔离**：设计用于纯内网高速缓存的组件（Redis）被错误地赋予了公网可达性，违背了服务不应暴露在不受信任网络的架构底线。
- **服务本身未启用身份校验**：Redis 配置 `redis.conf` 中未设置 `requirepass` 密码，或配置为弱口令，使得暴露后的防线瞬间崩溃，任何扫描到该端口的人都能对其进行任意操作。
- **违背最小权限运行原则**：Redis 守护进程以系统的 `root` 权限运行，而非专用的、受限的低权限系统账户（如 `redis`）。这导致 Redis 能够向极为敏感的系统级目录（如 `/root/.ssh`、`/var/spool/cron`）写入文件，直接导致了远程代码执行。

## 策略借鉴
1. **严格限制网络边界与绑定地址**：在 `redis.conf` 中强制配置 `bind 127.0.0.1` 或者指定的内网专用 IP，并在云防火墙或底层操作系统防火墙（iptables/ufw）上严格拦截外部一切对 6379 端口的访问请求，仅允许应用服务器的 IP 接入。
2. **以最低系统权限运行非核心进程**：绝对禁止使用 `root` 身份启动 Redis、Memcached 等缓存和中间件服务。必须为其创建独立的、不具备系统级登录和敏感目录写权限的系统用户（如 `redis` 账号），从根源上斩断通过覆写系统配置提权的路径。
3. **开启强密码认证并重命名危险命令**：必须配置复杂且高强度的 `requirepass`。此外，为了防御利用动态配置进行攻击的手法，应在配置文件中使用 `rename-command` 将诸如 `CONFIG`, `FLUSHALL`, `SAVE` 等高危命令重命名为难以猜测的随机字符串，甚至彻底禁用，极大地压缩攻击面。
