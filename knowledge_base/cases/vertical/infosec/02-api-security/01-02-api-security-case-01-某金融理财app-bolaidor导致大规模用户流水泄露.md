# 02-api-security Case 01: 某金融理财App BOLA/IDOR导致大规模用户流水泄露

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 越权访问 (BOLA/IDOR)

### 事件回顾
2023年某知名互联网金融理财App被曝出严重的API数据泄露事件。安全研究人员发现，该App在查询用户历史交易流水的接口设计上存在严重漏洞。虽然该接口在请求时包含了合法的JWT凭证，但其资源定位完全依赖于URL路径中的数字型User ID（如 `/api/v2/users/817263/transactions`）。攻击者通过简单地遍历或枚举该ID，成功绕过了业务层面的授权检查，窃取了近百万用户的资金流水、投资偏好及绑定的银行卡尾号等高度敏感的金融数据。

### Red View (利用路径)
1. **API发现与流量抓取**：攻击者注册一个普通账户，使用代理工具（如Burp Suite）抓取手机端App与服务器的交互流量。
2. **定位脆弱接口**：在分析流量时，发现查询账单的接口端点为 `/api/v2/users/{user_id}/transactions`。
3. **参数篡改与越权测试**：攻击者将URL中的 `{user_id}` 替换为另一个已知用户的ID，并发起请求。
4. **自动化遍历窃取**：发现服务器返回了 HTTP 200 OK 并包含了目标用户的交易数据后，攻击者编写Python脚本，利用Intruder模块并发枚举 User ID，实施大规模数据爬取。

### Blue View (防御缺失)
1. **缺失对象级授权 (Missing Object Level Authorization)**：API网关或后端服务仅验证了用户JWT Token的有效性（身份认证），但未在业务逻辑层校验"当前发起请求的用户"是否有权限访问"请求路径中指向的目标资源"。
2. **可预测的资源标识符**：系统使用自增的连续整型数字作为 User ID，使得攻击者可以轻易猜测和遍历有效目标。
3. **缺乏异常访问行为监控**：在短时间内同一个IP或同一个Token发起了大量针对不同User ID的请求，风控系统未能识别这种典型的越权爬取特征并及时熔断。

### 策略借鉴
1. **实现严格的基于属性/角色的访问控制 (ABAC/RBAC)**：在API控制器层或微服务入口强制执行对象级授权（BOLA）校验，确保 `Token.user_id == Request.target_id`，或查询数据库验证归属关系。
2. **采用不可预测的资源标识符**：将业务系统对外暴露的主键从自增ID替换为高熵的 UUIDv4 或 Nanoid，极大增加遍历枚举攻击的难度。
3. **建立基于API行为的基线与风控策略**：在API网关层部署速率限制（Rate Limiting）和行为异常检测机制，当检测到单个用户凭证在短时间内高频访问非自身数据实体时，立即阻断请求并封禁账号。
