# 02-api-security Case 02: 某大型电商平台 Mass Assignment 导致权限提升

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 批量赋值 (Mass Assignment)

### 事件回顾
某跨国电商平台的商家后台API遭遇了恶意利用，导致部分普通卖家账号被违规升级为了“平台超级管理员”，从而能够修改其他商家的商品价格并免除平台佣金。经过应急响应与溯源分析，发现其根本原因在于注册或更新用户资料的RESTful API接口没有对客户端传入的JSON负载（Payload）进行严格的字段过滤。后端直接将传入的JSON对象反序列化并绑定到ORM模型上持久化到数据库中。攻击者通过在正常更新资料的请求中注入了隐藏的敏感字段，成功实现了垂直越权。

### Red View (利用路径)
1. **对象结构推测**：攻击者首先通过调用获取个人信息的接口 `GET /api/v1/profile`，观察到服务器返回的JSON数据中不仅包含 `username`, `email`，还包含布尔类型的隐藏字段 `"is_admin": false` 和 `"commission_rate": 0.05`。
2. **构造恶意Payload**：攻击者拦截修改个人资料的 `PUT /api/v1/profile` 请求。
3. **参数注入与批量赋值**：在原本只包含 `{"email": "hacker@test.com"}` 的请求体中，追加敏感字段，构造为 `{"email": "hacker@test.com", "is_admin": true, "commission_rate": 0.00}`。
4. **权限提升与利用**：后端框架（如Spring Boot或Rails）的数据绑定机制自动将这些字段映射到用户对象并覆盖数据库记录。攻击者随后利用管理员权限调用管理后台API，篡改竞争对手的商品数据。

### Blue View (防御缺失)
1. **缺乏输入数据白名单机制**：后端应用程序盲目信任了客户端提交的数据包，直接将外部输入绑定到了内部的领域模型（Domain Model）上，未区分“允许用户修改的字段”和“仅限系统内部控制的字段”。
2. **读写模型未分离 (DTO缺失)**：API层没有使用独立的数据传输对象（DTO, Data Transfer Object）来接收输入，导致数据暴露面和接收面完全重合。
3. **敏感属性直接暴露**：`GET` 接口返回了不应向普通客户端展示的系统级属性（如 `is_admin`），为攻击者指明了渗透方向。

### 策略借鉴
1. **引入严格的数据传输对象 (DTO) 与反序列化白名单**：在API入口处定义专用的请求接收类（如 `UpdateProfileRequest`），只声明 `email` 等允许修改的属性。坚决避免将ORM实体类直接作为控制器的方法参数。
2. **实施显式的字段绑定与过滤**：在业务逻辑层，明确声明允许更新的属性列表。即使使用了框架的自动绑定功能（如 Laravel的 `$fillable` / `$guarded`，Spring的 `@JsonIgnore`），也必须确保系统级字段受到保护。
3. **最小化API响应数据**：根据不同权限级别裁剪API的返回包，普通用户的API响应中绝对不应包含内部状态属性或权限标识符，减少信息泄露。
