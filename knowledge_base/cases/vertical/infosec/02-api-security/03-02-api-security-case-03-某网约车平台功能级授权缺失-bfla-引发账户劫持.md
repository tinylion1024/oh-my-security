# 02-api-security Case 03: 某网约车平台功能级授权缺失 (BFLA) 引发账户劫持

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 缺失功能级授权 (Broken Function Level Authorization)

### 事件回顾
针对某全球网约车应用的一次红队演练中，安全工程师成功实现了对任意乘客账户的劫持，并能以受害者身份叫车。漏洞的根源在于该平台的后台管理API接口（用于客服重置用户密码或手机号）错误地暴露在了公网，并且仅依赖于前端UI的隐藏来“保护”这些管理功能。由于后端API没有进行实质性的角色和权限校验，普通用户只要猜测或发现了该API的URL，就可以越权调用高权限的管理功能。

### Red View (利用路径)
1. **API枚举与逆向工程**：攻击者对该网约车App的安装包（APK）进行反编译，提取代码中硬编码的所有API端点，或者通过目录扫描字典发现了 `/api/admin/v1/users/reset_phone` 这个隐藏接口。
2. **方法与参数推测**：攻击者分析出该接口需要使用 HTTP PUT 或 POST 方法，并需要提供目标用户的ID和新的手机号码作为参数。
3. **权限验证绕过**：攻击者使用自己普通乘客账号的合法JWT Token发起对该管理员接口的请求：`POST /api/admin/v1/users/reset_phone`，Payload为 `{"user_id": 9999, "new_phone": "13800000000"}`。
4. **成功劫持**：因为API网关仅校验了Token是否过期，而内部服务未验证调用者是否具备 `Admin` 角色，密码找回验证码被发送到了攻击者手机上，导致目标账户被完全接管。

### Blue View (防御缺失)
1. **后端验证形同虚设**：开发者错误地认为，只要在前端管理控制台的UI上限制了普通用户的访问入口，后端API就是安全的（Security by Obscurity），缺乏后端的强制功能级拦截。
2. **垂直权限隔离失败**：对于属于不同功能域（如C端普通业务与B端管理业务）的API，没有实现物理或逻辑上的访问控制隔离，所有的API共享了一套松散的网关鉴权机制。
3. **管理接口暴露面过大**：高危的管理员API直接暴露在了互联网环境中，未进行IP白名单或内网隔离限制。

### 策略借鉴
1. **强制执行默认拒绝 (Default Deny) 的访问控制策略**：在API路由网关或拦截器层，默认拦截所有请求，必须通过集中的权限校验中间件，验证用户Token中的角色Claims是否包含访问目标URI所需的特定权限（如 `role: admin`）。
2. **网络层面的微服务隔离**：将管理类API（Admin API）与公共类API（Public API）在网络架构上进行物理隔离。管理类API应部署在内部网络，或要求必须通过可信的VPN/零信任网关（ZTA）才能访问。
3. **实施基于OAS/Swagger的API契约测试**：在CI/CD流水线中集成自动化安全测试，依据API设计契约自动生成未授权/越权测试用例，确保所有高权限接口无法被低权限凭证调用。
