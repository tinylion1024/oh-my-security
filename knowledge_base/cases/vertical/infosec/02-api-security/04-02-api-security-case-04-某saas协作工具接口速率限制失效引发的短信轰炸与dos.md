# 02-api-security Case 04: 某SaaS协作工具接口速率限制失效引发的短信轰炸与DoS

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 资源消耗缺乏限制 (Unrestricted Resource Consumption)

### 事件回顾
某企业级SaaS项目管理协作工具遭遇了严重的业务逻辑DoS攻击和资金损失。该平台提供了一个“邀请团队成员”的功能，允许用户通过填写邮箱或手机号发送注册邀请链接。由于负责发送短信通知的后端API `/api/v1/invitations/send_sms` 未实施任何形式的速率限制和验证码保护，黑客利用简单的自动化脚本，在短短几个小时内通过该接口发送了数百万条验证码短信，不仅导致了该SaaS平台单日数十万元的短信通道费用损失，还引发了第三方通信提供商的风控熔断，致使平台正常用户的注册功能彻底瘫痪。

### Red View (利用路径)
1. **业务逻辑梳理**：黑客注册免费的试用版账号，登录后访问“团队邀请”页面。
2. **抓包与分析**：抓取点击“发送邀请”时的HTTP请求，发现请求接口为 `POST /api/v1/invitations/send_sms`，参数仅包含目标手机号 `{"phone": "138xxxxxx"}`。
3. **并发重放测试**：黑客将该请求发送至Burp Suite的Repeater模块，对同一个目标手机号连续发送100次请求，发现服务器全部返回200 OK，目标手机瞬间收到大量短信。
4. **分布式攻击发起**：黑客编写多线程Python脚本，结合动态代理IP池，以极高的频率向海量随机手机号批量调用该API，实现短信轰炸，耗尽服务器短信额度。

### Blue View (防御缺失)
1. **接口层面缺乏限流降级机制**：对于消耗计算资源或产生财务成本的API，系统没有在IP维度、用户身份（Token）维度或全局维度设置调用频率阈值（Rate Limiting）。
2. **缺失人机交互验证**：在触发敏感动作（如发送短信、邮件）前，没有强制引入图形验证码（CAPTCHA）、滑动验证等机制来有效拦截自动化脚本工具。
3. **业务逻辑防刷设计空白**：后端未记录并校验特定实体（同一个目标手机号）在短周期内的接收上限（例如：同一个手机号1分钟只能发1次，一天最多发5次）。

### 策略借鉴
1. **多维度部署API限流策略 (Rate Limiting & Throttling)**：在API网关层（如Kong, Nginx, AWS API Gateway）引入基于漏桶或令牌桶算法的限流策略，结合 `Client IP` 和 `User-ID` 实施双重频控。
2. **植入行为验证与风控阻断**：对于涉及外部服务调用或高成本动作的API，强制对接人机验证组件。对异常频次调用触发报警，并自动将恶意IP拉入WAF黑名单。
3. **完善业务逻辑层的安全校验**：在微服务内部利用Redis等缓存机制，硬性限制单一目标凭证（如手机号、邮箱）的发送冷却时间（TTL）与每日请求上限。
