# 02-api-security Case 05: 某互联网医疗平台影子API泄露敏感健康数据

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 资产管理不当 (Improper Inventory Management) / 影子API

### 事件回顾
某大型互联网医疗门户在进行系统版本升级和微服务重构后，发生了一起严重的PHI（受保护健康信息）数据泄露事件。安全团队在溯源时发现，泄露的数据并非来自当前主线版本的接口 `/api/v3/patients`，而是来自一个本该被废弃但仍在后台静默运行的旧版本接口 `/api/v1/patients/export`。这个被称为“影子API（Shadow API）”的旧接口不受新版WAF规则和现代认证体系的保护，缺乏必要的权限校验。黑客通过扫描历史残留接口发现了它，并直接导出了大量包含用户诊断记录和处方的隐私数据。

### Red View (利用路径)
1. **资产探测与历史接口扫描**：攻击者利用目录爆破工具（如ffuf）或通过分析该应用早期版本的旧版Android APK，收集到了该平台曾使用的V1版本API端点信息。
2. **端点存活测试**：攻击者尝试访问 `https://api.medical-app.com/api/v1/patients/export?date=2023-01`，发现服务器虽然提示缺少参数，但并未返回404或403错误，确认该影子API依然存活。
3. **绕过新版安全机制**：由于V1接口开发年代较早，其身份验证机制仅依赖于简单的固定Token（甚至是无需认证即可访问），完全绕过了当前V3版本基于OAuth 2.0的动态授权机制。
4. **数据批量窃取**：攻击者编写脚本，通过改变 `date` 参数等查询条件，利用这个无人监管的僵尸API，持续拖库长达数周而不被察觉。

### Blue View (防御缺失)
1. **API资产生命周期管理缺失**：开发与运维团队没有建立完善的API资产清单。旧版服务在被新版替代后，没有执行严格的下线（Deprecation）和关停（Decommission）流程。
2. **缺乏统一的网关路由管控**：旧版微服务仍然直接暴露在负载均衡器或公网入口中，未能通过集中式API网关进行统一的路由收敛和鉴权拦截。
3. **安全可见性盲区**：监控与日志告警系统主要聚焦于活跃的主线业务流量，对于低频的历史版本API请求缺乏基线对比和异动监控，导致攻击行为长期未被发现。

### 策略借鉴
1. **建立自动化的API资产盘点库**：结合CI/CD流水线、API网关配置和流量分析工具，动态发现并记录所有暴露的API端点，杜绝未知（Shadow）和废弃（Zombie）API的存在。
2. **强制执行API版本控制与下线机制**：制定严格的API退役策略。当旧版本API到达生命周期终点时，必须从API网关彻底移除路由规则，并关闭对应的后端服务容器。
3. **统一API安全网关拦截**：所有对外接口必须经过统一的API安全网关，实行全局一致的强身份验证机制（如JWT/OAuth），确保没有任何“后门”接口能绕过核心安全屏障。
