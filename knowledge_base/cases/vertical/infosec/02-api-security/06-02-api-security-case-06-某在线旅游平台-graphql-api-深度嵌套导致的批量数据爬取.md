# 02-api-security Case 06: 某在线旅游平台 GraphQL API 深度嵌套导致的批量数据爬取

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 资源消耗缺失 / 越权访问 (GraphQL Security)

### 事件回顾
随着GraphQL的普及，某知名在线旅行（OTA）平台将其订单查询系统从REST迁移到了GraphQL架构。然而，安全配置的疏忽导致该系统在上线后不久便遭遇了严重的数据爬取攻击。由于GraphQL允许客户端高度自定义数据查询结构，黑客利用查询的深度嵌套特性（Deep Nesting）和别名特性（Aliasing），在一个单独的HTTP POST请求中构造了包含成百上千个不同子查询的庞大Payload，瞬间耗尽了数据库连接池，并绕过了基于单个HTTP请求计算的速率限制，非法拉取了大量同行的酒店底价和旅客订单信息。

### Red View (利用路径)
1. **内省查询探测**：攻击者向端点 `POST /graphql` 发送了标准的Introspection（内省）查询，服务器返回了完整的Schema结构，暴露了所有可用的Query类型、对象及其关联字段。
2. **识别关联漏洞**：在分析Schema时，攻击者发现 `Hotel` 对象可以关联查询 `Bookings`，而 `Bookings` 又可以关联查询回 `User`，进而查询 `User` 关联的其他 `Bookings`。
3. **构造恶意嵌套/批量查询**：攻击者编写了一个极其复杂的GraphQL Payload，利用层层嵌套（`hotel { bookings { user { bookings { hotel { ... } } } } }`）并使用别名机制并行查询上千个订单ID。
4. **绕过限流并引发拒绝服务**：因为所有操作都在一个HTTP请求内完成，传统的WAF基于请求频次的防护失效。后端为了解析这个复杂的AST树并执行相应的N+1数据库查询，导致CPU飙升，同时向攻击者返回了大量的敏感商业数据。

### Blue View (防御缺失)
1. **过度开放的内省机制**：在生产环境中未关闭GraphQL的Introspection功能，等于向攻击者直接提供了API的详细使用说明书。
2. **缺失查询深度与复杂度控制**：服务器使用的GraphQL执行引擎没有配置最大查询深度（Max Depth）和查询复杂度（Query Complexity）限制，导致客户端可以任意索取服务器计算资源。
3. **细粒度对象权限失效**：GraphQL解析器（Resolvers）在处理深层嵌套的数据拉取时，未能为子节点对象（如跨用户的订单）进行实时的所有权校验（BOLA防护缺失）。

### 策略借鉴
1. **实施 GraphQL 查询复杂度与深度限制**：在GraphQL中间件中配置严格的防护策略，拒绝解析深度超过阈值（如5级）或总体计算复杂度评分过高的恶意查询请求。
2. **生产环境禁用 Introspection 并控制暴露面**：在系统上线前，务必关闭内省查询功能。并考虑使用持久化查询（Persisted Queries），只允许执行服务器端预先白名单验证过的查询模板。
3. **在 Resolver 层强制细粒度权限校验**：绝不能仅在入口层面做鉴权。在图结构的每一个敏感节点的Resolver函数中，必须注入用户上下文（Context），并校验当前请求者是否有权获取该节点数据。
