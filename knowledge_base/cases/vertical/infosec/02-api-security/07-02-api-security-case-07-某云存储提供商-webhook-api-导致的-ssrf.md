# 02-api-security Case 07: 某云存储提供商 Webhook API 导致的 SSRF

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 服务端请求伪造 (Server-Side Request Forgery)

### 事件回顾
某云存储服务提供商允许用户通过Webhook API在文件上传成功时触发第三方服务回调。该API接收用户提供的一个目标URL，并由后端的任务处理服务器主动向该URL发起HTTP POST请求推送元数据。由于该Webhook处理引擎对用户输入的URL未做严格的安全校验和过滤，攻击者成功将其作为跳板，诱导内部服务器向受保护的内网基础设施发起请求。最终，攻击者通过该API成功读取了AWS云环境中 EC2 实例的元数据凭证，导致云环境被部分渗透。

### Red View (利用路径)
1. **功能点探测**：攻击者在控制面板中找到“配置Webhook”的API接口，可以输入任意URL，例如 `http://attacker-server.com/receive`。
2. **内网探测与盲注SSRF**：攻击者将Webhook URL修改为内网私有IP地址或特定服务地址，例如 `http://127.0.0.1:6379`（Redis）或 `http://192.168.1.100:8080`，观察接口返回的时长或报错信息来推断内网结构。
3. **利用云元数据服务**：攻击者将URL修改为AWS实例元数据的魔法IP：`http://169.254.169.254/latest/meta-data/iam/security-credentials/admin-role`。
4. **窃取高权限凭证**：后端服务器未经校验直接请求了该地址，并将获取到的包含 AccessKeyId 和 SecretAccessKey 的敏感响应作为“推送失败的Debug日志”返回给了攻击者，导致云主机权限完全沦陷。

### Blue View (防御缺失)
1. **不可信输入直接进入网络请求**：后端使用HTTP客户端库发起请求前，未对用户传入的外部URL进行协议、域名和IP层面的严格白名单或黑名单过滤。
2. **缺乏网络层隔离边界**：执行Webhook推送任务的工作节点（Worker）没有被隔离在专用的DMZ网络中。该节点依然拥有访问内部敏感服务（如云环境Metadata服务、内部数据库）的网络路由权限。
3. **响应数据过度泄露**：API在处理外部请求失败时，将底层网络请求的原始响应数据返回给了前端客户端，为盲打SSRF提供了清晰的回显通道。

### 策略借鉴
1. **建立严密的URL解析与IP黑名单机制**：在发起请求前，必须解析URL的真实IP地址（防范DNS Rebinding攻击），并确保最终IP不属于 `127.0.0.0/8`、`10.0.0.0/8`、`169.254.169.254` 等私有和特殊保留地址。
2. **网络隔离与零信任微隔离**：将负责外发Webhook请求的服务器剥离至独立的VPC子网或使用网络安全组（Security Groups），默认切断其访问任何内部网络资源的路由。
3. **剥离或清洗响应回显**：对于对外发送的网络请求，切勿将获取的HTTP Body原样返回给用户。应仅返回业务层面的状态（如 "推送成功" 或 "推送失败"），阻断SSRF的带外回显。
