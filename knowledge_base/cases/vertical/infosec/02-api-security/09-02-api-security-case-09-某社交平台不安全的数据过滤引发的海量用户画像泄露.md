# 02-api-security Case 09: 某社交平台不安全的数据过滤引发的海量用户画像泄露

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 数据暴露过多 (Excessive Data Exposure)

### 事件回顾
一家主打陌生人交友的社交平台因隐私泄露遭到监管机构调查。事件起因是安全社区披露了该平台的用户信息获取接口 `/api/v2/user/match` 存在过度数据暴露问题。该接口用于在App前端展示匹配用户的简要卡片信息（仅显示昵称、头像和距离）。然而，后端API开发为了节省重复写接口的麻烦，复用了底层的通用数据查询服务，导致该接口向移动端返回了目标用户的全量数据库记录。黑客只需抓取网络包，就能清晰看到并爬取无数用户的精准GPS坐标、真实姓名、手机号以及隐藏的黑名单列表等高度敏感的隐私数据。

### Red View (利用路径)
1. **App正常交互抓包**：攻击者使用中间人代理工具拦截客户端滑动匹配用户时的网络请求包。
2. **发现冗余数据包**：攻击者观察到，虽然手机屏幕上只展示了对方的名字和头像，但对应的HTTP响应包（JSON格式）却异常庞大。
3. **数据解构分析**：展开JSON响应体后，攻击者发现了诸如 `real_name`, `exact_lat`, `exact_long`, `phone_number` 以及 `account_status` 等极为隐私的明文字段。
4. **编写定向爬虫自动化收割**：攻击者编写多线程脚本，连续调用该匹配接口并自动解析保存Response中的敏感字段，导致平台大量深层隐私数据以极低成本被窃取贩卖。

### Blue View (防御缺失)
1. **依赖客户端进行数据过滤**：开发团队将数据过滤的责任错误地推给了前端/移动端App（认为只要UI不显示，用户就看不见），而后端API直接做了简单的 `SELECT * FROM users` 并序列化返回。
2. **缺乏面向视图的API设计 (BFF缺失)**：系统架构中缺乏 "Backend For Frontend" (BFF) 层，没有针对具体的业务场景定制专属的数据结构视图，导致“通用大接口”在特定场景下返回大量无关紧要的冗余数据。
3. **敏感数据未做脱敏处理**：即使是内部流转的接口，涉及到高敏数据（如经纬度、手机号）也未实施打码（Masking）或泛化处理，直接将明文抛出至公网环境。

### 策略借鉴
1. **在后端强制执行数据裁剪与DTO映射**：API响应绝不能直接返回数据库实体对象（Entity）。必须在控制器层定义严格的数据传输对象（DTO），仅挑选和映射当前业务场景绝对必要的最少字段进行序列化。
2. **引入前端友好的定制层 (BFF API)**：为不同的客户端（如Web、iOS、Android）或不同功能模块建立专属的聚合接口，确保数据的查询、拼接和裁剪过滤都在服务端安全的边界内完成。
3. **部署API层面的敏感数据防泄漏监控 (DLP)**：在API网关层集成数据防泄漏策略，自动识别流出流量中的电话号码、身份证、精确GPS坐标等敏感特征，当检测到未脱敏的高敏数据流出时立即触发拦截与告警。
