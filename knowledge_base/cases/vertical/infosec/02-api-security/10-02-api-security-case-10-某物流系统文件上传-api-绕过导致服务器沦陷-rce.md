# 02-api-security Case 10: 某物流系统文件上传 API 绕过导致服务器沦陷 (RCE)

- **所属领域**: 信息安全 / API安全
- **威胁类型**: 不安全的内容校验 (Improper Content Validation) / RCE

### 事件回顾
某大型国际物流公司的内部运单管理系统遭遇勒索软件攻击，核心数据库被加密。经过溯源调查，黑客的突破口在于一个供外部合作伙伴上传发票扫描件（PDF/JPG）的API接口：`/api/v1/invoices/upload`。该API虽然在网关层面校验了调用者的身份凭证，并在后端限制了上传文件的拓展名必须为 `.jpg`，但未对文件的实际内容和MIME类型进行深度校验，同时错误地配置了存储目录的执行权限。攻击者通过利用该接口上传了伪装成图片的Webshell脚本，成功在目标服务器上执行任意命令。

### Red View (利用路径)
1. **定位上传API**：攻击者利用合法合作伙伴的凭证登录系统，找到发票上传功能，抓包记录上传API的端点和参数结构。
2. **绕过后缀名限制**：后端可能存在逻辑漏洞（如解析文件名时的截断漏洞，或解析方式不严谨），攻击者构造了一个名为 `invoice.php.jpg` 或通过空字节绕过 `invoice.php%00.jpg` 的文件。
3. **内容伪装注入**：攻击者修改HTTP请求的 `Content-Type` 为 `image/jpeg`，但在文件主体内容中植入了合法的图片文件头（Magic Bytes，如 `FF D8 FF E0`），随后追加了恶意的PHP一句话木马 `<?php system($_GET['cmd']); ?>`。
4. **获取RCE执行权限**：文件成功被API存储在公开访问的 `/uploads/invoices/` 目录下。由于Web服务器未禁止该目录的脚本解析权限，攻击者直接访问该路径执行命令，最终拿下服务器系统权限。

### Blue View (防御缺失)
1. **表面化的内容验证机制**：API后端仅仅依赖HTTP头部中的 `Content-Type` 和文件名后缀来判断文件安全性，没有针对上传负载的二进制Magic Bytes及实际结构进行深度合法性校验。
2. **存储与执行隔离失败**：上传的文件被直接放置在了Web应用程序的Root目录下（即对外提供Web解析服务的目录），且该目录没有被配置为“禁止脚本执行”的权限。
3. **缺乏恶意特征扫描组件**：API的上传工作流中，没有引入防病毒引擎（AV）或文件威胁清洗（CDR）机制，使得恶意负载畅通无阻地落盘。

### 策略借鉴
1. **实施强化的文件合法性与深度校验**：在API层对接收到的文件流进行严格解析，不仅要验证扩展名白名单，还必须通过流式读取确认文件的Magic Bytes以及解析该格式是否合法，甚至可以通过图像重绘剥离隐藏的恶意代码。
2. **存储解耦与权限收敛**：上传的文件绝对不要存储在Web应用的执行目录内。应当将文件存储在独立的云对象存储（如AWS S3）或隔离的文件服务器上，并由专用的静态资源服务以只读方式提供下载。
3. **使用随机化重命名策略**：无论用户上传何种文件，API在落地存储时都必须放弃原始文件名，采用强随机算法（如UUID）生成新的无特征文件名，不给攻击者留下可预测的执行路径。
