# Apache Log4j2 JNDI 注入漏洞 (Log4Shell) 引发的开源供应链危机
- **所属领域**: 信息安全
- **威胁类型**: 开源组件漏洞 / 零日攻击
- **事件回顾**: 2021年12月，Java 生态中最流行的日志记录框架之一 Apache Log4j2 被爆出存在严重的远程代码执行 (RCE) 漏洞 (CVE-2021-44228，被称为 Log4Shell)。由于该组件被几乎所有主流的 Java 商业系统、中间件和云服务广泛使用，漏洞的波及范围极其广泛。攻击者只需在用户输入中构造特定的 JNDI 查找字符串 (如 `${jndi:ldap://...}`)，若该字符串被 Log4j2 记录，就会导致目标服务器从攻击者控制的远程服务器下载并执行恶意代码。
- **Red View (利用路径)**: 攻击者扫描互联网上暴露的各种服务 (如 Web 表单、API 参数、HTTP 请求头 User-Agent 等)，将带有恶意 JNDI Payload 的数据注入其中。当后端的 Java 应用调用 `logger.info()` 等方法记录这些数据时，Log4j2 的字符串替换机制会解析 JNDI 协议，并通过 LDAP 或 RMI 协议向外发起请求，加载恶意的 Java class 文件至目标系统内存中执行，从而直接获取服务器的最高控制权，常被用于部署勒索软件或挖矿木马。
- **Blue View (防御缺失)**: 事件凸显了开源软件供应链中的可见性缺失问题。许多企业甚至不知道自己的系统中是否间接依赖了 Log4j2 组件。此外，漏洞的成因在于日志组件提供了过度设计的功能 (动态协议解析)，而大多数用户仅将其作为简单的日志记录工具。在边界防御上，许多内部服务允许不受限制地向公网发起 LDAP/RMI 协议请求，使得 JNDI 注入得以成功利用。
- **策略借鉴**: 建立完善的软件成分分析 (SCA) 体系和 SBOM 物料清单，确保能够分钟级定位受影响资产；遵循默认安全的原则，禁用组件中不需要的高级功能 (如在 Log4j2 中禁用 lookup 机制)；在网络边界严格限制服务器的出站连接策略，只允许白名单内的必要端口和目标 IP 访问，阻断恶意 payload 下载路径。
