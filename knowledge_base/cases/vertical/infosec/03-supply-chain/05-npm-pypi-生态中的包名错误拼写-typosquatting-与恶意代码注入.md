# NPM / PyPI 生态中的包名错误拼写 (Typosquatting) 与恶意代码注入
- **所属领域**: 信息安全
- **威胁类型**: 开源包仓库污染 / 社会工程学
- **事件回顾**: 近年来，开源包管理器 (如 npm, PyPI, RubyGems) 频繁成为供应链攻击的重灾区。攻击者利用开发者在输入包名时的拼写错误 (Typosquatting，例如将 `requests` 写成 `requesrs`)，或是利用依赖混淆 (Dependency Confusion) 技术，将包含恶意代码的包上传至公共仓库。一旦开发者或自动化构建系统不慎安装了这些恶意包，攻击者便能窃取开发者主机上的敏感信息，甚至将恶意代码合并至最终产品中。
- **Red View (利用路径)**: 攻击者编写具有与流行开源库相同 API 接口的恶意包，并在其安装脚本 (如 npm 的 `preinstall` 或 Python 的 `setup.py`) 中嵌入恶意负载。这些负载通常在后台静默运行，搜索 `.bash_history`、`.aws/credentials`、`.ssh/id_rsa` 以及各类环境变量中的令牌，并通过 DNS 或 HTTPS 将数据外发。更为高级的攻击如"依赖混淆"，攻击者通过在公共仓库发布与企业内部私有包同名且版本号更高的包，诱骗企业的包管理器优先从公网下载带有后门的组件。
- **Blue View (防御缺失)**: 开发者在使用 `npm install` 等命令时往往缺乏严格的审查，盲目相信开源仓库的安全性。企业缺乏统一的依赖解析策略和本地镜像代理，使得开发和构建环境直接与不可控的公网包管理器交互。此外，开源平台早期缺乏针对命名相似性、恶意行为的自动化审核机制，给攻击者留下了可乘之机。
- **策略借鉴**: 部署私有包仓库管理工具 (如 Nexus, Artifactory)，配置严格的包解析路由规则，防止依赖混淆攻击；在 CI/CD 和开发环境中集成 SCA (软件成分分析) 和自动化的漏洞/恶意包扫描工具；禁用包管理器的自动安装脚本执行功能 (如使用 `npm install --ignore-scripts`)，防止在安装阶段执行任意代码。
