# GitHub Actions 工作流劫持与隐藏挖矿供应链攻击
- **所属领域**: 信息安全
- **威胁类型**: CI/CD 计算资源滥用 / 恶意 Pull Request
- **事件回顾**: 随着 GitHub Actions 成为开源项目和企业首选的 CI/CD 工具，攻击者开始利用该平台进行滥用，特别是进行加密货币挖矿。在多起事件中，攻击者通过向知名开源项目提交恶意的 Pull Request (PR)，利用配置不当的 GitHub Actions 工作流自动执行恶意代码，从而窃取 GitHub 的计算资源。虽然这并未直接窃取企业数据，但它展示了 CI/CD 流程作为供应链一部分的脆弱性。
- **Red View (利用路径)**: 攻击者通常会 Fork 一个目标仓库，然后在 `package.json` 或 CI 配置文件中添加隐蔽的挖矿脚本。随后，攻击者向目标仓库发起 PR。如果目标仓库的 GitHub Actions 配置为 "基于 Pull Request 自动触发构建" 且未设置适当的权限限制（例如对未关联贡献者的 PR 进行审批），Actions runner 会自动检出攻击者的恶意分支并执行构建脚本。这使得挖矿程序能够在 GitHub 提供的虚拟机上运行，直到触发超时或被发现。
- **Blue View (防御缺失)**: 攻击之所以成功，主要是因为开源项目的维护者为了提升效率，过度放开了 CI/CD 的触发条件。对于任何外部提交的代码，如果没有任何人工审核就直接在具有某些权限（例如可以访问 Secrets）的 CI 环境中运行，是极其危险的。此外，CI/CD 环境往往被认为仅仅是"构建工具"，而忽略了其作为计算资源和潜在跳板的价值。
- **策略借鉴**: 对公开仓库的 CI/CD 配置实施严格的触发策略，例如要求外部贡献者的 PR 必须经过维护者手动批准 (Approve) 才能触发构建 (使用 GitHub 提供的 "Require approval for all outside collaborators" 功能)；在 CI 运行环境中禁止非必要的出站网络连接，切断挖矿程序的矿池通信；利用自动化工具检查 PR 中引入的新依赖或修改的构建脚本；定期审计 CI/CD 流水线的执行日志和资源消耗情况。
