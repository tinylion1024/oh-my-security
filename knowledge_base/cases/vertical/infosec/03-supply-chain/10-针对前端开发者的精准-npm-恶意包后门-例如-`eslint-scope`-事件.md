# 针对前端开发者的精准 npm 恶意包后门 (例如 `eslint-scope` 事件)
- **所属领域**: 信息安全
- **威胁类型**: 开源仓库账户接管 / 恶意包分发
- **事件回顾**: 2018年，npm 生态系统中极为常用的工具包 `eslint-scope` 遭遇了供应链攻击。该包由于是 ESLint 的依赖项，每周有数百万次的下载量。攻击者窃取了该包维护者的 npm 凭证，发布了一个包含恶意代码的补丁版本 (3.7.2)。这次攻击明确针对开发者的工作环境，目的是窃取他们在本地配置的环境变量和凭证。
- **Red View (利用路径)**: 攻击者首先通过凭证复用或钓鱼手段，获取了拥有 `eslint-scope` 发布权限的 npm 账号。随后，他们在包的代码中隐蔽地加入了一段能够收集本地 `.npmrc` 内容以及各种环境变量的脚本。一旦开发者在项目中执行 `npm install` 升级到受污染的版本，安装后脚本（如未被禁用）或在代码运行时，这段恶意代码就会悄悄地将开发者的 npm token、AWS 密钥等信息发送到攻击者配置的 pastebin 或 C2 服务器上。
- **Blue View (防御缺失)**: 事件的核心在于开源维护者的账号安全薄弱。在早期，npm 平台并未强制要求高下载量的包维护者启用双因素认证 (2FA)。此外，由于开发者习惯于自动更新小版本号依赖（如 `^3.7.0`），受污染的 `3.7.2` 版本在发布后迅速通过 CI 流水线和本地安装传播。企业层面缺乏对第三方依赖变更的及时感知和阻断机制。
- **策略借鉴**: 强制要求所有软件仓库发布者启用多因素认证 (MFA) 甚至硬件安全密钥；在项目开发中使用锁定依赖版本文件（如 `package-lock.json` 或 `yarn.lock`），避免自动拉取未经验证的小版本更新；引入自动化的供应链安全监控工具（如 Dependabot/Snyk），在依赖出现异常更新或被标记为恶意时及时告警；审查并限制本地开发环境和 CI 容器中环境变量的暴露范围。
