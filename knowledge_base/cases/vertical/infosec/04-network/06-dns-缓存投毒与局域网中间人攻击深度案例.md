# DNS 缓存投毒与局域网中间人攻击深度案例
- **所属领域**: 信息安全
- **威胁类型**: 域名解析劫持 / 中间人攻击
- **事件回顾**: DNS 是互联网的地址簿，但在很多企业内网中，DNS 解析依然使用明文的 UDP 协议。DNS 缓存投毒（如经典的 Kaminsky 漏洞重现或局域网内的 ARP 欺骗辅助 DNS 劫持）允许攻击者将合法域名的解析结果指向恶意 IP。一旦内部 DNS 服务器或客户端的缓存被污染，员工在访问公司内部系统、SaaS 服务或网银时，会被无缝重定向到攻击者伪造的钓鱼网站。
- **Red View (利用路径)**: 在局域网环境中，攻击者常用工具（如 Responder 或 Ettercap）进行 ARP 欺骗，使自己成为网关。随后，监听网络中的 DNS 请求。当受害者请求 `intranet.company.com` 时，攻击者抢在真实的 DNS 服务器响应之前，向受害者发送伪造的 DNS 响应包，将解析地址设为攻击者主机的 IP。如果是在更高级的外部 DNS 缓存投毒中，攻击者会向目标 DNS 递归解析器发送大量伪造的响应，试图猜测 DNS 事务 ID 和源端口。一旦猜中，递归服务器就会将恶意记录缓存下来，导致该网段内所有用户均被重定向到恶意站点，从而进行凭证收割或漏洞利用。
- **Blue View (防御缺失)**: 许多企业的内部 DNS 基础设施陈旧，未启用 DNSSEC（DNS 安全扩展）来验证响应的数字签名。内网网络层缺乏端口安全（Port Security）和动态 ARP 检测（DAI）机制，使得二层网络欺骗变得异常容易。此外，客户端和服务器之间的通信如果未使用强制的 HSTS 或强相互认证的 TLS，即使重定向到伪造站点，用户也难以察觉。
- **策略借鉴**: 在企业内部和外部 DNS 基础设施上全面部署并强制验证 DNSSEC；在交换机层面启用 DHCP Snooping 和动态 ARP 检测 (DAI) 防御局域网中间人攻击；推广使用 DNS over HTTPS (DoH) 或 DNS over TLS (DoT) 以加密 DNS 查询，防止监听和篡改；实施严格的端点网络监控，检测异常的 DNS 响应来源和高频的解析失败。
