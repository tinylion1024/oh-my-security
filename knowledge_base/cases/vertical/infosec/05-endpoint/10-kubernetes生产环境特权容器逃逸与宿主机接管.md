# [Kubernetes生产环境特权容器逃逸与宿主机接管]
- **所属领域**: 信息安全 - 端点安全 (云原生端点)
- **威胁类型**: 容器逃逸 / 提权 / 宿主机接管
- **事件回顾**: 某互联网公司云原生生产环境遭破坏。攻击者利用Web应用RCE漏洞获取普通微服务容器访问权限。由于容器配置错误，攻击者成功实施“容器逃逸”，突破隔离边界，获取了底层Kubernetes Node（宿主端点）的root权限，随后横向控制了整个K8s集群。
- **Red View (利用路径)**: 攻击者进入容器后枚举发现，该Pod为方便调试错误开启了特权模式（`securityContext: privileged: true`）。利用特权容器特性，攻击者挂载了宿主机的根文件系统（如 `/dev/sda1`），并通过修改宿主机的 `cron` 计划任务，或直接 `chroot` 到宿主机文件系统，以宿主机root身份执行反弹shell，完成逃逸。
- **Blue View (防御缺失)**: 云原生端点配置严重偏离安全基线。开发团队滥用高危权限，集群缺乏Pod安全策略（Pod Security Admissions）硬性拦截特权容器部署。宿主机节点缺乏针对容器运行时异常行为（如敏感目录写入、非预期进程逃逸）的实时监控手段。
- **策略借鉴**: 云原生端点防御需全面加固：1）严格实施最小权限原则，通过准入控制器禁止部署特权容器，限制不必要Linux Capabilities（如 `CAP_SYS_ADMIN`）；2）强制使用只读根文件系统，限制挂载宿主机敏感目录（如 `docker.sock`, `/etc`）；3）在宿主端点部署基于eBPF技术的云原生运行时安全监控平台，阻断逃逸。
