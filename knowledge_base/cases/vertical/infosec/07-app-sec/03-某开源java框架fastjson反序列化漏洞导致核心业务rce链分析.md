# 某开源Java框架Fastjson反序列化漏洞导致核心业务RCE链分析
- **所属领域**: 信息安全 (应用安全)
- **威胁类型**: 反序列化漏洞 (Insecure Deserialization) / RCE
- **事件回顾**: 某头部电商平台在年度大促前夕遭遇了突发的系统宕机与核心数据被加密的严重事件。通过应急响应团队对主机内存dump和流量日志的分析，确认此次攻击是通过某老旧业务系统中使用的Fastjson库的反序列化漏洞（类似于CVE-2017-18349等变种）发起的。攻击者通过构造恶意的JSON数据包，触发了存在缺陷的Java类的实例化过程，并在服务器上成功执行了任意命令（RCE），最终获取了主机的最高控制权，横向移动进入核心数据库网段。
- **Red View (利用路径)**: 攻击者利用了目标业务系统中暴露的订单更新API，该API在接收JSON格式数据时使用了未打补丁的Fastjson版本。攻击者精心构造了利用链（Gadget Chain），例如通过注入 `@type` 指定了常用的存在反序列化风险的类（如 `com.sun.rowset.JdbcRowSetImpl`），并利用其 `dataSourceName` 属性指向攻击者控制的恶意JNDI/RMI服务端。当应用后端将传入的JSON数据反序列化为对象时，触发了JNDI注入，自动从恶意服务端加载并执行了包含木马程序的字节码文件，直接建立了反向Shell连接。
- **Blue View (防御缺失)**: 
  1. 依赖组件管理失控：核心业务系统大量使用了开源组件，但缺乏有效的软件成分分析（SCA）工具，导致高危版本的Fastjson长期滞留于生产环境未得到升级。
  2. 反序列化过滤缺失：未启用或未正确配置Fastjson的AutoType安全机制（SafeMode），允许了任意不可信类的反序列化操作。
  3. 网络隔离与出网策略薄弱：应用服务器具备直接访问外部互联网的权限（DNS/HTTP/RMI），导致JNDI注入时可以顺利下载外部恶意类，缺乏严格的出网白名单控制。
- **策略借鉴**: 建立完善的软件物料清单（SBOM）与自动化漏洞扫描机制，强制升级存在反序列化风险的基础库；在架构层面限制Java应用接收不受信任的数据流进行原生反序列化，优先采用更加安全的数据绑定方式；在主机防火墙或零信任网关层实施严格的“默认拒绝”出网策略，阻断非业务必须的外部网络连接。
