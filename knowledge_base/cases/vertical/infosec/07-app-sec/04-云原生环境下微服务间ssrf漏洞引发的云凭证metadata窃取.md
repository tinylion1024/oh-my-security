# 云原生环境下微服务间SSRF漏洞引发的云凭证(Metadata)窃取
- **所属领域**: 信息安全 (应用安全)
- **威胁类型**: 服务端请求伪造 (SSRF) / 云凭证泄露
- **事件回顾**: 某基于AWS构建的互联网金融平台发生严重的客户信息泄露事件。经查，问题源于内部的一个“发票PDF生成微服务”。该服务提供了一个功能，允许用户传入一个URL，系统会抓取该URL对应的HTML页面并渲染生成PDF。由于对传入的URL缺乏严格的校验与白名单过滤，攻击者成功利用SSRF（服务端请求伪造）漏洞，读取了云主机（EC2）实例的元数据（Metadata）端点，窃取了与该实例绑定的IAM临时高权限凭证，进而调用AWS API转储了S3中的高密数据。
- **Red View (利用路径)**: 攻击者在发票模板的URL参数中注入了云提供商的元数据API地址（例如 `http://169.254.169.254/latest/meta-data/iam/security-credentials/`）。服务在后台解析该参数后，发起了HTTP GET请求。由于请求是由EC2实例本地发出的，云环境默认信任并返回了包含 `AccessKeyId`、`SecretAccessKey` 和 `Token` 的JSON响应。攻击者通过渲染后的PDF文件读取到了这些凭证信息。利用获取的凭证，攻击者在本地配置了AWS CLI，直接以该微服务的IAM角色权限，枚举并下载了存储着数百万交易记录的受保护S3存储桶数据。
- **Blue View (防御缺失)**: 
  1. 输入校验与白名单缺失：微服务未对目标URL的协议（如仅限http/https）、域名（拦截内网或保留IP段）以及重定向行为进行严格验证。
  2. 云安全配置过宽：EC2实例被赋予了过高的IAM权限，业务实际上只需读取特定S3资源，却被赋予了列出和读取所有S3桶的广泛权限（Over-privileged IAM Roles）。
  3. 元数据防护机制未启用：未强制启用IMDSv2（Instance Metadata Service Version 2），导致不需要请求头验证的简单GET请求即可窃取凭证。
- **策略借鉴**: 强制云环境中所有计算实例启用IMDSv2，要求所有Metadata请求必须包含特定的Header；在应用层实现健壮的SSRF防御组件，屏蔽所有私有IP地址（127.0.0.0/8, 169.254.0.0/16等），禁止DNS重绑定；遵循最小权限原则（PoLP）配置云IAM策略，通过异常行为检测监控工具（如AWS GuardDuty）实时发现异常的凭证使用行为。
