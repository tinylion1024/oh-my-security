# 某金融支付APP基于JWT弱签名算法(None)伪造管理员Token漏洞
- **所属领域**: 信息安全 (应用安全)
- **威胁类型**: 身份认证绕过 / JWT算法混淆
- **事件回顾**: 某新兴数字货币支付APP在举行空投活动时，遭遇恶意用户大量套取资金。风控系统检测到异常的大额转账请求均来自于拥有管理员权限的API调用。安全团队排查发现，该APP使用了JWT（JSON Web Token）进行前后端无状态身份验证。开发人员在引入第三方JWT解析库时，未强制指定签名校验的算法白名单，导致系统接受了使用“None”算法签名的恶意Token。攻击者借此伪造了管理员身份，绕过了所有的权限校验控制。
- **Red View (利用路径)**: 攻击者首先抓包分析了正常的登录流量，提取了自己普通账号的JWT凭证。他们发现该凭证的Header部分标明了 `alg: HS256`。攻击者编写了脚本，将Header中的 `alg` 字段修改为 `None`，同时在Payload部分将 `role: user` 篡改为 `role: admin`，并将 `user_id` 修改为系统预设的管理员ID。然后，攻击者删除了JWT的Signature部分（即第三部分），只保留Header和Payload的Base64编码，并在末尾保留一个点号（`.`）。由于服务器端解析库存在漏洞，遇到 `alg: none` 时直接跳过了签名验证，信任了被篡改的Payload，使得攻击者成功接管了管理员会话并执行了批量转账操作。
- **Blue View (防御缺失)**: 
  1. 密码学配置缺陷：服务端JWT鉴权中间件信任了客户端传入的 `alg` 字段，而不是在服务端硬编码强制要求特定的强加密算法（如RS256或RS512）。
  2. 权限校验依赖单一：高敏感操作（如大额转账）仅仅依赖无状态的JWT进行权限判断，未在服务端加入进一步的风控规则和敏感操作二次验证（如支付密码或MFA）。
  3. 库文件版本过低：使用了已经被披露存在“None”算法绕过漏洞的过时开源库，未能及时修复。
- **策略借鉴**: 在后端API验证JWT时，必须显式限制并硬编码允许的签名算法（避免信任Header中的 `alg` 字段）；针对高危资金和管理操作，实施Token失效机制（如Redis黑名单）以及风险操作的二次因子认证；集成自动化SAST扫描，杜绝引入存在已知安全漏洞的鉴权库组件。
