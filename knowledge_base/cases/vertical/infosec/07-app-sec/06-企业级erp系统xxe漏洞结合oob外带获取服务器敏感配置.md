# 企业级ERP系统XXE漏洞结合OOB外带获取服务器敏感配置
- **所属领域**: 信息安全 (应用安全)
- **威胁类型**: XML外部实体注入 (XXE) / 信息泄露
- **事件回顾**: 某大型制造企业的内部ERP系统遭到外部黑客攻击，服务器上的敏感配置文件（如数据库连接字符串、内网网段架构以及私钥文件）被窃取。黑客利用获取的数据库凭据进一步横向渗透至内网核心区域。经溯源发现，该ERP系统的一个报表导入模块支持用户上传基于XML格式的自定义模板文件，而系统后端的XML解析器未禁用外部实体的加载，导致严重的XXE（XML External Entity）漏洞。由于系统不回显报错信息，攻击者利用了OOB（Out-Of-Band）带外数据传输技术完成了数据窃取。
- **Red View (利用路径)**: 攻击者在测试报表导入功能时发现系统接受XML文件且使用旧版Java解析库。为了读取系统内的 `/etc/passwd` 以及包含数据库凭证的配置文件，攻击者在上传的XML文件中精心构造了恶意DTD（文档类型定义）。由于系统是“盲”（Blind）XXE，不直接将解析结果返回给前端，攻击者通过参数实体（Parameter Entity）构造了一条请求：首先利用外部实体读取本地文件内容，然后将读取到的内容作为URL参数，触发一次对攻击者控制的公网服务器（如DNSLog或HTTP服务器）的外部请求。服务器在解析该XML时，默默地将敏感文件内容发送到了攻击者的监控服务器上。
- **Blue View (防御缺失)**: 
  1. 默认安全配置缺失：Java后端使用的 `DocumentBuilderFactory` 或 `SAXParserFactory` 未进行安全加固，未主动禁用外部通用实体和外部参数实体的解析 (`FEATURE_SECURE_PROCESSING` 未启用)。
  2. 隔离与出网策略缺失：ERP系统所在的内网服务器缺乏严格的外网访问控制。允许应用服务器自由向外部发起HTTP/DNS请求，为OOB带外攻击提供了网络基础。
  3. 日志与异常检测盲区：对解析失败的XML或后台异常发起的外部连接没有部署实时的告警监控机制。
- **策略借鉴**: 彻底在代码层加固所有的XML解析器，严格禁用 DTD (`disallow-doctype-decl`) 及其外部实体的处理功能；如果业务确实需要处理结构化数据，推荐使用更安全的格式如JSON代替XML；在数据中心网络层实行零信任策略，严格封堵后端业务服务器不必要的主动出网请求，切断外带泄露通道。
