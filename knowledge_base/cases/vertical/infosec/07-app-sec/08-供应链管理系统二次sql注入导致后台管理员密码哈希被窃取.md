# 供应链管理系统二次SQL注入导致后台管理员密码哈希被窃取
- **所属领域**: 信息安全 (应用安全)
- **威胁类型**: 二次SQL注入 (Second-Order SQLi) / 权限提升
- **事件回顾**: 某大型物流企业的核心供应链管理系统遭遇入侵，大量供应商的合同底价与系统超级管理员密码哈希被窃取。黑客随后利用破解出的密码直接登录了后台系统。事后审计日志时，安全团队未能立刻在常规的查询接口发现典型的SQL注入攻击痕迹。经过深入的代码审计与数据流追踪，发现问题存在于一个隐蔽的“二次SQL注入”场景中。攻击者在注册个人信息时写入了恶意载荷，而在系统后台生成报表时触发了该载荷，导致内网数据库沦陷。
- **Red View (利用路径)**: 攻击者首先通过前台的“供应商注册”模块，在“公司名称”或“备注地址”字段中填入了精心构造的SQL注入Payload（例如：`TestCompany', (SELECT user_pass FROM admins limit 1))--`）。由于注册接口虽然使用了参数化查询防止了直接注入，但系统将这些包含了SQL语句的字符串原封不动地存储到了数据库中。几天后，内部管理员登录后台，调用“生成月度账单报表”功能。该功能的后端代码在从数据库提取“公司名称”进行字符串拼接操作并动态构建复杂查询时，未进行二次安全转义。之前存入的Payload被激活并执行，将管理员表的数据拼接到报表字段中，使得攻击者通过合法渠道下载报表即可获取敏感数据。
- **Blue View (防御缺失)**: 
  1. 内部信任滥用：开发团队错误地认为“从数据库中读取出的数据已经是安全的”，因此在后台进行数据二次处理或动态拼接SQL时，未再次进行严格的参数化处理。
  2. 深度防御机制不完善：未实行全链路的输入验证与输出编码（Input Validation & Output Encoding），仅仅在前台入口进行了粗粒度拦截。
  3. 数据库权限未隔离：后台报表生成模块的数据库账户权限过高，能够越权读取与该业务无关的系统管理员账户表（`admins`）。
- **策略借鉴**: 强制推行ORM框架或全链路的预编译查询（Prepared Statements），彻底摒弃任何场景下的SQL字符串动态拼接，不论数据来源是用户输入还是数据库读取；实施最小权限原则，为不同的业务模块分配相互隔离的数据库读写账号，避免跨表越权查询；在应用输出层进行严格的编码，以防止包括注入和XSS在内的多维度攻击。
