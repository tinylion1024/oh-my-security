# 开发人员硬编码AWS AK/SK至GitHub公有仓库引发的数据毁灭性勒索
- **所属领域**: 信息安全 (数据保护)
- **威胁类型**: 凭证泄露 / 内部威胁漏洞 / 勒索攻击
- **事件回顾**: 一家专注于人工智能模型训练的创新型科技公司在数小时内失去了其在云端积累数年的核心数据集与训练模型，业务彻底瘫痪。攻击者不仅删除了S3桶内的所有数据，还删除了全部的快照备份，并留下勒索信要求支付巨额比特币。事后深度复盘发现，事件的导火索是某后端研发工程师在向GitHub的开源仓库（公共代码库）提交自动部署脚本时，意外将具备AWS全局管理员权限的Access Key和Secret Key硬编码在了配置文件中，被黑客的爬虫系统在几秒钟内捕获。
- **Red View (利用路径)**: 恶意黑客团伙运营着针对公共代码托管平台（如GitHub、GitLab）的实时监控机器人，不断抓取最新提交的Commit记录，通过正则表达式匹配寻找典型的云厂商凭证（如以 `AKIA` 开头的AWS Key）。在工程师提交代码仅仅几分钟后，爬虫便提取出了高权限凭据。攻击者随即利用这些凭据通过AWS CLI接入了目标账户。在确认权限后，攻击者并未立刻窃取庞大体积的数据，而是编写脚本迅速清空了RDS数据库实例并删除了所有S3存储桶和EBS卷的快照。最后，他们创建了带有勒索信息的存储桶和文件，试图以此要挟公司进行支付。
- **Blue View (防御缺失)**: 
  1. 代码安全审查缺失：在CI/CD流水线中，未引入诸如GitLeaks或TruffleHog等敏感信息扫描工具（Secret Scanner），无法在代码推送到远程仓库前拦截硬编码密钥。
  2. IAM权限过度分配：分配给开发人员或部署脚本的云凭据权限过大（如赋予了 `AdministratorAccess`），违背了最小特权原则，导致凭证一旦泄露，攻击者能直接进行毁灭性操作。
  3. 核心数据容灾机制脆弱：备份数据与生产环境处于同一云账号和权限体系下，未实施跨账号隔离备份或不可变存储（Object Lock）策略，导致备份被一并销毁。
- **策略借鉴**: 在代码库提交端（Pre-commit hook）和CI/CD链路全面部署凭据扫描阻断机制；云原生环境应使用IAM Role（动态临时凭证）或外部的凭证管理服务（如HashiCorp Vault）替代静态长效AK/SK；建立异地/异云和具备不可变属性（WORM技术）的数据离线容灾备份体系，确保兜底恢复能力。
