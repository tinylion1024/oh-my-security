# 某车企车联网API接口传输明文数据导致车辆实时轨迹与隐私被大规模抓取
- **所属领域**: 信息安全 (数据保护)
- **威胁类型**: 传输中数据未加密 (Data in Transit) / API信息泄露
- **事件回顾**: 某知名新能源汽车品牌爆出严重隐私安全事件，黑客宣称掌握了大量车主的实时行驶轨迹、家庭住址、车内录音片段甚至车辆控制权限。安全团队彻查了该车企的云端架构与车端通信协议，发现问题出在车辆与云端控制中心（TSP平台）之间通信的一个核心API网关上。由于开发为了“便于调试和降低延迟”，该网关部分接口在传输高清媒体流和实时遥测数据时，回退到了未加密的HTTP协议，或者使用了存在严重缺陷的弱加密配置。
- **Red View (利用路径)**: 攻击者首先通过逆向工程该品牌车主的手机APP，获取了车联网API的通信地址与部分结构。他们发现，尽管关键的控制指令（如开锁、启动）使用了HTTPS，但为了节省带宽，车辆上传的高频GPS定位包、车内监控摄像头快照以及诊断日志等数据，竟是通过明文的HTTP协议或者未校验SSL证书（可轻易发动中间人攻击）的通道进行传输的。攻击者在运营商网络骨干节点附近，或在公共充电桩搭建的免费Wi-Fi热点处，通过ARP欺骗与流量嗅探工具（如Wireshark），实施中间人攻击（MITM）。由于缺乏加密保护，攻击者如入无人之境，直接还原并记录了大量车主长期的行驶路径和私密生活数据。
- **Blue View (防御缺失)**: 
  1. 传输层加密缺失与降级：对涉及个人敏感信息的车联网数据流未实行强制的、端到端的TLS 1.2/1.3强加密传输，允许协议降级为明文HTTP。
  2. 证书双向认证（mTLS）未落实：车辆终端和TSP服务端之间缺乏相互身份认证。APP端也未实现证书锁定（SSL Pinning），导致中间人攻击可以轻易替换伪造证书而系统不告警。
  3. 数据匿名化不足：车辆上传的遥测数据直接关联了明文的车辆识别码（VIN）和用户ID，未在终端侧做假名化处理，导致截获的数据直接可对应到具体个人。
- **策略借鉴**: 强制全域系统（无论数据类型）实施基于TLS 1.3的强加密传输，在代码层硬编码拒绝一切非加密的HTTP流量；在IoT车机端和移动APP端必须部署严格的证书锁定（SSL Pinning）机制以及双向证书认证（mTLS）体系；在数据采集源头落实数据脱敏和去标识化，以切断轨迹数据与具体车主身份的直接联系。
