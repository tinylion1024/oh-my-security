# 某大型互联网公司CI/CD流水线遭服务账号越权入侵事件
- **所属领域**: 信息安全
- **威胁类型**: 流水线入侵 / 权限提升
- **事件回顾**: 2023年某大型互联网企业的一名外包开发人员账号被窃取，攻击者利用该账号登录了内部的GitLab平台。通过审查源码，攻击者发现一个部署自动化服务（Service Account）的令牌配置不当，并被明文硬编码在某个不再维护的旧项目中。随后，攻击者利用该服务账号的高级权限，成功在CI/CD流水线的构建节点中植入了恶意后门代码，进而横向移动至生产环境。
- **Red View (利用路径)**: 通过社会工程学获取基础研发人员凭证 -> 登入代码托管平台枚举历史仓库 -> 发现废弃代码中硬编码的Service Account Token -> 利用该Token向核心CI/CD构建脚本提交恶意修改 -> 触发自动构建将后门打包进Docker镜像 -> 在生产环境执行恶意负载建立C2回连。
- **Blue View (防御缺失)**: 代码托管平台缺乏对僵尸仓库和硬编码密钥的常态化扫描；服务账号（Service Account）未遵循最小权限原则，存在严重的权限过度分配；CI/CD构建流水线未引入代码签名与镜像安全性强制校验环节，导致恶意修改未被阻断。
- **策略借鉴**: 全面引入DevSecOps机制，在代码提交阶段（Pre-Commit）部署密钥硬编码检测工具（如TruffleHog/Gitleaks）；强制实施凭证轮转策略并使用安全保管库（如HashiCorp Vault）；对CI/CD流水线进行权限最小化治理，并强制要求关键构建步骤的多人代码审查（Code Review）。
