# 知名开源NPM生态依赖混淆与供应链投毒攻击事件
- **所属领域**: 信息安全
- **威胁类型**: 供应链攻击 / 依赖混淆
- **事件回顾**: 2022年，一家著名SaaS公司的内部构建系统突然瘫痪，安全团队排查发现是由于新版本构建中拉取了恶意NPM包。攻击者在公共NPM仓库上注册了与该公司内部私有依赖包同名的项目，并将版本号设置为99.9.9。由于公司内部的包管理工具在解析依赖时，默认优先拉取版本号更高的包，导致开发环境和CI服务器自动下载并执行了恶意的公共包，执行了信息窃取脚本。
- **Red View (利用路径)**: 分析目标公司的开源代码、招聘信息或公开报错日志获取内部私有包命名规则 -> 在公共注册中心（如NPM/PyPI）注册同名包并设置极高版本号 -> 在包的安装钩子（preinstall）中植入环境信息收集与外发脚本 -> 等待目标公司的构建系统自动解析并拉取恶意包 -> 收集窃取的环境变量及内网凭证。
- **Blue View (防御缺失)**: 内部包管理与依赖解析机制存在漏洞，未对内部包与外部包的拉取顺序和信任源进行严格区分；开发环境和构建节点缺乏出口网络流量管控，允许任意的外部连接；缺乏对引入开源组件的成分分析（SCA）与行为沙箱验证。
- **策略借鉴**: 配置内部私有镜像仓库（如Nexus/Artifactory）时，强制屏蔽对内部私有包名称的外部公网查询；实施严格的软件物料清单（SBOM）管理与SCA扫描；禁用NPM等包管理器的自动执行生命周期脚本（如`--ignore-scripts`），限制构建节点的网络外联权限。
