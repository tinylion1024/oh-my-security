# Terraform基础设施即代码(IaC)配置失误致S3敏感数据泄露
- **所属领域**: 信息安全
- **威胁类型**: IaC配置错误 / 数据泄露
- **事件回顾**: 某金融科技公司在向AWS云迁移的过程中，使用Terraform编写云基础设施配置文件。开发工程师在编写S3存储桶的ACL配置时，为了方便临时测试，将Bucket的权限错误地设置为了`public-read`。由于CI/CD自动化部署并未在基础设施代码发布前进行安全扫描，导致包含大量用户交易日志和个人身份信息（PII）的S3存储桶在公网直接裸奔暴露长达三周，最终被安全研究人员通过自动化工具发现。
- **Red View (利用路径)**: 利用云端公开资源枚举工具（如AWSBucketDump）进行大规模自动化扫描 -> 匹配到特定的S3存储桶并发现匿名访问权限 -> 发送GET请求列出Bucket内所有的Object对象 -> 批量下载并提取其中的敏感金融交易和PII数据。
- **Blue View (防御缺失)**: IaC（基础设施即代码）代码在合并与部署前缺乏自动化安全语法与合规性扫描；云平台生产环境缺乏配置漂移的实时监控；未能遵循云安全最佳实践强制开启“阻止所有公共访问”的安全基线策略。
- **策略借鉴**: 在CI/CD流水线中集成IaC安全扫描工具（如Checkov、tfsec），在代码合并前自动阻断高危配置；在AWS云管平台层面全局强制开启S3 Block Public Access；部署云安全态势管理（CSPM）平台，实时监测和自动修复配置漂移与违规基线问题。
