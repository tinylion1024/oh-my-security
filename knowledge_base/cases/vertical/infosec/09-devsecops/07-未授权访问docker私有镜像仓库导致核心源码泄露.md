# 未授权访问Docker私有镜像仓库导致核心源码泄露
- **所属领域**: 信息安全
- **威胁类型**: 未授权访问 / 数据泄露
- **事件回顾**: 某物联网公司将自研微服务的Docker镜像存储在自建的私有Registry（如Harbor/Docker Registry）中，但运维人员在配置时遗漏了认证模块（Auth），导致该镜像仓库监听的5000端口对公网敞开。恶意竞争对手通过全网扫描发现了这一未授权访问节点，并利用API接口直接拉取了所有业务的最新镜像。通过逆向分析镜像中的层文件，他们不仅获取了完整的核心业务源代码，还提取了打包在镜像中的数据库连接密码。
- **Red View (利用路径)**: 对目标资产的特定端口（如5000/80/443）进行端口和服务扫描 -> 发现暴露的Docker Registry API (`/v2/_catalog`) -> 列出所有存在的镜像仓库与Tag -> 下载所有层的Blob文件包解压 -> 使用逆向工具提取镜像文件系统中的源码与配置文件。
- **Blue View (防御缺失)**: 镜像仓库基础设施缺乏最基本的身份认证与鉴权配置；资产测绘与公网暴露面收敛机制失效；在制作Docker镜像时违反了安全原则，将源代码和明文密码打包至最终镜像中。
- **策略借鉴**: 对所有内部使用的Registry平台实施强身份认证（RBAC）与TLS加密，并限制访问白名单（IP ACL）；在CI/CD阶段集成镜像扫描工具（如Trivy或Clair），检测镜像中是否包含敏感凭证；推广多阶段构建（Multi-stage Build），确保最终发布的镜像中不包含源码及多余环境。
