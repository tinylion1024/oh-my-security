## 📊 阶段 1: 发现 (Detect)

### 发现渠道

| 渠道类型 | 响应方式 | 响应时间要求 |
|---------|---------|-------------|
| 监控告警 | 自动触发 → 立即查看 | P0/P1 < 5分钟 |
| 用户反馈 | 工单/邮件 → 确认验证 | < 30分钟 |
| 自动检测 | 定时任务 → 日志分析 | < 1小时 |
| 第三方通知 | 安全厂商/平台 → 验证处理 | < 4小时 |

### 发现检查清单

```markdown
□ 是否有异常告警触发？
  - 服务可用性告警
  - 异常流量告警
  - 错误率飙升告警
  - 安全事件告警

□ 是否有用户反馈异常？
  - 功能无法使用
  - 数据异常
  - 账号问题
  - 支付问题

□ 是否有自动检测发现？
  - 日志异常分析
  - 安全扫描结果
  - 合规检查失败
  - 性能异常检测

□ 是否有外部通知？
  - 安全研究员报告
  - 平台安全通知
  - 供应商告警
  - 监管机构通知
```

### 发现阶段记录模板

```yaml
事件发现记录:
  发现时间: YYYY-MM-DD HH:MM:SS
  发现渠道: [监控告警/用户反馈/自动检测/第三方通知]
  发现人员: [姓名/系统名称]
  初步描述: |
    [简要描述发现的问题]
  相关证据: |
    - 告警截图
    - 日志片段
    - 用户反馈内容
  初步判断: [疑似安全事件/待确认/误报]
```

## 🔍 阶段 2: 评估 (Assess)

### 评估流程

```
┌─────────────────────────────────────────────────────────┐
│                    事件评估流程                          │
└─────────────────────────────────────────────────────────┘

Step 1: 确认事件真实性
        ├─→ 验证告警/报告是否准确
        ├─→ 排除误报可能
        └─→ 确认是否为安全事件

Step 2: 确定事件类型
        ├─→ 账号安全事件
        ├─→ 数据泄露事件
        ├─→ 支付安全事件
        ├─→ API 攻击事件
        ├─→ 服务器入侵事件
        └─→ 其他安全事件

Step 3: 评估影响范围
        ├─→ 影响用户数量
        ├─→ 影响业务功能
        ├─→ 数据敏感程度
        └─→ 持续时间

Step 4: 判定严重等级
        ├─→ P0 紧急严重
        ├─→ P1 高优先级
        ├─→ P2 中等优先级
        └─→ P3 低优先级

Step 5: 决定响应策略
        ├─→ 立即遏制
        ├─→ 监控观察
        └─→ 排期处理
```

### 严重等级评估矩阵

| 维度 | P0 | P1 | P2 | P3 |
|------|----|----|----|----|
| **业务影响** | 完全中断 | 核心功能失效 | 部分功能受损 | 无明显影响 |
| **数据影响** | 敏感数据泄露 | 非敏感数据泄露 | 无泄露风险 | 无数据风险 |
| **用户影响** | >50% 用户 | 10-50% 用户 | <10% 用户 | 单个用户 |
| **控制权** | 完全失控 | 部分失控 | 可控 | 完全可控 |
| **可恢复性** | 难以恢复 | 可恢复但复杂 | 容易恢复 | 简单修复 |

## 🛑 阶段 3: 遏制 (Contain)

### 遏制策略选择

```
┌─────────────────────────────────────────────────────────┐
│                    遏制策略决策                          │
└─────────────────────────────────────────────────────────┘

                    是否需要立即遏制？
                           │
              ┌────────────┴────────────┐
              │                         │
            是 (P0/P1)                否 (P2/P3)
              │                         │
              ↓                         ↓
      选择遏制方式                   监控观察
              │                    - 加强监控
              │                    - 准备方案
              │                    - 等待时机
              │
    ┌─────────┼─────────┐
    │         │         │
  软遏制    硬遏制    混合遏制
    │         │         │
    ↓         ↓         ↓
 限流降级   完全隔离   组合策略
 关闭功能   下线服务
 切换备用   断开网络
```

### 常见遏制措施

| 事件类型 | 软遏制措施 | 硬遏制措施 |
|---------|-----------|-----------|
| **账号被盗** | 强制下线、重置密码 | 禁用账号、冻结账户 |
| **数据泄露** | 撤回访问权限 | 关闭服务、隔离数据库 |
| **支付异常** | 暂停支付功能 | 关闭支付通道 |
| **API 攻击** | 限流、封禁 IP | 下线 API、切换域名 |
| **服务器入侵** | 隔离进程 | 断网、关机、快照 |

### 遏制执行检查清单

```markdown

□ 隔离受影响系统
  - [ ] 网络隔离（如需要）
  - [ ] 服务隔离
  - [ ] 数据隔离

□ 保存证据
  - [ ] 截图保存

  - [ ] 流量抓包
  - [ ] 内存快照（如需要）

□ 记录操作
  - [ ] 记录每个操作及时间
  - [ ] 保存操作日志
  - [ ] 标记关键证据

□ 通知相关方
  - [ ] 内部通知（如需要）
  - [ ] 用户通知（如需要）
  - [ ] 监管报告（如需要）
```

## 🔧 阶段 4: 恢复 (Recover)

### 恢复流程

```
┌─────────────────────────────────────────────────────────┐
│                    服务恢复流程                          │
└─────────────────────────────────────────────────────────┘

Step 1: 制定恢复计划
        ├─→ 确定恢复顺序（核心优先）
        ├─→ 准备恢复方案
        └─→ 评估恢复风险

Step 2: 修复根本原因
        ├─→ 定位问题根因
        ├─→ 实施修复方案
        └─→ 测试修复效果

Step 3: 分阶段恢复
        ├─→ 恢复核心服务
        ├─→ 恢复辅助服务
        └─→ 恢复全部功能

Step 4: 验证恢复效果
        ├─→ 功能测试
        ├─→ 性能测试
        └─→ 安全验证

Step 5: 加强监控
        ├─→ 提高监控频率
        ├─→ 增加监控指标
        └─→ 设置告警阈值

Step 6: 通知恢复状态
        ├─→ 内部通知
        ├─→ 用户通知
        └─→ 公开声明（如需要）
```

### 恢复优先级矩阵

| 服务类型 | 恢复优先级 | 恢复顺序 |
|---------|-----------|---------|
| 核心业务服务 | 最高 | 第 1 批 |
| 用户数据服务 | 高 | 第 2 批 |
| 支付相关服务 | 高 | 第 2 批 |
| 辅助功能服务 | 中 | 第 3 批 |
| 后台管理服务 | 低 | 第 4 批 |

### 恢复验证检查清单

```markdown
□ 功能验证
  - [ ] 核心功能正常
  - [ ] 用户流程正常
  - [ ] 数据读写正常
  - [ ] 接口调用正常

□ 性能验证
  - [ ] 响应时间正常
  - [ ] 吞吐量正常
  - [ ] 资源使用正常
  - [ ] 无性能退化

□ 安全验证
  - [ ] 漏洞已修复
  - [ ] 权限配置正确
  - [ ] 访问控制有效
  - [ ] 加密措施到位

□ 监控验证
  - [ ] 监控指标正常
  - [ ] 告警规则生效
  - [ ] 日志记录正常
  - [ ] 无异常告警
```

## 📝 阶段 5: 复盘 (Review)

### 复盘流程

```
┌─────────────────────────────────────────────────────────┐
│                    事后复盘流程                          │
└─────────────────────────────────────────────────────────┘

Step 1: 梳理时间线
        ├─→ 收集所有记录
        ├─→ 整理时间顺序
        └─→ 标记关键节点

Step 2: 根因分析
        ├─→ 使用 5 Whys 方法
        ├─→ 识别直接原因
        ├─→ 识别根本原因
        └─→ 识别系统性原因

Step 3: 影响评估
        ├─→ 用户影响
        ├─→ 业务影响
        ├─=> 财务影响
        └─→ 声誉影响

Step 4: 响应评估
        ├─→ 响应及时性
        ├─→ 措施有效性
        ├─→ 流程合理性
        └─→ 工具适用性

Step 5: 改进计划
        ├─→ 技术改进
        ├─→ 流程改进
        ├─→ 工具改进
        └─→ 培训改进

Step 6: 文档沉淀
        ├─→ 更新应急响应文档
        ├─→ 更新技术文档
        ├─→ 分享经验教训
        └─→ 归档事件记录
```

## 👥 角色分工（独立开发者版）

### 独立开发者角色映射

作为独立开发者，你需要承担传统团队中的多个角色：

| 传统角色 | 独立开发者对应 | 注意事项 |
|---------|--------------|---------|
| **事件指挥官** | 你自己 | 保持冷静，快速决策 |
| **技术响应** | 你自己 | 优先止血，再求完美 |
| **沟通协调** | 你自己 | 准备好模板，高效沟通 |
| **法律合规** | 外部咨询 | 重大事件寻求专业帮助 |
| **公关处理** | 你自己 | 诚实透明，及时更新 |

### 决策授权矩阵

| 决策类型 | P0 | P1 | P2 | P3 |
|---------|----|----|----|----|
| 下线服务 | ✓ 立即决定 | ✓ 立即决定 | ⚠ 评估后决定 | ✗ 不需要 |
| 数据删除 | ✓ 立即决定 | ⚠ 谨慎决定 | ✗ 不建议 | ✗ 不需要 |
| 用户通知 | ✓ 立即通知 | ✓ 及时通知 | ⚠ 评估后通知 | ✗ 不需要 |
| 公开声明 | ⚠ 咨询后决定 | ⚠ 评估后决定 | ✗ 通常不需要 | ✗ 不需要 |
| 外部求助 | ✓ 立即求助 | ⚠ 评估后求助 | ✗ 通常不需要 | ✗ 不需要 |

**图例**: ✓ 立即执行 | ⚠ 谨慎评估 | ✗ 不建议/不需要

### 何时寻求外部帮助

```
┌─────────────────────────────────────────────────────────┐
│              何时寻求专业帮助                            │
└─────────────────────────────────────────────────────────┘

立即寻求帮助（P0）:
  ✓ 法律顾问 - 涉及用户隐私、数据泄露
  ✓ 安全公司 - 服务器完全被控制
  ✓ 公关顾问 - 重大公关危机
  ✓ 执法机关 - 涉及犯罪行为

考虑寻求帮助（P1）:
  ✓ 技术顾问 - 超出技术能力范围
  ✓ 云服务商 - 云平台相关问题
  ✓ 支付渠道 - 支付相关问题

通常不需要（P2/P3）:
  - 可自行处理的技术问题
  - 影响范围有限的事件
  - 常规安全加固工作
```

## 🛠️ 应急响应工具箱

### 必备工具清单

| 工具类型 | 推荐工具 | 用途 |
|---------|---------|------|
| **日志分析** | grep, awk, ELK | 快速分析日志 |
| **网络分析** | tcpdump, Wireshark | 抓包分析流量 |
| **进程分析** | ps, top, htop | 查看异常进程 |
| **文件分析** | find, md5/sha256 | 检查文件变更 |
| **备份恢复** | rsync, snapshots | 快速备份恢复 |
| **监控告警** | Prometheus, Grafana | 实时监控 |

### 快速命令参考

```bash
# 查看最近的错误日志
tail -f /var/log/error.log | grep -E "ERROR|FATAL|Exception"

# 查看异常进程
ps aux | grep -E "defunct|Z|<defunct>"

# 查看网络连接
netstat -antp | grep ESTABLISHED

# 查看最近的文件修改
find /var/www -mtime -1 -type f

# 快速备份
tar -czf backup_$(date +%Y%m%d_%H%M%S).tar.gz /path/to/data

# 检查文件完整性
find /path -type f -exec md5sum {} \; > checksums.txt
```
