# 服务宕机

## 严重程度
🟠 高

## 识别信号
- 服务监控告警（HTTP 5xx 错误激增）
- 用户报告无法访问
- 服务器资源耗尽（CPU/内存/磁盘）
- 数据库连接异常
- 网络连接中断
- 容器或进程异常退出
- 负载均衡器健康检查失败
- API 响应超时或超慢

## 立即行动 (0-15分钟)

1. **确认故障**
   - 验证服务状态
   - 确认影响范围
   - 记录故障开始时间
   - 确定故障类型

2. **紧急恢复**
   - 尝试重启服务
   - 检查是否需要回滚
   - 扩容资源（如需要）
   - 启用备用系统

3. **沟通协调**
   - 通知运维团队
   - 更新服务状态页面
   - 通知客服团队
   - 准备用户通知

4. **问题定位**
   - 查看系统日志
   - 检查资源使用情况
   - 分析网络连接
   - 排查最近变更

## 短期处理 (1-4小时)

1. **根因分析**
   - 分析日志确定故障原因
   - 检查是硬件/软件/网络问题
   - 确定是否安全事件导致
   - 评估是否需要外部支援

2. **服务恢复**
   - 实施修复方案
   - 逐步恢复服务
   - 监控系统稳定性
   - 验证功能正常

3. **流量管理**
   - 启用降级策略
   - 管理用户流量
   - 处理积压请求
   - 通知排队用户

4. **沟通汇报**
   - 定期更新状态
   - 向管理层汇报
   - 准备事故报告
   - 协调跨部门配合

## 恢复步骤

1. **完全恢复**
   - 确认所有服务正常
   - 清理临时措施
   - 恢复完整功能
   - 解除降级状态

2. **数据一致性检查**
   - 验证数据完整性
   - 处理不一致数据
   - 恢复丢失数据（如有）
   - 验证业务数据准确性

3. **监控验证**
   - 加强系统监控
   - 验证告警机制
   - 确认恢复稳定
   - 持续观察系统状态

4. **用户补偿**
   - 通知服务恢复
   - 处理用户投诉
   - 必要时提供补偿
   - 感谢用户耐心

## 复盘要点

- [ ] 宕机的根本原因是什么？
- [ ] 从发现问题到恢复用了多长时间？
- [ ] 监控和告警是否及时有效？
- [ ] 应急响应流程是否顺畅？
- [ ] 是否有冗余和容灾机制？
- [ ] 最近的变更是否引入问题？
- [ ] 是否需要增加资源容量？
- [ ] 如何防止类似问题再次发生？
