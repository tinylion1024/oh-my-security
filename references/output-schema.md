# Security Master 输出模板 (Output Schema)

为了保证输出的专业性与统一性，所有 Agent 在最终合成报告时，必须严格遵守以下对应模式的 Markdown 模板。

---

## 1. Risk Brain 模式 (架构诊断与风险评估)
**适用场景**：对现有架构、网络拓扑或业务逻辑进行全面体检。

```markdown
# 🛡️ 架构风险诊断报告 (Risk Brain)

> **评估结论**: [Critical / High / Medium / Low] (附带基于 CVSS 的最高分风险简述)

## 1. 资产与暴露面 (Asset & Surface)
- **核心资产**: [如：用户 PII 数据、支付网关]
- **关键暴露面**: [如：公网开放的 API、第三方依赖]

## 2. 威胁推演 (Red Team View)
| 威胁编号 | 攻击向量 | 潜在影响 | CVSS 预估 |
|---|---|---|---|
| TR-01 | [利用链简述] | [对 CIA 的影响] | [如: 8.5] |

## 3. 防御差距与挑战 (Blue & Skeptic View)
- **缺失的防线**: [如：缺乏边界内的零信任验证]
- **脆弱的假设**: [Skeptic Agent 指出的现有防线可能被绕过的方式]

## 4. 治理建议 (Actionable Recommendations)
### 🚨 立即行动 (Immediate - 24h内)
- [具体的操作，如：修改配置、阻断 IP]

### 🏗️ 架构优化 (Strategic - 1-3个月)
- [具体的架构改进，如：引入 RASP、重构鉴权]

## 5. 专家依据 (References)
- 关联案例: [链接到 knowledge/cases/]
- 推荐武器: [链接到 knowledge/weapons/]
```

---

## 2. Threat Model 模式 (专项威胁建模)
**适用场景**：在系统设计阶段（如 PRD 评审期），针对特定模块进行 STRIDE 建模。

```markdown
# 🎯 专项威胁建模分析 (Threat Model)

> **分析对象**: [如：AI 客服对话模块]
> **采用框架**: [如：STRIDE]

## 1. 数据流向分析 (DFD)
- **信任边界**: [描述边界划分]
- **交互过程**: [简述 1, 2, 3]

## 2. STRIDE 威胁矩阵
| 威胁元素 | 具体表现 (Red View) | 缓解措施 (Blue View) | 验证状态 |
|---|---|---|---|
| **S**poofing | 身份伪造 | 引入双向 TLS | 待落地 |
| **T**ampering| 数据篡改 | 对 API 载荷签名 | 已设计 |
| ... | ... | ... | ... |

## 3. 合规底线 (Compliance Constraints)
- [Compliance Agent 提供的相关法规要求，如：必须对用户对话日志进行匿名化处理 (PIPL)]
```

---

## 3. BizSec Guard 模式 (业务安全治理)
**适用场景**：解决由于业务逻辑导致的黑灰产攻击（如：薅羊毛、爬虫、作弊）。

```markdown
# 💼 业务安全治理方案 (BizSec Guard)

> **治理目标**: [如：阻断自动化刷单，控制误伤率 < 1%]

## 1. 黑灰产画像 (Threat Profiling)
- **作弊动机**: [获利点分析]
- **特征提取**: [设备指纹、IP 聚集度、行为时序]

## 2. 纵深拦截策略 (Defense in Depth)
- **第一层 (端侧)**: [如：集成强对抗验证码、环境监测]
- **第二层 (网关)**: [如：基于行为频次的动态限流]
- **第三层 (业务)**: [如：高危操作延迟结算、人工审核池]

## 3. 业务平衡评估 (Biz Agent View)
- **性能损耗**: [预估延迟增加]
- **体验折损**: [是否增加用户操作步骤]
- **ROI 预估**: [阻断损失的资金 VS 引入安全组件的成本]

## 4. 实战案例参考
- [从知识库检索出的高度相似的历史对抗案例]
```
